## 一、TG 扫码：一次失败换来正确路线（本轮最重要的方案修正）

**时间线与发现**：
1. 外部 QR 工具第一轮 8 分钟未扫上（图片查看器缓存旧码）→ 重写为**浏览器自刷新页**
   （3 秒换新码、20 分钟窗口、一个窗口永远最新码）；
2. 老板扫码**成功**——登录的是 **Katie +639674531086**（不是老号；老号登录态已被 TG 服务端
   REVOKED，属预期）；工具打印 ✅ emoji 在 GBK 终端崩溃，但登录态已落盘，手工完成 swap；
3. 实例重启后引擎报 `no such column: number`——**深挖发现根本不兼容**：
   外部工具用 Telethon（session schema v7），而 **chengjie 引擎的协议栈是 Pyrogram**
   （老 session 是 Pyrogram schema）。Telethon session 无法被引擎使用；
4. **正确路线（引擎本来就有）**：统一收件箱 → **账号管理 → ＋扫码新增**——引擎内置
   `TelegramQrLogin`（pyrogram QR + DC 迁移处理 + session_string 注册表管理 + 支持多账号
   协议多开）。外部工具方案废弃，登录一律走后台 UI。

**最终形态（已定型）**：
- overlay `telegram:` 段保留 `api_id/api_hash`（后台扫码通道 `resolve_credentials` 读它），
  `phone_number/session_name` 显式置空 → 主线协议 client 安全跳过（日志实证
  "Telegram 协议号未配置，跳过协议客户端初始化；统一收件箱/QR 登录不受影响"）；
- Telethon session 归档（`*.telethon_archive`），老 REVOKED session 归档；
- **session 区别与功能划分**（回答老板问题）：一号一 session 一用途；协议号 session 由引擎
  registry 按 account 管理（多账号原生支持）；总机/交付号/测试号在**渠道台账页**登记划分。

## 三、chengjie「签发即台账」静默丢单修复（agent 抓到的真 bug）

- **bug**：部署侧把 `CHENGJIE_LEDGER_OUTBOX` 注入为**目录**（`<数据根>\ledger_outbox`，
  实例骨架预建），而引擎钩子把它当**文件路径**——`open(目录,'a')` 必败且 fail-silent，
  **签发记录会无声丢失**（等真实签发发生时才会踩到，此时已丢数据）；
- **修**：env/参数为已存在目录时自动落标准文件名 `ledger_outbox.jsonl`（与 run_export
  license 段首选候选一致），文件路径语义向后兼容；smoke 实测目录形态落盘成功；
- 顺手：实例启动的 `Win32_Process.Create` 加 `ShowWindow=0`——开机自启不再弹黑窗。
