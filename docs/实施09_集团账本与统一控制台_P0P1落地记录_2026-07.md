## 二、实施中的方案再优化（相对原方案的变更及理由）

1. **PostgreSQL → SQLite 先行（PG-ready）**：原 P0 计划"订单/留资迁 PostgreSQL"。实施时改为 better-sqlite3（WAL）：VPS 1C1G 无 PG、SQLite 已是全栈熟路、私有化交付友好；DDL 刻意保守（TEXT 主键 / ISO8601 / raw JSON 文本），迁 PG 只需换驱动不改结构。
2. **"迁移"→"影子账本 + 双写 + 幂等回填"**：不动现有 JSON 主真相源与任何 API 行为，账本先做只读镜像；对单人团队可随时停、可随时重放（backfill 幂等），零回滚风险。切主放到账本稳定运行一段时间之后。
3. **授权统一"先台账后签发"**：两套 Ed25519 签发实现完全不动，只做只读导出→归一化→导入台账。发现 chengjie 签发端根本不留台账（token 即发即走），使"统一台账"从锦上添花升级为**必须品**——这是本次实施发现的最大收入风险敞口。
4. **事件总线本期只做落盘契约**：不建 HTTP 收集器，先冻结信封/命名/隐私红线与 spool 格式（断网可攒、恢复可补），引擎接入零网络依赖。传输层下阶段再加，契约不会返工。
5. **ID 双语言实现 + 已知答案向量**：Python/TS 各一份实现，规范里放了三组编码锚点向量保证逐字节兼容，杜绝"两边 ULID 不是同一种 ULID"的隐性坑。
6. **console 口令面收窄**：CONSOLE_KEY 一旦设置，ADMIN_KEY/SETUP_KEY 对 /console 即失效（严格回退链），避免口令面扩大；页面级也做会话守卫，未登录时业务数据不进 RSC payload。

## 四、VPS 上线 runbook（website 侧）

1. 常规发布流程照旧（deploy-stage 打包 → VPS `npm ci && npm run build`）；better-sqlite3 在 Node 20 有预编译二进制，若下载失败：`apt install -y python3 make g++` 后 `npm rebuild better-sqlite3`。
2. `.env.local` 新增 `CONSOLE_KEY=<独立强随机>`（不要与 ADMIN_KEY 同值）。可选 `LEDGER_DB` 自定义账本路径（默认 `~/hualing-leads/group-ledger.db`，随现有备份策略一起备份）。
3. nginx 对 `/console` 与 `/api/console` 加 IP 白名单（皇冠资产最小暴露）：`location ~ ^/(console|api/console) { allow <运营IP>; deny all; proxy_pass ...; }`。
4. 首次上线后执行一次历史回填：`node scripts/ledger-backfill.mjs`（读生产 orders-db.json/leads-db.json，幂等可重复跑）。
5. 授权台账导入（在厂商机/部署机上执行导出，产物拷到 VPS 导入）：
   - avatarhub 机：`python tools/license_ledger/export_avatarhub.py --out ah.json`
   - chengjie 部署机：`python tools/license_ledger/export_chengjie.py --out cj.json`
   - VPS：`node scripts/ledger-import-licenses.mjs ah.json && node scripts/ledger-import-licenses.mjs cj.json`（幂等，可周期重跑）
6. 双写自此自动生效：新订单/留资/状态变更实时镜像进账本；账本坏了不影响下单（best-effort 已验证）。
