## 二、三件事的落地细节

### 1. 全站私域化（不被搜索/公开推广）

- `robots.ts`：`{ userAgent: "*", disallow: "/" }`——从原来的 `allow: "/"`（允许全站索引，与"私域"自相矛盾）反转。
- `layout.tsx`：`metadata.robots = { index:false, follow:false, nocache:true, googleBot:{index:false,follow:false} }`——即使某爬虫无视 robots.txt，页面级 meta 也拦。
- 删 `sitemap.ts`：不再向爬虫暴露页面清单。
- **效果**：站只能靠"拿到链接的人"访问，不进 Google/Bing 收录，符合"只做私域"。

### 2. 中英按系统/手机语言自动

- 逻辑：① 用户手动切过 → 记住（`hl-lang`）；② 否则读 `navigator.languages`，**任一为中文**（`zh*`）→ 中文，**其余一律英文**。
- 契合"东南亚华人 + 中英双语"：中文系统手机看中文，英文/马来/泰语等系统看英文（国际默认）。
- `/en` 显式英文路由保留（分享英文链接用）。
- **已知小瑕疵**：首屏 SSR 默认中文，英文用户会有一次极快的中→英切换闪烁。彻底消除需加 Next middleware 读 `Accept-Language` 头在服务端定语言（列为可选增强，未做，避免引入 hydration 复杂度）。

### 3. 域名防封机制（多域名 + 短链 + 稳定触点）

设计原则：**Telegram 是永不被封的锚点，域名是可替换的消耗品。**

- **`lib/domains.ts`（单一事实源）**：
  - `MIRROR_DOMAINS`：镜像域名池，环境变量 `NEXT_PUBLIC_MIRROR_DOMAINS` 配置（逗号分隔），回落已知 `usdt2026.cc`。
  - `PRIMARY_DOMAIN`：本次部署主域名，环境变量 `NEXT_PUBLIC_PRIMARY_DOMAIN`。
  - `STABLE_TOUCHPOINTS`：Telegram 客服/频道/群/Bot/小程序（来自 `site.ts`）——最终兜底入口。
  - `SHORTLINK_TARGETS`：短码→目的地表（客服/频道/站内页），改一处全站生效。
- **`/api/active-domain`**：返回当前主域名 + 镜像池 + 触点。**Bot/分发脚本查它拿"现在该发哪个地址"**——域名换了，分发侧自动跟随。
- **`/go/<slug>`**：统一短链。外部触点 302 到 t.me（跨域名不失效），站内页 302 到相对路径（自动跟随当前访问域名，换镜像零改动）。未知短码回落客服，绝不 404 丢客户。
- **`FallbackNotice`**：页脚常驻，写明"此域名打不开就回 Telegram 拿新址"，列频道/群/客服，并 localStorage 记住用户上次能打开的域名。

## 三、运营怎么用这套防封机制（部署说明）

1. **多镜像部署**：把同一套站部署到多个域名（如 `a.cc / b.net / c.app`）。每个部署设环境变量：
   ```
   NEXT_PUBLIC_MIRROR_DOMAINS = "a.cc,b.net,c.app"
   NEXT_PUBLIC_PRIMARY_DOMAIN = "a.cc"   # 该镜像自己的域名
   ```
2. **分发只用两种链接**：
   - Telegram 触点（`t.me/...`，永不失效）——放在所有物料的"找回入口"。
   - `/go/<slug>` 短链——日常分发用。
3. **Bot 接入**：Telegram Bot 定期请求某个活着的镜像的 `/api/active-domain`，把"当前主域名"写进 Bot 的自动回复/频道置顶。域名被封时，运营改还活着的镜像继续服务，Bot 播报新址。
4. **换域名 SOP**：注册新域名 → 部署镜像（设 env）→ 更新其他镜像的 `MIRROR_DOMAINS` → Bot/频道播报 → 旧域名弃用。用户永远能从 Telegram 找回来。

> 注：站内所有跳转应尽量用**相对路径**（已确认 Footer/落地页用相对 `Link`），这样换域名时站内链接零改动。绝对 URL 仅用于 Telegram 触点。

### ⚠ 关键风险：3 域名同一 IP

三个域名都解析到 **165.154.233.121**。这意味着：
- ✅ 能防「**域名/DNS 级封禁**」——某个域名被墙/被 registrar 下架，换另一个域名仍可访问（同一台服务器）。
- ❌ 防不了「**服务器 IP 级封禁**」——如果 165.154.233.121 这台服务器被 GFW 封 IP 或被机房因投诉下架，**三个域名会同时挂掉**。

**真正抗封的两个方向（建议下一步）**：
1. **分散源站**：镜像分别部署到不同 IP/不同机房/不同国家，域名池指向不同 IP。
2. **前置 Cloudflare（推荐，成本低）**：三个域名都套 Cloudflare，隐藏真实源站 IP、只暴露 CF 的 IP；封禁者拿不到源站 IP，且 CF 自带 DDoS/IP 轮换。这是私域站最常用、性价比最高的抗封做法。

> 现状（3 域名同 IP）比单域名强，但只是"域名消耗品"层面的冗余，不是"服务器抗封"。Telegram 触点仍是最终兜底。
