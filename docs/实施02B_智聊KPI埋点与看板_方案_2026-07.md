**日期**：2026-07-12
**阶段**：实施第二阶段 A 类（纯设计/待评审）
**目标**：把销售话术里全是 TODO 的 KPI，变成"从代码真实数据算出来、进看板、能对客户讲"的数字。这是 SKU-B 智聊能卖动的前提（三视角报告市场 P0）。

## 零、一句话结论

> **销售要讲的数字分两类：一类"运营效果"（加粉成功率/打招呼回复率/引流转化/AI 自动处理/D7 留存）现在就能从代码算出来，只差接成一张卡——这是本阶段最高性价比；另一类"单客获取成本 CAC / 渠道归因 UTM"代码里完全没有，必须补一个"成本录入 + 归因贯通"的最小模块。最难的"从 FB 触达到 LINE 成交"的跨仓统一漏斗，卡在两个仓库物理分离 + 事件命名不一致，放到最后做。**

**关键结论**：获客侧（mobile）和承接侧（telegram）**各自都已有相当完整的看板**，缺的不是"算不出来"，是"没拼成一张销售能一眼看懂的脸"，且**两张脸中间断裂**。

### ❌ 完全缺、必须补

| 缺口 | 影响销售话术 | 严重度 |
|---|---|---|
| `lead_cost` / CAC（单客获取成本） | 讲不了"每个客户花多少钱获取"——B 端最关心的 ROI | 🔴 高 |
| `utm_*` 渠道/素材归因 | 无法按渠道拆漏斗、无法证明"哪条广告有效" | 🟠 中 |
| `campaign` 与事件关联（`campaigns` 表脱节） | 无法按活动维度算效果 | 🟠 中 |
| `source` 未进 `fb_contact_events` | 来源只能到 friend_requests 层，事件流看不到 | 🟡 低 |
| B 侧 `first_text_received`/`platform_tag`（契约与代码不一致，实际是 `line_first_reply`） | 跨仓转化率 join 算不出 | 🔴 高（卡跨仓） |
| 跨仓 `canonical_id` ↔ `contact_id` join 未工程化 | 无"触达→成交"单一漏斗 | 🔴 高（卡跨仓） |

## 二、三层 KPI 看板设计

```
┌─ L1 运营效果卡（本阶段做，现成数据）──────────────────────┐
│  获客侧：加粉成功率 / 打招呼回复率 / 引流转化率           │
│  承接侧：AI 自动处理% / D7 留存 / 活跃关系 / 人均轮次      │
│  → 分两段讲，各自数据源现成，最快见效                     │
├─ L2 成本与归因（阶段2，补埋点）────────────────────────┤
│  CAC 单客成本 = 期间总成本 / 新增有效客户                 │
│  渠道漏斗：按 utm_source/campaign 拆 L1 各指标            │
│  → 需补"成本录入模块 + utm 贯通"                          │
├─ L3 全链路统一漏斗（阶段3，跨仓打通）──────────────────┤
│  提取→加友→通过→打招呼→引流→[跨仓]→LINE首复→成交→D7留存  │
│  → 需统一 B 侧事件名 + platform_tag + 跨仓 join           │
└──────────────────────────────────────────────────────┘
```

## 三、阶段 1（本阶段主攻）：运营效果卡 — 现成数据接一张脸

**思路**：不追求跨仓统一（那是阶段3），先把**已经能算的数字**聚成一个销售/客户都能看的"效果概览"，分"获客"和"承接"两段诚实呈现。

**最小改动方案（低风险，可先做）**：
1. **不改任何统计逻辑**，只做一个**只读聚合端点**，把现有几个端点的关键数字汇到一起：
   - mobile 侧：新增 `GET /analytics/sales-headline` —— 内部调 `get_funnel_metrics()` + `referral_funnel()`，吐出 `{加粉成功率, 打招呼回复率, 引流发送数, 引流回复数, 引流转化率, 漏斗阶段计数}`。
   - telegram 侧：`GET /api/workspace/roi` 已有 automation + relationship 段，直接用。
2. **一页"销售效果卡"**：mobile dashboard 加一个 headline 卡（复用现有 dashboard 模板），或做一份静态"数字快照导出"（CSV/截图）供销售 PPT 用。
3. **口径固定**：每个数字标注"统计口径 + 时间窗 + 样本量"，避免销售乱讲（如"加粉成功率=通过/发出，近 30 天，样本 N"）。

**为什么先做这个**：零 schema 迁移、零埋点、不碰运行中服务的写路径，**1–2 人日**就能让销售第一次有真实数字可讲。

**验收**：能导出一份"近 30 天获客效果 + 承接效果"数字快照，每个数字可溯源到端点。

## 四、阶段 2：补成本(CAC) + 归因(UTM) — 让 ROI 叙事成立

### 4.1 CAC 成本录入模块（缺口最大、最值钱）

代码里**完全没有成本概念**。最小可用设计：
- 新增轻量表 `acquisition_cost`（mobile 侧 `openclaw.db`）：`period, cost_type(ad/labor/device/proxy), amount, currency, scope(device/preset/campaign), note, at`。
- 成本**手工录入**（老板/运营每周填广告费、人力、设备摊销）——不追求自动采集，先能算。
- CAC = `SUM(acquisition_cost.amount in window) / COUNT(有效新客 in window)`；"有效新客"口径建议用 `stage_friend_accepted` 或 `wa_referral_replied`（按销售定义拍板）。
- 端点 `GET /analytics/cac?since=&until=&scope=`。

### 4.2 UTM / 渠道归因贯通

- `fb_target_personas` / 引流话术 / 落地页注入 `utm_source/utm_campaign`，随 `fb_contact_events.meta_json` 落库（**利用现成的 meta_json 自由字段，零 schema 迁移**）。
- `record_contact_event` 补 `source` 进事件（当前只在 friend_requests 层）。
- 端点支持按 `utm_source/campaign` 分组返回 L1 指标。

> 优先级：**CAC > UTM**。CAC 是"讲不了就没法谈 ROI"的硬缺口；UTM 是"锦上添花的拆分"。

## 五、阶段 3：跨仓全链路统一漏斗（最难，放最后）

**障碍（探查实证）**：
1. 两仓物理分离（各自 SQLite/服务），无共享 fact 表。
2. **事件语义漂移**：`INTEGRATION_CONTRACT.md` 契约写 `first_text_received`，telegram 代码实际写 `line_first_reply`；`platform_tag` 字段仅文档、代码未实现。
3. 身份桥接：mobile 有 `canonical_id`，telegram 有 `contact_id` + token merge，但无现成跨仓转化率 SQL/API。

**方案**：
1. **先对齐事件名**（低风险纯代码）：telegram `gateway.on_line_first_text` 在写 `line_first_reply` 时**追加**一个契约别名 `first_text_received` + 补 `platform_tag`（加法、不破坏既有）。这是跨仓 join 的前提。
2. **建跨仓 ETL / BI 视图**：一个独立只读脚本（不塞进任一主服务），按 `INTEGRATION_CONTRACT.md §7.7.3` 的双键 `(peer_canonical_id, source_platform)` + 时间窗，join A 的 `wa_referral_replied` 与 B 的 `first_text_received`，算跨仓转化率，落一张 BI 快照表。
3. **一页统一漏斗**：提取→加友→通过→打招呼→引流发送→引流回复→[跨仓]→LINE 首复→成交→D7 留存，中间断层用 BI 快照补上。

**判断**：阶段3 工作量大且需两仓协调，**不建议 MVP 期做**；阶段1+2 的数字已足够支撑销售起步。等有真实客户、要证明端到端 ROI 时再做阶段3。
