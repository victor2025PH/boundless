# 渠道账号架构：Telegram / WhatsApp / Messenger / LINE 是通用还是独立？

> 日期：2026-07-19 ｜ 结论先行：**平台之间必然独立（凭据体系不同），产品实例之间必须独立（会话冲突+风控），
> 但客户看到的品牌必须统一（同名同头像同话术）。推荐"总机 + 产品线服务号"矩阵。**

---

## 〇、引擎事实（先核对了代码再下结论）

| 渠道 | 接入方式 | 登录态形态 | 归属 |
|---|---|---|---|
| Telegram | MTProto 协议号（Telethon） | `sessions/<号码>.session` 文件 | **实例数据根**（智聊/通译各自的 `sessions/`） |
| WhatsApp | baileys 侧车（WA 多设备协议，扫码配对） | 侧车自己的 auth 目录 | 每实例独立起一份侧车 |
| Messenger | messenger-web 侧车（网页会话） | 浏览器会话档案 | 同上 |
| LINE | LINE RPA（桌面自动化）+ 状态文件 | RPA 绑定的桌面登录 | 同上 |
| 网页客服 web_chat | 站点 widget，无账号概念 | — | 实例配置开关 |

**硬约束**：同一个 `.session` 文件被两个进程同时使用 → Telegram 互踢、消息互抢、触发风控。
所以"智聊和通译共用一个 TG 号"在技术上就是不成立的——**一号一实例是铁律**。
统一收件箱做的是**聚合**：一个实例内的多渠道（TG+WA+Messenger+LINE+web）会话都进同一个工作台。

## 一、三视角分析

### 1. 开发工程师视角——"账号 = 文件级资产，隔离是生命线"

- 登录态是文件（session/auth 档案），属于实例数据根；备份数据根 = 备份账号资产
  （这次能找回老 TG 号 `639952947442.session` 就是证明）。
- 一号多进程 = 高危风控信号（IP/设备指纹漂移），封号还会连坐同 IP 其它号。
- 各平台凭据体系完全不同（TG=手机号+api_id/hash、WA=扫码配对、FB=账号 Cookie、LINE=桌面登录），
  **没有任何"通用账号"的技术可能**；能通用的只有：品牌素材、话术库、AI 人设、知识库（这些走配置分发）。
- 引擎已预留扩量能力：智聊有账号编排器（`platform_login.orchestrator`，协议号多开池），
  扩量期打开即可管理多号轮换，现在不用。

### 2. 市场经理视角——"号 = 流量入口 = 归因单元 = 风险单元"

- 智聊（AI 成交）与通译（翻译工作台）客群不同：同一个号接待两种业务，定位混淆、
  转化漏斗互相污染、广告归因算不清。
- **独立号 = 干净归因**：每个投放渠道/产品线用独立号，封一个不伤全局（风险分散）。
- 但号太多养不动（每个号要养权重、要有人应答）——初期矩阵要**精简**。
- 品牌资产统一：所有号统一 ∞ 头像、命名规范（无界 · XX / Boundless XX）、签名与话术
  （bot-knowledge 单一口径）——客户在任何触点看到的都是同一家公司。

### 3. 用户（客户）视角——"我只想加一个号把事办完"

- 号多且杂 = 像骗子矩阵，不信任；一个号问啥都行 = 方便，但响应慢显得不专业。
- 最佳体验参照银行模式：**客服总机**（随时有人应答、AI 秒回）+ **客户经理**
  （成交后的专属对接号，处理交付/售后）。
- 客户换平台（TG 聊完转 WA）时希望"你还认识我"——这靠统一收件箱的跨渠道身份归并
  （contacts/gateway 已有 channel_identity 机制），不靠共用账号。

## 二、推荐方案：「总机 + 产品线服务号」矩阵（初期 3 个号起步）

| 号 | 平台 | 挂载实例 | 职责 |
|---|---|---|---|
| **官方总机**（复用老号 +639757135247，session 已找回） | Telegram | 智聊实例 | 官网/广告统一入口；AI（销售 KB + 真人顾问人设）自动接待、留资、报价、导给人工 |
| **交付服务号**（新开或备用号） | Telegram | 智聊实例（不同 account） | 已成交客户的交付/售后群；不接陌生流量 |
| **web 客服** | 官网 widget | 智聊（auto_ai）/通译（manual） | 已上线，零账号成本 |

扩展节奏（按市场推进）：
1. 东南亚市场起量 → 开 **WhatsApp**（baileys 侧车挂智聊实例），LINE 视日泰市场再开；
2. 通译产品的坐席客户 → 通译实例**不直接持有获客号**：通译是工作台工具，客户的号登在
   客户自己的部署里；我们的通译演示号仅内部用；
3. 任何一个号做大后（日咨询 > 200），用账号编排器拆分流量池。

**纪律三条**：
1. 一号一实例，登录态跟数据根走，备份数据根即备份号；
2. 所有对外号统一品牌素材与话术口径（改口径 = 改 bot-knowledge/KB，一处改处处生效）；
3. 新号必须登记：哪个平台、哪个实例、哪个用途、谁保管手机号——进集团控制台台账（后续加"渠道账号"页）。
