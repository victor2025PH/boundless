# 实施 10 · 合规隔离：换脸业务迁往独立域 13x.lol（独立域名 + 独立主体）

**日期**：2026-07-18
**目的**：把换脸/直播换脸（及未来的情感陪伴 companion）从合规主站 `bd2026.cc` 物理隔离到独立域 `13x.lol`，实现"低风险主站干净可推广、高风险业务独立域独立主体运营"。
**范围**：本次代码改动仅动 SEO/索引层（sitemap / robots / 页面 noindex + 单一真相 `lib/isolation.ts`）；域名注册、收款主体、部署分流属运维/法务动作，见第五节清单。

---

## 一、为什么必须隔离

1. **监管收紧**：2026 年 deepfake/换脸类产品是全球监管焦点——欧盟 AI Act 对合成媒体的透明标注义务已生效，美国多州（加州、德州等）出台 deepfake 专门法，各主流平台（Google/Meta 广告、App 商店）对 face swap 类目一律从严审查甚至直接拒登。
2. **支付红灯**：卡组织（Visa/Mastercard）与主流收单机构把 face swap / deepfake 列为高风险 MCC 审查对象；一旦主站域名与换脸产品绑定，整个主体的收款通道都可能被风控冻结或拒绝入网。
3. **连坐效应**：主站上翻译、语音克隆（合规话术下）、数字人、同传都是可正常推广的低风险产品线。只要 `/face` 还挂在 `bd2026.cc` 的可索引面上，广告投放尽调、支付通道尽调、域名信誉评级都会把整个站按最高风险项目打分——低风险业务被高风险业务连累，这是最大的商业损失。
4. **尽调视角**：广告平台与收单方的尽调是"按注册域 + 按主体"进行的：同一注册域下的任何页面、同一收款主体下的任何业务，都会被合并评估。这决定了隔离必须做到**注册域**和**法律主体**两层，光藏页面没用。

## 二、推荐架构（三方案对比）

### 方案 A：主域二级子域（如 `face.bd2026.cc`）——不推荐

- 子域与主域共享**同一注册域**（registrable domain / eTLD+1），Google 质量评估、广告尽调、支付风控都按注册域聚合。
- 同一主体、同一域名信誉池：子域出事，主域连坐；主域投广告，尽调会扫到子域内容。
- 结论：形式上"分开了"，实质隔离为零。**仅当业务本身低风险、只是想做内容分区时才用子域**，与本案目标不符。

### 方案 B：独立注册域 `13x.lol` + 独立公司/收款主体 + 互不链接——推荐

- **独立注册域**：`13x.lol` 与 `bd2026.cc` 无注册域关联，各自独立的 robots.txt / sitemap.xml / 搜索控制台资产，域名信誉互不传导。
- **独立法律主体 + 独立收款**：换脸业务用单独公司注册收款账户（卡收单、USDT 地址均独立），主站的支付主体尽调不会扫到换脸流水；反之换脸侧被风控也不波及主站通道。
- **互不链接**：主站导航、落地页、页脚一律不出现 `13x.lol` 的链接；`13x.lol` 也不回链主站。爬虫与尽调无法通过链接图把两个资产关联起来。
- **母品牌层**：「无界科技 BOUNDLESS」作为母品牌可在商务层面统一背书（销售话术、客服体系可共用），但**支付主体、域名、可索引页面三者必须分离**——这是"品牌统一、风险隔离"的边界线。

### 方案 C：`13x.lol` 下按产品再分二级域（如 `swap.13x.lol` / `live.13x.lol`）+ 准入访问——可选增强

- 在方案 B 的独立域内部，用子域把图片换脸、直播换脸等产品再分区，配合准入（gated access：邀请码/协议签署后才能访问定价与下载）。
- 好处：单个产品出问题可以只下线一个子域；准入机制本身也是合规抗辩点（非公开传播、用户已签署授权承诺）。
- 成本：多一层部署与证书管理，早期流量小时收益有限。

### 推荐结论

**以 B 为主，C 视规模叠加**：先落地"独立注册域 + 独立主体 + 不互链"，这是隔离有效性的底线；等 `13x.lol` 上产品线增多、单产品风险差异变大时，再按 C 拆二级域与准入分层。方案 A 明确排除。

## 三、代码侧已落地（本次，2026-07-18）

| 文件 | 动作 | 说明 |
|---|---|---|
| `website/lib/isolation.ts` | 新增 | 合规隔离单一真相：`ISOLATED_DOMAIN = "13x.lol"`、`ISOLATED_SITE_URL`（可用 `NEXT_PUBLIC_ISOLATED_URL` 覆盖）、`ROUTE_VISIBILITY`（当前仅 `"/face": "gated"`）、`GATED_SLUGS` / `isGatedSlug()` / `isPublicSlug()`。零依赖，任何层可安全引用 |
| `website/lib/seo.ts` | 编辑 | 新增 `publicPages()` = `SITE_PAGES` 过滤掉 gated slug；`indexableUrls()`（IndexNow 推送数据源）改为基于 `publicPages()`。`/face` 仍保留在 `SITE_PAGES`（页面存在、hreflang 查询可用），只是不再进可收录清单 |
| `website/app/sitemap.ts` | 编辑 | 改为消费 `publicPages()`——`/face` 与 `/en/face` 不再出现在 sitemap.xml |
| `website/app/robots.ts` | 编辑 | disallow 列表在原有 `/admin`、`/api/`、`/robot-stage` 基础上追加所有 gated slug 及其 `/en` 版本（当前即 `/face`、`/en/face`），由 `GATED_SLUGS` 派生，后续新增 gated 路由自动生效 |
| `website/app/face/page.tsx` | 编辑 | metadata 合并 `robots: { index:false, follow:false }`，页面内容不变 |
| `website/app/en/face/page.tsx` | 编辑 | 同上（英文版） |

**三层防线**：① sitemap/IndexNow 不再主动申报 → ② robots.txt disallow 阻止抓取 → ③ 页面 meta noindex 兜底（即使有外链导致爬虫直达，也不收录）。页面本身未删除，拿到链接的老客户/受控渠道仍可直达。
