**日期**：2026-07-11
**范围**：`D:\workspace` 全目录
**用途**：技术负责人 / 老板内部决策 + 实施依据
**版本**：V1.0

## 一、最重要的结论

这个文件夹不是 6 个独立项目，而是"一条社交获客业务线"的完整工作区：真正自研的核心系统只有 **2 个**，其余是同一项目的副本、第三方引擎和文档。当前最大的问题不在功能，而在**磁盘冗余、副本混乱、密钥裸奔、合规几乎为零**。

三个最该立刻处理的事实：

1. **`telegram-mtproto-ai`、`ai-baseline-check`、`ai-p0-integration` 是同一个 Git 仓库的三份拷贝**（remote 都是 `yunkeji2026/telegram-mtproto-ai`），不是三个项目。三份合计约 4.1GB，主副本有 134 处未提交改动未落盘。
2. **`index-tts` 占了 15.9GB（全工作区约 78%）**，是第三方开源 TTS 引擎（`index-tts/index-tts`，浅克隆），其中模型权重 8.6GB + 虚拟环境 7.2GB。它是 `telegram` 项目的语音克隆后端依赖，不该与源码同层，更不该纳入备份/上传。
3. **明文密钥散落多处**：根目录 `服务器密码.txt`、`config.yaml` 真实 API Key、WhatsApp `creds.json` 真实账号凭证、`cluster.yaml` 的 `shared_secret`、`.env` 的数据库密码、`device_registry.json` 的真实设备指纹。这是当前最高优先级风险。

## 三、三视角分析

### 视角 A — 开发工程师（能不能长期维护）

优点：功能完整度高，两系统合计约 50 万行、约 9000 个测试用例、有 CI 矩阵和完整架构文档，FB 获客全链路与多平台承接均为真实现。

核心技术债：

| 问题 | 证据 | 影响 |
|---|---|---|
| 上帝文件 | `facebook.py` 13194 行、`inbox/store.py` 6719 行、`executor.py` 6633 行、`skill_manager.py` ~5900 行、`web_i18n.py` ~15668 行 | 无法 review、改一处牵全身 |
| 入口臃肿 | `main.py` ~4098 行把所有启动编排堆在一起 | 启动逻辑脆弱 |
| 仓库卫生差 | 根目录 40+ 个 `investigate_*/tmp_*/check_*/_*.py` 调试脚本、大量 `.xml`/`.png` dump、9 份 `*_PLAN.md` | 现场调试驱动开发，脚本从不回收 |
| 依赖不锁版本 | 两项目 `requirements.txt` 全 `>=`，无 lock 文件 | 环境与 CI 易漂移 |
| 单进程 + SQLite | 文档明确"同一 config 不可多进程写" | 水平扩展需重构 |
| 副本失控 | 3 份 telegram 拷贝 + 主副本 134 处未提交 | 改动分散、随时丢代码 |

### 视角 B — 市场经理（能不能拿出去卖）

已具备：`ai-p0-integration` 已有 Next.js + Tailwind 营销落地页（含语音/换脸/口译页），P0 分支做了"免费试用字符额度、养号风控包装成卖点、客户资产页"；`docs-business` 已有完整 CAC/CPL/ROAS 测算与 MVP 预算（USD 8k–20k）。

短板：

| 问题 | 说明 |
|---|---|
| 卖点分散，无统一叙事 | 客服/陪伴/换脸/口译混杂，客户不清楚核心价值 |
| KPI 全是 TODO | `CAPABILITIES.md` 加粉成功率、响应时间、转化率全空，销售无数据可讲 |
| 合规硬伤卡海外 | GDPR/CCPA 几乎为零，欧美 B2B 尽调过不了 |
| 业务定性敏感 | 虚拟人设+情感陪伴+批量私信引流属高风险，限制市场/渠道选择 |

### 视角 C — 用户 / 运营（好不好用、稳不稳、安不安全）

优点：`mobile-auto0423` 有 `start/stop/status.bat` 三件套 + 一键迁移 + 完整 RUNBOOK 故障字典，退出码规范。

痛点：

| 问题 | 证据 | 后果 |
|---|---|---|
| 密钥全裸奔 | `服务器密码.txt`、`config.yaml`、WhatsApp `creds.json`、`cluster.yaml`、`.env` | 拿到目录=拿到全部账号和服务器 |
| 合规能力≈0 | 全库无 consent/GDPR/CCPA/数据删除实现；`opt_out` 只是营销漏斗状态 | 投诉/DSAR 即封号+法律风险 |
| 默认弱口令 | `web_admin.auth_token: admin`、Dashboard 默认 `123456`、API Key 空时完全开放 | 后台裸奔 |
| 端口配置打架 | `.env`=8000、`launch.env`=18080、`cluster.yaml`=18080 | Worker 心跳指错端口 |
| 平台封号风险内生 | `human_behavior.py` 目标"avoid detection"，批量加友违反 ToS | 账号随时批量掉 |

## 四、优化方案（按优先级）

> 涉及删除/移动大文件和副本的操作有破坏性，执行前逐条确认，采用"隔离而非删除"。

### P0 — 本周必做（安全 + 止血）

1. **密钥治理（最高优先级）**：移出 `服务器密码.txt`；轮换已暴露凭证；敏感配置加入 `.gitignore` 并从跟踪移除；改掉默认弱口令。
2. **磁盘瘦身（回收约 15GB+）**：`index-tts` 移出工作区到 `D:\_runtime_assets`，用环境变量指向；`.venv` 可删重建，`checkpoints` 单独存放，永不进备份/上传。
3. **副本收敛**：主副本 134 处未提交先落盘；`ai-baseline-check` 确认无改动后删除；`ai-p0-integration` 保留并尽快合回 main。

### P1 — 本月（工程化 + 可维护）

4. 依赖锁定（pip-tools/freeze 产出 lock）。
5. 拆巨石（`main.py` → `bootstrap/`；`facebook.py`、`inbox/store.py` 分模块）。
6. 仓库卫生（调试脚本收进 `scripts/scratch/`；pre-commit 拦截误提交；端口配置单一源）。
7. 文档去噪（`*_PLAN.md` 归档，根目录只留 README + 当前架构）。

### P2 — 本季度（架构 + 商业化 + 合规）

8. 合规最小可用层（consent、opt-out、数据保留 TTL、DSAR 删除、AI 身份披露）。
9. 归因字段入库（source/campaign/ad/utm/lead_cost，打通 CAC/CPL/ROAS）。
10. 架构演进（PostgreSQL + 消息队列替换 SQLite 单写假设）。
11. 统一产品叙事，填上真实 KPI。

## 五、开发建议（落地路线）

- 先安全、再瘦身、后收敛，最后才谈功能。
- 按 `docs-business` MVP 思路：1 国家 + 1 平台 + 1 链路，30–45 天验证真实 CAC。
- 把 `ai-p0-integration` 商业化成果合回主干，停止长期多副本开发。
- 合规不是文案是产品能力，MVP 阶段并行搭最小合规层。
- 业务定性提前与法务确认，明确市场与渠道红线。
