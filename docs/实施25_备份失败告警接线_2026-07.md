# 实施25 · 备份失败告警接线（静默失败 → 主动 Telegram 通知）

> 日期：2026-07-19 ｜ 上承：实施24（集团库备份上线 / 两机冗余 / 恢复演练）
> 主题：实施24 的备份失败此前只落日志——**静默失败=虚假安全感**。本轮把失败接进 VPS 现有
> Telegram 告警通道，让"备份没跑成"能主动通知到人。

## 一、复用现有告警通道（不另造）

VPS `health-watchdog.sh` 已有成熟的 `notify()`：Telegram Bot API，收件人 = `.env.local` 的
`ALERT_CHAT_ID` ∪ `hualing-leads/admin_chats.json` 绑定的 chat；`tr -d '\r'` 容忍 CRLF env。
现状核查：`TELEGRAM_BOT_TOKEN` 已配、`admin_chats.json` 有 1 个绑定 chat（告警通道是**活的**）。

**抽出复用**：把该 notify 逻辑独立成 `scripts/notify-telegram.sh "消息"`，供备份及未来任何 cron 复用，
避免每个脚本各抄一份告警代码。

## 二、备份 cron 包装器

`scripts/ledger-backup-cron.sh`：
- 跑 `ledger-backup.mjs --out-summary`，捕获输出与退出码进 `~/ledger-backup.log`；
- **失败**（退出码非 0）：截末 3 行错误 → `notify-telegram.sh` 发告警 → 落 `~/.ledger-backup.down` 幂等标志；
- **成功**：静默（不刷屏，与 health-watchdog 同哲学）；若此前失败过（down-flag 在）→ 补发
  「已恢复」并清标志；
- crontab 从"裸 node 命令"切到本包装器（每日 03:15）。

## 三、三路径实测（VPS 实跑）

| 路径 | 结果 |
|---|---|
| 成功（真备份） | 静默 exit 0，无告警 ✅ |
| 失败（不可写目录 /root/nope） | exit 1 + 告警发出 + `~/.ledger-backup.down` 落盘 ✅ |
| 恢复（失败后再成功） | 发「已恢复」+ 清 down-flag ✅ |
| 通道自测 | `notify-telegram.sh "🧪 自测"` 直发绑定管理员 chat，exit 0 ✅（老板 TG 应收到测试条） |

> 边界确认：缺库（LEDGER_DB 指向不存在文件）被 ledger-backup.mjs 判 `skipped_missing`
> **不算失败**（VPS 初期语义），不误告警——只有真备份/校验失败才告警。

## 四、相对方案的二次优化

1. **抽公共 notify 而非内联**：备份告警与未来 cron（KPI 周报失败、grants_sync 失败等）可共用一个通道脚本，
   一处维护；
2. **down-flag 幂等**：备份连续失败不会每天刷屏，只在"失败→恢复"边沿各发一次（与 health-watchdog 对齐）；
3. **成功静默**：避免告警疲劳（每天一条"备份成功"会让人忽略真告警）；
4. **skipped_missing 不告警**：区分"库还没建"与"备份真失败"，不制造假警报。
