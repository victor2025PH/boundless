## 三、核查结论（对原报告的修正）

### 修正 1：telegram 项目没有 git 密钥泄露

原报告担心 `config.yaml`、WhatsApp `creds.json` 泄露。实测：`telegram-mtproto-ai/.gitignore` 已完善忽略 `config.yaml`、`config.local.yaml`、`.env.local`、`sessions/*.session`、`models/`、`data/`、所有 `*.db`、`*.wma`、`voice_samples/` 等；WhatsApp baileys 目录 git 只跟踪 `package.json` / `package-lock.json`（不含密钥）。**结论：telegram 本地有真实密钥（运行必需，正常），但未泄露到 git。**

### 修正 2：git 密钥泄露只在 mobile-auto0423，且已推送

以下 5 个文件被 git 跟踪，且已推送到 origin 的 main + 约 94 个远端分支：

| 文件 | 敏感内容 |
|---|---|
| `config/cluster.yaml` | `shared_secret: 80dc965e...`（集群通信密钥，已泄露到远端） |
| `config/device_registry.json` | 真实设备序列号 / android_id / 内网 IP |
| `config/device_aliases.json` | 真实设备别名 / Wi-Fi ADB 地址 |
| `config/cluster_state.json` | Worker 内网 IP |
| `config/notify_config.json` | 1600+ 行设备断连历史（含数十个真实序列号 + 内网 IP） |

**关键推论**：因为已推送到远端，仅 `git rm --cached` 只能防未来，**无法消除已泄露的历史**。真正止血靠"轮换密钥值让旧值失效"。

### 修正 3：index-tts 正在运行，ai-baseline-check 已消失

- `index-tts` 是**独立 TTS 服务**（`aitr_indextts2_server.py` 常驻 `127.0.0.1:7899`，`start_indextts2.bat` 守护），telegram 通过 HTTP 调用它。此刻进程在跑，**不能移动/删除**。
- `ai-baseline-check` 在核查期间已消失（worktree 检出被回收），冗余副本少一个。
- `ai-p0-integration` 干净（0 未提交）、含 11 个有价值提交，**保留待合并**。

## 七、磁盘瘦身执行记录（2026-07-11，已完成）

### 执行方式（比原 delete_reclaim 方案再优化）

原方案设想：删 `.venv` + 把 `checkpoints` 移到其他盘，之后语音功能需改配置指向新路径 + 重下模型。
**实际优化为**：`checkpoints` 用「迁移 + 目录联接（junction）」，做到**移出 D 盘物理空间但对程序完全透明、零配置改动、模型权重不用重下**。

| 步骤 | 操作 | 结果 |
|---|---|---|
| 1 | 调 `POST /api/voice/local-tts/toggle {enabled:false}` | 用程序自身 API 关闭 coupled autostart（不手改文件） |
| 2 | `taskkill /PID 24180 /T /F` 停 TTS 进程树 | 7899 无监听，未被重新拉起，`main.py`(24476) 存活 |
| 3 | `rmdir /s /q index-tts\.venv` | 删除 7.05GB 虚拟环境（25s） |
| 4 | `robocopy checkpoints F:\_runtime_assets\...\checkpoints /MOVE` | 8.45GB 模型权重迁到 F 盘（27s） |
| 5 | `mklink /J D:\workspace\index-tts\checkpoints F:\...\checkpoints` | 建跨盘 junction，原路径透明穿透 |

### 结果

- **D 盘净回收 15.63GB**（83.2GB → 98.83GB 可用）
- `index-tts` 在 D 盘物理占用从 15.9GB 降到 ~37MB（仅源码/assets），`checkpoints` 走 junction 指向 F 盘
- **主链路全程未中断**：`main.py` 未重启，文字 AI 客服/获客承接不受影响
- **语音功能当前降级**：本机 IndexTTS2 已停，`local_autostart.enabled=false`；语音走云端 fallback 或暂不可用

### 如何恢复本机语音克隆功能

```powershell
# 1) 重建虚拟环境（uv 按 uv.lock 装依赖，需联网下 torch 等，约 7GB 回到 D 盘）
cd D:\workspace\index-tts
uv sync
# 2) 通过程序 API 重新开启 coupled 托管（main.py 会自动拉起 TTS）
$h=@{Authorization="Bearer <你的web_admin.auth_token>"; "Content-Type"="application/json"}
Invoke-RestMethod -Uri "http://127.0.0.1:18799/api/voice/local-tts/toggle" -Method POST -Headers $h -Body '{"enabled": true}'
# 3) 验证：/api/voice/local-tts/status 应回 reachable=true, model_loaded=true
```

> 注：`checkpoints`（模型权重）已永久移至 F 盘且**无需重新下载**，junction 会让 `uv run` 和 TTS server 照旧从 `D:\workspace\index-tts\checkpoints` 读到。恢复语音只重建 `.venv` 依赖即可。若彻底不用本机语音，保持现状即可，云端 fallback 兜底。
