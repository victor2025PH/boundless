## 一、人设总线闭环（P5）——三引擎 purge 执行器全部到位

上一轮只有 website 侧注册表与机器同步通道；本轮补齐引擎侧消费者，全域清除协议**首次跑通完整闭环**。

| 引擎 | 导出器 | 清除执行器 |
|---|---|---|
| avatarhub | 已有（实施11） | `engines/avatarhub/persona_purge_agent.py` |
| chengjie | `tools/persona_bus/export_chengjie_personas.py`（AI 人设/话术/术语库） | `engines/chengjie/scripts/persona_purge_agent.py` |
| huoke | `tools/persona_bus/export_huoke_personas.py`（养号人设，脱敏） | `engines/huoke/src/persona_purge_agent.py` |

**执行器统一设计**（三引擎同款）：
- 消费机器通道 `GET/POST /api/sync/personas/purges?system=<engine>`（Bearer EVENT_INGEST_KEY）；
- **缺省 dry-run 演练**（只打印将删清单，不删不回执），`--commit` 才真删+回执，`--once`/`--loop N` 轮询；
- **软删除**：删除物先移入 `secrets|config|data/purged_trash/<日期>/`（保留回收期，真正物删交运维），而非直接 unlink——这是比契约更稳妥的姿势，已建议补进契约；
- 安全护栏：source_key 白名单正则、路径 resolve 后确认在引擎根内、异 source_system 指令拒执行、commit 删除失败明确报错不 ack（绝不谎报清除完成）、回执未达则下轮补发不重复删。

**闭环实证**（本机集成冒烟）：导入 3 个 avatarhub 人设 → 控制台授权幻影 → 发起全域清除（status=purge_pending）→ 引擎执行器 dry-run 正确不动 → `--commit` 回执 → persona status=purged、机器队列清空。跨引擎目标推导（avatarhub/chengjie/huoke）正确。

## 二、KPI 口径固化（P4 收口）

- `platform/observability/KPI_DEFINITIONS.md`：把全域漏斗（线索→订单→支付→激活→活跃→留存→续费）的**唯一口径**写死——每阶段用哪个事件/账本状态计数、按 customer_id 还是 event_id 去重、时间窗口归属；激活跨引擎合并去重定义；各产品"活跃"定义与北极星指标；数据来源边界（行为来自 events 库、钱权来自 ledger 库，**内容与生物特征永不参与**）；口径变更流程（改口径=改本文件+通告，禁止各报表各算）。
- `website/scripts/kpi-weekly-report.mjs`：全矩阵周报生成器（只读双库聚合），五段——全域漏斗+环比、分产品分区、授权与到期、Top 事件、数据健康告警（某产品 0 事件即提示埋点未上报）；库缺失/空时输出骨架报告并给回填指引，不报错。验证含假数据断言与空库骨架两条路径。
