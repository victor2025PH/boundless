## 一、客户归并实时化（双写钩子即建档）⭐

**问题**：`ensureCustomerForOrder/Lead` 原本只"匹配已有客户"（`linkCustomer`），从不建档——
所以全新联系方式的订单永远没有客户主档（customers=0 的根因）。批处理脚本
`ledger-link-customers.mjs` 能建，但要人工/定时跑，不实时。

**方案**：增强 `ensureCustomerForOrder/Lead`（双写钩子 `ledger-sync.ts` 已在每次入账时调它们，
无需改钩子）：匹配未命中且存在**强信号**（`@handle` / `t.me/handle` / email / phone / tg id）时
**自动建客户主档 + 挂身份 + 回填 customer_id + 记 auto_create 审计**；弱信号/自由文本/仅
fingerprint **不建档**（避免热路径产生垃圾客户）。分类规则与批处理脚本 `classifyContact` 同源。

**安全**：`createCustomer`/`attachIdentity` 各自内置事务、顺序调用（不嵌套事务，规避
better-sqlite3 限制）；双写钩子本就 try/catch 兜底，建档失败绝不阻断下单/留资主链路。

**TS 自测（tsx，8 项全 PASS）**：强信号 handle 单建档 → 同 handle 二单匹配不重建 →
自由文本单不建档 → 留资 tg id 建档+tg 身份 → 幂等重入不重复。

## 二、is_test 升为正式 schema 列（v6 迁移）

- 实施21 的 `ledger-mark-testdata.mjs` 用裸 ALTER 给生产库加了 is_test，但 DDL 未声明——
  fresh DB / 迁移路径不会有该列，报表查询 `WHERE is_test=0` 会崩。
- 本轮把它升为**正式迁移 v6**（ledger.ts + ledger-lib.mjs 两侧同步）：**函数式条件迁移**——
  先 `PRAGMA table_info` 探列再 ALTER，生产库（列已被脚本抢先加）重跑幂等不报
  duplicate column。fresh 与"已有列"两路径实测均 version=6、is_test 就位。

## 三、报表/商机排除测试数据

- **商机引擎**（`opportunities.ts`）：单一 choke point `collect()` 末尾用 `testCustomerIds`
  过滤——测试客户不再生成任何跨售/续费信号；旧库无列时返回空集不误伤。
- **KPI 周报**（`kpi-weekly-report.mjs`）：`testExclClause` 按表探列生成
  `AND COALESCE(is_test,0)=0` 片段，注入 leads/orders/licenses 全部漏斗与缺口查询；
  只读快照无列时空串降级。

**生产实证**：部署后 schema_version=6；orders is_test=1 为 9/9；KPI 漏斗
线索/订单/支付从 9/5/1 → **0/0/0**（测试噪声已排除），激活 3（来自 6 条真实授权）保留。
首份真实 KPI 周报口径就此干净。

## 四、相对方案的二次优化

1. **不改双写钩子、只增强 ensure 函数**：`ledger-sync.ts` 早已在调 `ensureCustomerFor*`，
   把建档能力下沉到函数内，实时/批处理/回填三条路径一次全通，改动面最小；
2. **强信号才建档**：热路径若无差别建档会从脏 contact 造垃圾客户；只认 handle/email/phone/tgid，
   与批处理脚本philosophy 一致；
3. **迁移函数式而非裸 DDL**：直面"生产列已被脚本抢先加"的现实，条件 ALTER 幂等，
   避免 v6 迁移在生产炸 duplicate column；
4. **过滤集中在单 choke point**：商机过滤只在 `collect()` 末尾一处，三类规则全覆盖，
   不必逐规则改 SQL、不漏。
