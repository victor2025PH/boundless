# 实施28 · 引擎机运维脚本同步（绕开 LFS/脏树）+ 哨兵每日心跳

> 日期：2026-07-19 ｜ 上承：实施27（哨兵铺满三机；遗留「引擎机 git pull 被 LFS 挡」）
> 主题：把 .176/.198 的「靠 scp 应急同步」升级为可重复的路径级同步，并给哨兵加可选每日心跳。

---

## 一、根因复盘（比「装 git-lfs」更深一层）

试 pull 后发现阻塞不止 LFS：

| 机 | 表象 | 真因 |
|---|---|---|
| .176 幻声 | behind 57 + Aborting | **脏工作区约 89 文件** + 未跟踪脚本会挡住 merge；LFS 已装仍可能 smudge |
| .198 智拓 | Aborting | 实施27 **scp 落下的** `cron_sentinel.ps1` 成 untracked，会被 merge 覆盖而拒绝 |

结论：引擎机整仓 `git pull` 不适合当常规运维动作——业务不需要 showcase 视频，工作区也常不干净。

## 二、替代方案（本轮落地）

新增 `deploy/cron/sync_ops_scripts.ps1`：

1. `GIT_LFS_SKIP_SMUDGE=1` + 本 clone 永久 `filter.lfs smudge/process --skip` + `lfs.fetchexclude *`；
2. `git fetch origin main`；
3. **仅** `git checkout origin/main -- deploy/cron`（不移动 HEAD、不碰其它脏文件）；
4. 打印对齐的 remote SHA 与 `cron_sentinel.ps1` 字节数，便于核对。

`cron_sentinel.ps1` 增加可选每日心跳：

- `-Heartbeat` 或机器级 `SENTINEL_HEARTBEAT=1`；
- 全绿日首次经 `/api/ops/alert` 发 💚；flag 落 `%LOCALAPPDATA%\boundless-sentinel\heartbeat.YYYY-MM-dd`；
- 14 天前 flag 自动清理；无密钥只写日志。

README §6 写入上述口径。

## 三、两机落地动作

- 配 LFS skip（local config）→ 跑路径同步（或 scp 新脚本后再 sync）→ 开心跳 env → 手跑哨兵验证；
- 哨兵任务仍为 SYSTEM / 每 30 分钟；脚本更新后无需重装任务（同路径覆盖即可）。

## 四、相对原计划的二次优化

原计划：「装 git-lfs / `GIT_LFS_SKIP_SMUDGE=1` 后整仓 pull」。

本轮二次优化：

1. **路径级同步替代整仓 pull**——同时避开脏树与 LFS，且不动引擎机上无关 WIP；
2. **永久 skip 写进 clone 级 git config**——不必每次靠临时环境变量；
3. **心跳用 env 开关**——不必改计划任务 XML 即可三机统一打开；
4. **保留 HEAD 不强制 reset**——避免误抹引擎机本地未提交改动。
