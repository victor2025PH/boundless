# 实施26 · Windows 五机 cron 哨兵 + VPS 告警中继端点

> 日期：2026-07-19 ｜ 上承：实施25（VPS 备份失败告警）
> 主题：VPS cron 已能告警，但 **Windows 引擎机的 uploader/grants_sync/purge/export 等任务失败完全静默**
> ——数据停流的盲区。本轮用"VPS 中继 + Windows 哨兵"闭合，且不在各机分发 TG token。

---

## 一、架构决策：告警走 VPS 中继，不在 Windows 存 TG token

- Windows 机（.117/.176/.198）已持 `EVENT_INGEST_KEY`（uploader 上报用），但**没有** TG bot token；
- 与其把 token 分发到多台 Windows（扩大密钥暴露面），不如加一个 VPS 中继端点：
  **`POST /api/ops/alert`**（Bearer `EVENT_INGEST_KEY`）→ 转发到 Telegram 管理员 chat
  （`getAdminChats()`：env `TELEGRAM_CHAT_ID` ∪ `admin_chats.json`，与 health-watchdog 同源）。
  token 只留 VPS，密钥面最小化；人类口令体系（CONSOLE_KEY）完全独立。
- 生产实证：无鉴权 401；带密钥 `{ok:true,sent:1,recipients:1}`（老板 TG 收到自测条）。

## 二、Windows cron 哨兵

`deploy/cron/cron_sentinel.ps1`：
- 巡检本机 `\Boundless\` 所有计划任务的 `LastTaskResult`；非 0 且非良性码
  （267009 正在运行 / 267011 从未运行）即判失败；可选超期检测（`-StaleHours`，缺省 26h）；
- 有失败 → 经 `/api/ops/alert` 告警 + 落 down-flag；成功静默；失败后再全绿 → 补发「已恢复」清 flag
  （与 VPS ledger-backup-cron 同款幂等语义）；
- 用 `EVENT_INGEST_KEY`（机器 env）鉴权，基址缺省 `https://bd2026.cc`；不在本机存任何 TG 凭据。

## 三、三路径 + 端点 实测（.117）

| 路径 | 结果 |
|---|---|
| 中继端点鉴权 | 无 key 401 / 有 key sent=1 ✅ |
| 全绿 | 12→13 任务全绿，静默 exit 0 ✅ |
| 失败（造 rc7 测试任务） | 检出 `rc7` → 告警 sent=1 + down-flag ✅ |
| 恢复（删测试任务） | 补发「已恢复」+ 清 flag ✅ |
| SYSTEM 上下文注册运行 | `Boundless-cron-sentinel` 每 30 分钟，LastTaskResult=0 ✅ |

## 四、坑与修复

- **PS 5.1 无 BOM 读中文脚本按 GBK 崩**（实施19 同款）：`cron_sentinel.ps1` 落盘后补 UTF-8 BOM
  才能 `-File` 执行——已入档为 Windows cron 脚本的硬约束。
- SYSTEM 账户 `LOCALAPPDATA` 与 Administrator 不同 → 哨兵的 flag/log 落 SYSTEM profile 下，
  与手动运行（Administrator）各一份；功能不受影响（各账户独立幂等），已知不影响告警。

## 五、相对方案的二次优化

1. **中继而非分发 token**：一个 VPS 端点服务全部 Windows 机，token 零下放，密钥面最小；
2. **复用 EVENT_INGEST_KEY**：不引入新密钥，Windows 机现成就有；
3. **检"任务结果"而非"事件流"**：go-live 前无真实流量，事件流量检测会误报；查计划任务
   LastTaskResult 是与流量无关的可靠信号，现在就能用；
4. **良性码白名单 + down-flag 幂等 + 成功静默**：避免"正在运行/从未运行"误报与告警疲劳。
