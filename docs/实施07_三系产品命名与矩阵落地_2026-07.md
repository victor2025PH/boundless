## 一、命名体系（同系列，请老板拍板）

洞察：原产品命名其实**已隐含字系规律**——幻颜/幻声/幻影都是「幻」字，通译是「通」字，智聊是「智」字。据此体系化为**三系**，母品牌「无界」统领，每系一个中文字领衔，英文统一 `…X` 后缀；并补齐此前缺失的两个软件（私域获取、同声翻译），命名与原系列完全统一。

| 产品系 | 中文 · 英文 | 产品（中文 · 英文） | 对应软件 | 新增? |
|---|---|---|---|---|
| **智连系** | 智连 · Growth | 智拓 ReachX | 私域流量获取（真机 RPA 获客，mobile-auto0423） | 🆕 新增 |
| | | 智聊 ChatX | AI 聊天翻译坐席（telegram-mtproto-ai） | 沿用 |
| **幻境系** | 幻境 · Studio | 幻颜 FaceX | 换脸 | 沿用 |
| | | 幻声 VoiceX | 声音克隆 | 沿用 |
| | | 幻影 LiveX | 实时直播换脸换声 + 数字人 | 沿用 |
| **通达系** | 通达 · Lingo | 通译 LingoX | 实时聊天翻译 | 沿用 |
| | | 通传 VoxX | 同声传译（会议/直播口译，异机） | 🆕 新增 |

命名逻辑：**「字系」区分类别**（智=获客成交 / 幻=生成分身 / 通=跨语），**「双字 + …X」保持系列感**。三系正好对应无界「打破沟通与成交 / 容貌声音身份 / 语言」三重界。

## 三、深入思考中发现并处理的连锁问题（本轮再优化）

1. **加产品 = 连锁类型依赖，必须一并补齐**：`ProductKey` 是 `keyof BRAND.products`，加 2 个产品后，所有 `Record<ProductKey, …>`（图标/锚点/schema锚点/视图路由 4 处）若不补新 key 会**直接 TS 编译失败**。已全部补齐（grep 核对：`PRODUCT_IMG`/`PRODUCT_ANCHOR`/`PRODUCT_SCHEMA_ANCHOR`/`PRODUCT_VIEW` 均含 7 key）。这是"加一个产品牵一发动全身"的典型，方案阶段看不到、落地才暴露。
2. **"五条产品线"文案散落 8+ 处**：不止 UI，还在 bot 欢迎语、system prompt、每日文案生成 prompt、TG 命令描述里。只改 UI 会导致"矩阵显示三系、客服却说五条"的口径分裂。已全站同步。
3. **新产品无专属图标 → 会裂图**：`public/brand/products/` 只有 5 张图。新产品暂复用同系图（reachx→chatx 图、voxx→lingox 图）并注释"待补专属图标"，避免上线裂图。**待办**：补 `reachx.png`/`voxx.png`（原图标由 `scripts/build-boundless-marks.ps1` 生成）。
4. **底座卡位置优化**：原来"无界底座"是产品网格里的第 6 张卡（与产品平级，逻辑上不对——它是所有产品的共享底座）。分系后改为**网格下方独立横幅**，语义更准（底座托起三系）。
