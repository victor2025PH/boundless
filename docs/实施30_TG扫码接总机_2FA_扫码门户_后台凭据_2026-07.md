## 一、根因：为什么之前每次「扫了没反应」

铁证——事发时段智聊实例侧**零扫码痕迹**：`sessions/` 无新 `tg_login_*.session`、
`account_registry.db` 停在上次、`app.log` 无 `login/start`。据此定位两层病根：

| # | 现象 | 真因 |
|---|---|---|
| 1 | 扫码毫无反应 | 老板扫的是**下午遗留的静态过期二维码**（PNG 被图片查看器/旧浏览器页缓存）；后端没有活跃登录会话在等它。旧桥 `tg_backend_scan.py` 的 HTML 状态文字写死「等待扫码…」永不更新，扫成功与否也看不出来 |
| 2 | （潜在）扫码确认后卡住 | 引擎 `TelegramQrLogin` **不支持两步验证**——无 `SESSION_PASSWORD_NEEDED → CheckPassword` 分支；开了云密码的账号扫码确认后 poll 抛异常被标 `expired`，体感同样是「没反应」 |
| 3 | 扫成功也不接待 | `platform_login.orchestrator_enabled` 未开——号只落注册表，无 worker 拉起 |

## 二、修复落地（三层一次做全）

### 2.1 引擎补两步验证（2FA）— 主代码，UI/门户双受益

`engines/chengjie/src/integrations/telegram_protocol_login.py`：
- 新增状态 `password_needed`（介于扫码确认与授权之间的稳定等待态，`poll()` 在此短路不再打 `ExportLoginToken`）；
- `TelegramQrLogin.submit_password()`：走 pyrogram 高层 `check_password()`（内含 SRP），成功走与扫码成功一致的收尾（导出 `session_string`、抽身份、落盘、置 `authorized`）；密码错误保持 `password_needed` 可重试；
- 判定用 `_is_password_needed` / `_is_bad_password`（按异常类名 + 错误串双判，跨 pyrogram 版本稳）；
- `_finish` 重构为 `_finish_with_user`，token 成功与密码成功共用收尾；
- provider 暴露 `submit_password` 回调，扫码/密码两条路径都幂等落注册表。

`engines/chengjie/src/integrations/platform_login.py`：
- `LoginSession` 加 `submit_fn` 字段；`password_needed` 态**不受 TTL 过期影响**（`is_expired()` 返回 False、`_gc_locked` 不回收）——防等密码时会话被误清。

`engines/chengjie/src/web/routes/unified_inbox_login_routes.py`：
- 新增 `POST /api/platforms/{platform}/login/{login_id}/password`（body `{"password":"..."}`），密码错可重试，成功落库；
- start/status 透传 `submit_fn` 与 `password_needed` 状态。

单测 `tests/test_telegram_protocol_login.py`：原 6 + 新 6 = **12 全过**（覆盖判定纯函数、submit_password 成功/错密/非等待态、poll 短路）。

### 2.2 扫码门户 `deploy/instances/tg_scan_portal.py` — 根治体验

一个**独立本地 HTTP 服务**（默认 127.0.0.1:18790），与浏览器同源（`/state`、`/password` 的 fetch 无 file:// CORS 限制）：
- **码永远新鲜**：后台驱动线程持续 start/poll，`expired` 自动换码——绝不给静态旧码；
- **状态实时可见**：页面每秒轮询 `/state`，显示 等待/已扫/需两步验证/成功/换码；
- **两步验证自助**：`password_needed` 时页面直接弹密码框，老板本地输入 → 门户转交后端 `/password`，**密码不落中转日志**；
- 只调实例既有登录 API（Bearer=web_admin.auth_token）——session/registry/编排器全由引擎管。
