## 二、修复落地

### 2.1 status_instances.ps1 数据根自动探测（问题2）

每实例按序取第一个命中：**① 显式参数 → ② 端口持有引擎进程的 `AITR_DATA_DIR`**（start 脚本 cmd 链注入，
上溯父进程命令行解析，运行中最可信）**→ ③ 生产缺省 `D:\chengjie-instances\<实例>\data`**（仅当其
`config\config.yaml` 存在才认，防在别的机器误指）**→ ④ 仓库缺省**（向后兼容）。
输出增加 `root=<路径>（来源）` 行与 JSON 字段 `data_root/data_source/engine_owned`（供 watchdog 消费）。
验证：无参跑 = 双 GO、root 显示生产根（来源=进程）；显式传参行为不变。

### 2.2 watchdog_instances.ps1 探活自愈（问题1）

新增 `deploy/instances/watchdog_instances.ps1`，装成 `\Boundless\Boundless-chengjie-watchdog`（每 5 分钟）：

- **DOWN**（端口不在听）→ `start_<实例>.ps1 -DataDir <探测根>` 幂等拉起，等端口就绪（≤90s）；
- **假活**（在听但 HTTP 无响应）→ 连续 3 轮（≈15 分钟）才 `stop_instance` + `start` 强制重启，防启动中误杀；
- **端口被非引擎进程占用** → 只告警绝不清杀（与 start/stop 同一防呆哲学）；
- 告警经 `/api/ops/alert`（Bearer=EVENT_INGEST_KEY），失败边沿 + down-flag 幂等 + 恢复补发，与哨兵同款；
  **每次真实拉起动作必告警**（宕机事件要让人知道）；flag/日志落 `%LOCALAPPDATA%\boundless-watchdog\`；
- 退出码 0=全活（含本轮自愈成功）/1=未能自愈/2=探测故障——哨兵按 `\Boundless\` 前缀顺带巡检 watchdog 自身。

**账户 = 装机用户 S4U + Highest（安装器强制，不吃 `-RunAs`）**：引擎继承 watchdog 的账户，SYSTEM 会让引擎
改以 SYSTEM 身份跑（用户身份漂移）；S4U 跑 session 0、不存密码、不依赖交互登录——开机 ≤1 节拍自动拉起、
注销不掉。`install_tasks.ps1` 新增 watchdog 任务类型（chengjie 专属，缺省任务集自动含，同 config_snapshot）。
