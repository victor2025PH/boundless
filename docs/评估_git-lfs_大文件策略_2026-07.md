# 评估 · Git LFS 与大文件策略（2026-07）

> 日期：2026-07-19 ｜ 类型：技术评估（全程只读核查，本文未改动任何仓库配置）
> 背景：近两周官网连续入库展示媒体——campaign 传播图（PNG）、showcase 实录音视频
>（mp3/mp4）、开场 BGM（theme.mp3），单次提交最高 +21.4MB，引出「要不要上 git-LFS」
> 的疑问。本文先核实仓库体积、.gitattributes 现状与三类消费方式（生产部署 / CI / 开发机），
> 再评估三个候选方案，给出明确结论与触发重评的条件。

---

## 一、事实核查（2026-07-19，分支 feat/website-positioning）

### 1.1 仓库体积

`git count-objects -vH` 与补充口径：

| 指标 | 数值 | 说明 |
| --- | --- | --- |
| size-pack | **3.30 MiB** | 仅反映已打包对象；近期提交大量对象仍为松散状态，此数**严重低估** |
| 松散对象 | 172.68 MiB（6454 个） | 含未 gc 的近期媒体提交与可回收冗余（prune-packable 225） |
| 全部可达对象 | **130.74 MiB** | `git rev-list --disk-usage --objects --all`，本文采用的权威口径 |
| GitHub 远端 size | **约 121 MiB**（123900 KB） | `gh api repos/victor2025PH/boundless`，不含 LFS 存储 |
| 本地 .git 目录 | 340.51 MB | 含 LFS 本地缓存 69.71 MB、松散冗余等，gc 后会明显缩小 |

结论性读法：**仓库真实历史体积约 131 MiB（远端约 121 MiB），距 1GB 量级差一个数量级**。
单独引用 size-pack=3.30MiB 会误判「仓库极小」，引用 .git=340MB 会误判「已经很大」，均不可取。

### 1.4 .gitattributes 现状：LFS 前向规则已存在（本次核查最重要的事实修正）

仓库**并非「没有任何 LFS 规则」**。`a3f205c`（monorepo 初始化期）写入的 `.gitattributes`
已含前向 LFS 规则：`brand-assets/fonts/*.otf|ttf`、`*.psd`、`*.sketch`、`*.mp4`、`*.mov`。
且**实际在生效**：

- `git lfs ls-files` 共 **16 个文件**：4 个字体（约 51.7 MB，含 18MB 的 NotoSansCJKsc-Black.otf）
  + **12 个 showcase mp4（约 20.9 MB）**；本地 LFS 缓存 `.git/lfs` 共 69.71 MiB，两相吻合；
- 07-19 新入库的 showcase mp4 因命中 `*.mp4` 规则**自动走了 LFS**（提交时未必有人察觉）；
- **png / jpg / mp3 无 LFS 规则**：campaign PNG、theme.mp3、real/*.mp3 均为普通 git 对象
  （`git check-attr filter` 确认 unspecified）；
- 小瑕疵：.gitattributes 末尾存在一段重复的规则行（fonts/mp4/mov/psd 重复声明），无害但
  下次动该文件时顺手清理。

换言之：**「方案 B（前向 LFS）」已经对 mp4/字体部分落地**，真正待决策的只是
「要不要把 png/mp3 也纳入 LFS，或反过来收缩」。

### 1.5 消费方式：谁在读这个仓库

| 消费方 | 方式 | 与 LFS 的关系 |
| --- | --- | --- |
| 生产部署 | `website/scripts/deploy.ps1`：tar 打包**本地工作树**（排除 node_modules/.next/.git）→ SCP → 服务器 deploy.sh 原子部署。**不走 git clone** | tar 打的是工作树里的**实体文件**。前提：部署发起机装有 git-lfs 且 smudge 已实体化（当前开发机 git-lfs 3.7.1 已装，工作树 mp4 均为实体，前提成立） |
| CI | 唯一 workflow `.github/workflows/website-intro-qa.yml`：`actions/checkout@v4` 默认 **lfs:false** → showcase mp4 在 runner 上是**指针文件** | 当前不炸：QA 只验收 intro 动效（粒子/按钮），`next build` 不读取 public 内文件内容。**若未来 CI 要验收视频页播放，必须 checkout 加 `lfs: true`** |
| 开发机 | 直接工作树 | 需装 git-lfs（已装）；新开发机/新部署机漏装时，clone 出来的 mp4 是指针文件——此时跑 deploy.ps1 会把**指针文件打进 tarball，线上视频成建制 404**（theme.mp3 404 事故的放大版，是现状下最真实的运维风险点） |

## 二、三方案评估

### 方案 A：现在启用 LFS 并迁移历史（git lfs migrate）——否决

- **收益微薄**：可达对象仅 131 MiB，其中字体/mp4 大头（约 72 MB）已在 LFS；`migrate`
  能再挤出的只有 png/mp3 及其历史版本，量级约 15–20 MB；
- **代价极高**：`git lfs migrate import` 全量重写提交历史——所有协作者必须重新 clone、
  所有开放 PR 的 base 全断、与本仓库「多会话并行开发」的工作模式直接冲突（并行分支
  引用旧 SHA，强推后全部孤立）；
- 远端 121 MiB 无任何平台压力（GitHub 建议 <1GB、硬限 100MB/单文件），**毫无迁移紧迫性**。

结论：代价/收益严重倒挂，**否**。

### 方案 B：扩大前向 LFS（把 png/mp3 也纳入 .gitattributes）——不扩大，已有规则维持

- 现状已是「部分 B」：mp4/mov/psd/sketch/字体有前向规则且验证生效（见 1.4）；
- **扩大到 `*.png` 不可行**：website/public 的 png 多数是 <300KB 的 logo/icon/favicon，
  一刀切会把大量小图 LFS 化（LFS 对小文件纯属负担）；按目录写精细规则（如仅
  brand/campaign）则规则碎片化、后续目录调整极易漏；
- **跨环境运维成本是硬成本**：CI 需 `lfs: true`（多拉 LFS 带宽）、每台开发/部署机必须装
  git-lfs、GitHub LFS 免费额度（1GB 存储 / 1GB 月带宽）成为一个新的需盯配额——而当前
  普通对象里的大媒体（png/mp3，>500KB 口径）合计约 18 MB，买不到对应收益；
- **不回退已有规则**：已入 LFS 的约 72 MB（字体+mp4）若迁回普通对象，要么重写历史
  （同方案 A 的代价）、要么产生一次「指针换实体」的大提交且丢掉 LFS 的按需拉取优势，
  两头不讨好。维持现状即可。

结论：**不扩大、不回退**；沿用现有规则，新增 mp4/字体自然走 LFS。

### 方案 C：策略防线（推荐，作为主基调）

不动版本库机制，用约定+工具防线控制体积增长：

1. **单文件 >5MB 提交前需评估**（以本文档为约定）：评估内容=能否压缩、能否外链
   （对象存储/CDN）、是否运行时产物。当前普通对象最大文件 2.6MB，天花板尚有余量；
2. **运行时产物一律 .gitignore**：已有先例——`website/public/brand/campaign/teaser-war-latest.png`
   （cron 每周覆盖生成，已 ignore，生成逻辑在 `website/lib/war-card.ts`）。新增同类
   （定时生成图、导出报表、录屏）照此办理；
3. **展示媒体压缩预算**：mp4 单个 ≤3MB（现 12 个全部达标，最大 2.96MB 已贴线）、
   传播 PNG 单张 ≤1.5MB（bg-loong-clouds.png 2.6MB 超预算，列为下次压缩复查对象）、
   mp3 单个 ≤2MB（现最大 theme.mp3 1.6MB 达标）；
4. **每季度资产体检**：复查死资产（无引用文件）与超预算资产——07-19 已有一次体检先例
   （`5eaf9cc` 提交据体检结果放行 21 项展示资产），后续可固化进 `tools/repo_doctor` 或
   独立脚本，季度跑一次；
5. **补两条现状护栏**（针对 1.4/1.5 核实出的既有 LFS 风险）：
   - 新机器上手 checklist 加一条：装 git-lfs → `git lfs pull` → 抽查
     `website/public/videos/showcase/*.mp4` 为实体文件（>100KB）而非指针，再允许跑 deploy.ps1；
   - CI 一旦引入依赖媒体内容的验收（视频页截图/播放检查），`actions/checkout` 同步加
     `lfs: true`，并留意 LFS 带宽配额。

## 三、结论

**采用方案 C（策略防线）为主基调；已有的前向 LFS 规则（mp4/mov/psd/字体）维持不动、
不扩大到 png/mp3、不做历史迁移。**

数字支撑：

- 仓库可达对象 **131 MiB**、GitHub 远端 **约 121 MiB**——距 1GB 触发线差一个数量级，
  「体积失控」不成立；
- 大头已被既有 LFS 接走：字体+mp4 约 **72 MB** 在 LFS；普通对象里的大媒体（png/mp3，
  >500KB 口径）合计约 **18 MB**，普通对象最大单文件 **2.6 MB**（bg-loong-clouds.png）；
- 消费侧无 clone 型强依赖：生产走 tarball（不 clone）、CI 浅检出且 lfs:false（反而省带宽）、
  开发机长期持有工作树——LFS 扩面在每个环境新增「装 git-lfs + 配置 + 配额」的运维成本，
  当前规模下换不来收益；
- 两周 +31.6 MiB 属于 showcase 实录一次性补库（修 404）+ 双节庆传播图集中产出，
  非稳态增速，不构成改机制的依据。

## 四、触发重评的条件

满足任一条即重新评估（届时优先考虑扩大前向 LFS，历史迁移仍需单独论证）：

1. 可达对象（`git rev-list --disk-usage --objects --all`）或 GitHub API size 超过 **1 GiB**；
2. 出现 **>20MB 单文件**的入库需求（参照系：GitHub 警告线 50MB、硬限 100MB；库内现存
   最大 LFS 文件为 18MB 字体）；
3. 团队出现 **clone/fetch 带宽痛点**（新协作者首次 clone 过慢、新增全量 clone 型 CI/CD）；
4. CI 开始依赖 LFS 媒体实体内容（须 `lfs: true`，同时评估 LFS 月带宽配额是否够用）；
5. 展示媒体进入稳态高增长（连续两个月每月 +50MB 以上，而非本次的一次性补库）。
