## 三、对接开发方案（独立运行 + 整体链条）★核心

> 目标：**每个项目都能各自独立跑起来（断网/断总线也能用），接上总线又自动串成一条链。**
> 实现这一点的核心是两条铁律：**契约先行**、**总线可选**。

### 3.1 两条设计铁律

**铁律一 · 契约先行（Contract-first）**
产品/引擎之间**横向零 import**，任何跨产品交互只走 `platform` 契约（HTTP API / 事件 schema / SKU 注册表）。这沿用无界既有依赖规则（`website/products/engines → platform`，`platform` 绝不反向依赖）。好处：任一产品替换实现、单独升级、单独部署都不牵动别人。

**铁律二 · 总线可选（Bus-optional）**
每个产品启动时**探测无界总线是否可达**（一个环境变量 `BOUNDLESS_BUS_URL` + 健康探针）：

- **探测到** → 进入「联网/链条模式」：登录走统一身份、授权走 SKU 台账、线索推给承接中台、AI 回复调克隆音、埋点进统一漏斗。
- **探测不到** → 自动降级「单机/独立模式」：用本地实现（本地登录、本地卡密、本地 AI、无语音赋能），功能不缺失只是不联动。

> 这条铁律是"独立可运行"的技术保证：**总线是增强项而非依赖项**。任何单个项目 clone 下来配好自身环境即可运行，不需要先起全家桶。

### 3.2 五大平台契约（对接的骨架）

| 契约 | 做什么 | 关键接口（建议签名） | 谁提供 | 谁消费 |
|---|---|---|---|---|
| `identity/` 身份·资产总线 | 统一账号 SSO；克隆一次声音/形象→跨产品复用 Profile | `getProfile(userId)` · `bindVoice/Face(profileId, asset)` · `listAssets(profileId)` | avatarhub + 统一认证 | 全部产品 |
| `licensing/` 授权·计量·收款 | edition→额度→按量扣减→开通；卡密/订单/分销 | `checkLicense(machine, sku)` · `meter(userId, sku, units)` · `gate(feature, edition)` | **智控王 license_server + wallet 上升实现** | 全部产品 + 官网 |
| `leadbus/` 线索总线（新增） | 上游获客→中游承接的标准化线索流 | 事件 `lead.captured` · `POST /leads/ingest` | 智控王 / huoke | chengjie 承接坐席 |
| `enable/` 赋能能力网关（新增） | AI 回复配克隆音、翻译、数字人 | `POST /tts/clone-speak` · `POST /translate` · `POST /avatar/render` | avatarhub / 通达 | 智控王 / chengjie |
| `observability/` 可观测 | 统一漏斗埋点，打通 CAC/CPL/ROAS | `emit(event, props)` · `funnel(step)` · `latency(stage, ms)` | 全部产品 | 官网 console 看板 |

> `leadbus` 与 `enable` 是本融合**新增的两条契约**（无界现有 5 契约里没有显式的线索总线与赋能网关，它们正是"产业链条"落到接口层的体现）。

### 3.3 产业链数据流（一条线索的完整旅程）

```
[TG智控王] jiso 群搜索 / 成员提取 / 关键词捕获
      │  emit lead.captured { platform:"tg", userId, profile, intentScore, source }
      ▼
[leadbus] 标准化线索事件（带来源归因 utm / 意向分）
      │
      ▼
[chengjie] 承接坐席分配 → AI 多角色对话
      │  需要拟人语音？ → enable: POST /tts/clone-speak {text, profileId}
      │  客户说外语？   → enable: POST /translate {text, to}
      │  ← avatarhub 返回克隆音 wav / 通达返回译文
      ▼
[chengjie] 生成回复 → 回执给触达执行层
      │  （TG 平台由智控王执行发送：文本 / 语音消息；其他平台 chengjie 自发）
      ▼
[成交] emit deal.closed → 引导下单
      │
      ▼
[licensing] 卡密/订单开通（智控王 wallet + license_server 实现）
      │  meter(userId, sku, units) 计量扣减；分销返点结算
      ▼
[observability] 全链路漏斗：lead.captured → chat.started → deal.closed → payment.settled
      │
      ▼
[官网 console] 老板看真实 KPI（CAC / CPL / 转化率 / ROAS）
```

### 3.4 独立运行保证（降级矩阵）

每个项目在「单机模式 / 链条模式」下的行为对照——**证明各自都能独立运行**：

| 项目 | 单机模式（总线不可达） | 链条模式（接入总线） |
|---|---|---|
| **TG智控王** | 本地登录、本地卡密、本地 AI 自动聊，纯 TG 获客+成交（现状即如此） | 无界 SSO、SKU 台账授权、线索推给 chengjie、AI 回复调克隆音、埋点进统一漏斗 |
| **chengjie 承接** | 本地承接坐席，自采线索 | 消费 `leadbus` 线索、调 `enable` 赋能、授权走 licensing |
| **avatarhub** | 独立 GPU 服务，本地授权（license_admin 离线签发） | 作为 `enable` 与 `identity` 资产总线的提供方 |
| **huoke 获客** | 独立真机 RPA，本地跑 | 线索 emit 到 `leadbus` |
| **官网 bd2026.cc** | 静态展示 + 本地留资 | console 消费 observability、SKU 反查 licensing |

**降级实现要点**：每个契约客户端都实现「探针 + 本地兜底」。参考无界 `platform/compliance/client.py` 已有的"**可降级瘦客户端**"范式——总线在就转发，不在就返回本地默认。全部新契约客户端按此范式做。

### 3.5 代码组织与部署形态

- **代码组织**：TG智控王收编进无界单仓（统一版本 / CI / 提交纪律），但**架构上保持总线可选**。这同时满足"代码治理统一（单仓）"与"运行时各自独立（总线可选）"。
- **部署形态**：沿用无界五机集群拓扑；每个引擎有独立 Docker / 启动脚本；`BOUNDLESS_BUS_URL` 决定是否入链。
- **单仓纪律**（必须遵守，防副本分叉）：开工先 `git pull --rebase`、收工先 `commit+push`、改码机与算力机边界锁死、每台 Cursor 工作区默认只开自己主攻的 `engines/<x>`。

## 六、风险与红线

1. **能力重叠是最大工程债**：智控王与 chengjie 的 TG 能力必须先划边界再合并，否则维护成本翻倍、重演副本分叉。
2. **合规隔离是生死线**：智控王的 jiso 群搜索、批量加友，以及 `wushizhifu.txt` 记录的四方/违规支付通道属最高危红线。这些能力可保留，但**必须走 `ai2026.co` 独立域+独立主体+独立支付**，绝不能污染合规主站 `bd2026.cc`，否则拖垮整个矩阵的支付与尽调。
3. **总线可选不能变成总线依赖**：每个契约客户端必须有本地兜底，任一产品都要能脱离总线独立启动——这是"各自独立运行"的底线。
4. **单仓纪律**：收编后严守 pull/push 与改码机/算力机边界纪律。
5. **技术债不叠加**：两边都有上帝文件与不规范 DB 连接，合并时按域拆分，别把债搬到一起。
6. **监管趋势**：2026 年换脸/深度伪造立法、EU AI Act 合成媒体标注持续收紧——换脸/批量获客收窄隔离，克隆音产出保留 C2PA 可验真水印。

### 9.14 一个需要业务方决策的重要发现（非本次融合项目范畴，性质紧急；2026-07-20 已实证）

复核 `license_server.py` 支付相关代码时，**独立于 Phase 2 的既有审计、亲自重新逐行验证**，确认了一个可能影响真实收款的问题：`handle_create_payment`（创建订单，`/api/payment/create` 端点）里，写入 `orders` 表的 INSERT 语句用的列名（`product_id`/`duration_type`/`duration_days`/`coupon_id`）与 `database.py` 里 `CREATE TABLE orders` 定义的列名（`product_type`/`product_duration`/无`duration_days`列/`coupon_code`）**不一致**；带优惠券下单还会先在 `SELECT * FROM coupons WHERE code=...` 命中另一处列名不一致（建表是 `coupon_code`/`is_active`/`expire_at`/`total_count`，查询用的是 `code`/`status`/`expires_at`/`max_uses`）。两处都被同一个 `try/except` 包住，不会打崩服务进程，但会让下单请求收到 500 错误。

**2026-07-20 二次核实（用户要求先核实再进入第 5 阶段）——从"推测"升级为"实证"**：上一版结论只基于阅读代码里的 `CREATE TABLE` 语句，明确标注了"未访问真实数据库、可能与生产 schema 不同"的保留。这次直接找到并**只读**查询了本机真实在用的数据库文件（`backend/data/tgmatrix.db`，`config.py::DATABASE_PATH` 指向的即是它，几小时前刚被写过，非孤立测试库）：

- 用 `sqlite3.connect("file:tgmatrix.db?mode=ro", uri=True)` 只读模式打开（`PRAGMA`/`SELECT` 查询，物理上不可能写入或改动任何数据），查得 `orders`/`coupons` 两张表的**真实建表 SQL**（来自 `sqlite_master`，即建表时实际执行的语句）与 `database.py` 代码**逐字节完全一致**——确认不存在任何未纳入版本控制的历史 `ALTER TABLE` 弥合这个差异，`product_id`/`duration_type`/`duration_days`/`coupon_id`/`code`/`status`/`expires_at`/`max_uses` 这些列在真实表里**确定不存在**。
- 顺手查了行数：`orders` 表 **0 行**，`coupons` 表 **0 行**，`licenses`/`activations` 也都是 **0 行**——但作为对照的 `users` 表有 **10 行真实账号数据**。也就是说：**这套系统里已经有人真的注册在用，但从未有一笔订单/优惠券/授权成功写入过**，与"下单请求每次都在 INSERT/SELECT 处被打断、被 try/except 吞掉变成 500"的行为特征完全吻合。

**本轮仍未对 `license_server.py`/`database.py` 做任何改动**——"怎么修"（改 INSERT 对齐建表，还是趁机把 `duration_days`/`coupon_id` 这类业务上明显需要的字段正式补进表结构、并同步检查 `handle_payment_callback`/管理后台报表等所有下游读这两张表的代码）涉及产品决策，不是我能单方面拍板的代码风格选择。详细证据（精确行号、列名对照表、真实建表 SQL、行数）已写入 `platform/licensing/LICENSE_CONTRACT.md` §6①。

> 建议：**这条建议优先于本融合项目的其它任何事项**——`/api/payment/create` 目前看来自上线以来可能从未成功创建过一笔订单，这不再是"可能命中生产的推测"，是"本机实证的事实"。是否要立即修复、修复到什么程度，需要业务方拍板；但"这个问题是真实存在的"这一点，现在已经确定。
