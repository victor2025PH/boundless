## 〇、一句话结论

**设备功能本身是通的（选了真的生效、自动选择逻辑也在），坏的是"呈现层"：把操作系统的原始设备表原样倒给了用户。** 91 个输入里真正不同的设备只有约 8 个——其余全是 Windows 四种音频接口（MME / DirectSound / WASAPI / WDM-KS）对同一设备的重复上报，加上 Voicemeeter 一家就贡献 20+ 个通道。这是一个**不动音频引擎、纯显示层映射就能根治**的问题。

| 统计（以截图机器为例） | 数量 |
|---|---|
| 下拉里的"输入设备" | 91 个 |
| 去掉接口重复后的真实设备 | ≈ 8 个 |
| 其中普通用户该看到的 | ≈ 4 个（手机麦 / USB 麦 / 摄像头麦 / 主板声卡） |
| 选了会出直播事故的"毒选项"（回声/自激/无声） | ≥ 3 类，目前**混在列表里毫无防呆** |
| 对用户唯一正确的"输出"答案 | 1 个（CABLE Input），但埋在 60 个里 |

## 一、为什么列表这么乱（工程根因，五条）

1. **host API 重复**：`sounddevice` 枚举时，同一物理设备在 MME、DirectSound、WASAPI、WDM-KS 各出现一次 → 天然 ×3~×5。
2. **MME 31 字符截断**：`Voicemeeter Out B3 (VB-Audio Vo`、`Realtek High Definition Au` 这类"话说一半"的名字全是 MME 截断造成，本身就有完整版（WASAPI 行）但用户不知道两行是同一个设备。
3. **虚拟设备军团**：Voicemeeter（In 1~5 / AUX / VAIO3 / Out B1~B3 …）、VB-Cable（CABLE Input/Output/16ch）、SplitCam、DroidCam 各自注册多个端点；还有 RVC 侧追加的 `LOOPBACK:` 内放采集项。
4. **系统伪设备**：`Microsoft 声音映射器`、`主声卡驱动程序`、`立体声混音`——工程上有意义，对主播是噪音甚至陷阱。
5. **前端原样渲染**：`ui.html` 里 `<option :value="a" x-text="a">`，值和显示是同一个原始字符串，没有翻译、分组、排序、隐藏任何一层。

## 二、现状功能可用性审计表（代码事实）

| 功能 | 接线事实 | 判定 |
|---|---|---|
| 麦克风下拉 | `audioInput → rvc.inputDevice → RVC sg_input_device`，RVC 按**名字字符串精确匹配**，选了真生效 | ✓ 可用 · 呈现差 |
| 虚拟声卡下拉 | 同上，输出到 CABLE Input 后 OBS 采集 | ✓ 可用 · 呈现差 |
| 一键开播自动选设备 | `_pick_audio_devices()` 优先级链（DroidCam→蓝牙→麦克风→兜底），结果回传前端自动套用 | ✓ 可用 · 是本方案的地基 |
| 「自动配手机麦克风」按钮 | 前端按关键词过滤 DroidCam/iVCam 等并配好 CABLE，无手机时给正确指引 | ✓ 可用 |
| 推荐文案（"推荐 DroidCam / 推荐 VB-Cable"） | **静态写死**：没插手机也照样"推荐 DroidCam"，与真实检测无关 | ⚠ 名不符实 |
| 设备体检（录 3 秒测噪声底） | `sd.rec()` **没传 device 参数 → 测的是"系统默认麦"**，不是你在下拉里选中的那只。选了 DroidCam、体检的却可能是 Realtek | ✗ 真 bug |
| 危险设备防呆 | `device_enum.py` 有排除表（cable/voicemeeter/立体声混音…）但**只用于自动选**，下拉列表不过滤：用户可把 `CABLE Output` 当麦克风选中 → 自激回环 | ✗ 缺失 |
| 选完设备的验证闭环 | 不存在。`audioLive()` 电平是监听/馈线的间接信号，无法回答"我选的这只麦到底行不行" | ✗ 缺失 |
| 前置检查（8 项环境） | 真实检测，可用；但条目是工程话（"Python (facefusion)"、"RVC 工程"），且**无逐项修复入口**——后端明明已有 VB-Cable 安装向导端点（`/api/audio/setup_wizard`），没接到这里 | ⚠ 可用 · 引导断层 |
| 摄像头选择 | 有中文名 + 分辨率 + 热切换 + 自动优先手机 | ✓ 健康，**是音频侧改版的现成样板** |
| RVC 变声手调 | 参数真实生效；但「索引影响 / 保护强度」无解释，且需再点「应用变声配置」才生效（两步陷阱，无提示） | ⚠ 可用 · 术语黑话 |

**分类逻辑散落 4 处的暗病**（工程债）：`device_enum.py` 的 patterns、`hub.js rvcRefreshDevices` 的自动选、`rvcSetupPhoneAudio` 的 PHONE_MIC 表、`avatar_hub._pick_audio_devices` 的优先级链——四套关键词各自为政，已经开始漂移（有的认 `voicemeeter` 是排除项、有的认它是手机麦）。本方案顺手收拢成一处。

## 三、三视角分析

### 3.1 开发工程师视角

**痛点：**
- RVC 按名字精确匹配设备 → **不能改值，只能加显示层**：任何"翻译/去重/分组"必须保留原始字符串做 value，否则变声直接断；
- MME 截断名与 WASAPI 全名是同一设备，需要前缀配对去重；值优先保留 MME 行（现有代码已实证 MME 采样率最稳），标签用 WASAPI 全名补全；
- 四套设备关键词表并存，没有单一真相；
- 体检测错设备（默认麦 ≠ 选中麦）是承诺违背。

**主张：** hub 后端把 `/rvc/devices` 升级为结构化（原始值 + 显示名 + 分组 + 徽章 + 默认隐藏标记），一处分类、全站共用（开播页/手机页/体检/自动选）；体检补 device 参数。

### 3.2 市场经理视角

**痛点：**
- "已检测 91 个输入"是**吓人的负卖点**：新用户第一反应是"这软件是不是坏了/太专业我搞不定"；
- 装机交付时"教客户选设备"是远程支持工单的最大单一来源，且每台机器的设备名都不一样，支持话术无法标准化；
- 客户现场演示时选错输出 = 无声事故，一次翻车抵十次宣传；
- 系统里已经做好的"自动选设备、自动配手机麦"完全没有被包装成卖点，用户无感。

**主张：** 对标 C 端直播工具的做法——**不给设备表，给场景**（"用手机当麦 / 用电脑麦 / 用耳机"）；把自动配置包装成「设备零配置，插上就能播」的宣传口径；设备区文案全部人话化，让演示页面截图本身就是营销素材。

### 3.3 用户视角

**痛点：**
- 我的心智模型只有三个问题：**① 我对着什么说话（麦克风）② 观众听到哪一路（直播声卡）③ 我自己用什么监听（耳机/音箱）**——界面却甩给我两个 91/60 项的英文列表；
- 不认识 MME / WASAPI / B1 / B2 / VAIO3，不知道"前置声卡/耳机/音箱"对应哪一条；
- 不知道选错的后果是什么，不敢乱动——试错成本是一场直播事故；
- 选完没有任何正反馈：不知道选对没有，要等开播翻车才知道。

**主张：** 每类只留一个人话名字 + 一句"这是什么"；选完当场能试（说句话看电平、点一下听提示音）；危险选项默认看不见；出事前有拦截（"选这个观众会听不到，确定？"）。

**关于"前置声卡/前面板耳机"的诚实说明**：Windows 设备名无法 100% 区分前/后面板——Realtek 有的机器把前面板注册为独立的"耳机/2nd output"端点（可识别，按上表命名），有的机器前后共用一个"扬声器"端点（无法从名字判断）。方案的兜底是 **「试听」按钮**：点一下向选中输出放 1 秒提示音，哪个响就是哪个——比任何猜测都可靠。

### 4.3 接口后缀（MME/DirectSound/WASAPI/WDM-KS）的去向

- 显示层彻底隐藏，收进悬停 tooltip（"技术细节：MME 接口，最稳"）；
- 去重配对规则：去掉接口后缀 → MME 截断名与更长名做前缀匹配 → 同设备合并成一条；
- **value 永远用 MME 那一行的原始字符串**（RVC 按名匹配 + MME 采样率最稳，两个既有事实决定的）；
- 「显示全部 N 个原始设备」开关（默认关）：打开后回到今天的原始列表，专家/排障不受任何损失。

## 五、交互改版方案（三层递进）

### 5.1 用户模式：一张「声音线路卡」替代所有下拉

用户模式（现在设备卡整个隐藏）新增一张只读卡片，把链路讲成一句话：

```
🎤 手机麦克风(DroidCam) → 🎭 变声 → ✅ 直播声卡 → OBS → 观众
                                        [🔊 试音]  [更换设备]
```

- 数据全部现成：自动选择结果（`one_click_start` 回传的 `audio`）+ 馈线探针 rms（有声时链路图流动/点亮）；
- 「更换设备」才进专家设备区，默认永远不用碰；
- 这张卡同时是**营销截图素材**（市场视角诉求）。

### 5.2 专家模式：下拉框换结构化渲染

- `/rvc/devices` 升级为返回 `[{value, label, group, badge, hidden, hostapi, raw}]`（保留旧字段兼容 phone 页等旧调用方）；
- 前端 `<optgroup>` 按 4.1 分组渲染，显示名 + 徽章（✅ 推荐 / ⚠️ 易出事 / 📌 当前使用）；
- 「已检测 91 个输入」改为"**检测到 8 个设备（已合并 91 条系统重复项）**"——同一个事实，从吓人变成可信；
- 推荐行改为**动态生成**：真检测到 DroidCam 才说推荐 DroidCam，没有手机时改说"未发现手机麦克风 → 点这里配置"（消灭静态谎言）；
- 选中默认隐藏类设备（用户开了"显示全部"后）时弹一次确认：「"电脑内放"会让观众听到回声，确定用它当麦克风？」；输出选成音箱/耳机时提示"观众将听不到变声，仅适合自己试听"。

### 5.3 试音闭环（P1，本方案的体验主菜）

| 动作 | 实现 | 用户看到的 |

| 体检整合 | 上面三项纳入现有「设备体检」评分与开播就绪度 | 开播前一次点检全绿才安心 |

### 9.2 实施中超出原方案的再优化（边做边想的部分）

1. **「显示全部」语义升级**：原方案是"展开隐藏组"；实施改为**切回系统原始全列表**（91 条原样）——高级用户排障/调音台路由 100% 零能力损失，简化视图则更彻底；每条目带 `variants`（合并前所有原始名），任意历史配置串都能反查元数据、危险确认在两种视图下都生效。
2. **「观众能听到✓」不等开播**：原方案靠直播中的 OBS 馈线探针；实施改为 output_test 时**并行录 CABLE Output 自证回环**（独立 InputStream，不与 sd 便捷流互顶），开播前 1 秒就能拿到"观众侧收到 -8.2dB"的铁证。
3. **变体归一**：历史档案里存的 WASAPI 变体串自动映射回合并后的规范值（MME），修掉"下拉显示空白"的隐性坑。
4. **三把锁互斥**：试音/试听/设备体检共用音频便捷流，`_AUDIO_TEST_LOCK`+`_CHECKUP_MIC_LOCK` 双锁防互顶（sd.play 会顶掉 sd.rec 的全局流）。
5. **顺手修老坑**：F5/书签直达 `#stream` 时 `$watch('tab')` 不触发，设备/摄像头列表要切走再切回才加载——init 补齐装载。
6. **RVC 术语人话化**（原属 P1 文案项，顺手做完）：索引影响→**音色贴合度**（更自然↔更像目标音色），保护强度→**咬字保护**（关↔咬字最清楚），变调加「-24 变男声 / +24 变女声」助记，原名进 tooltip。
7. **门禁固化**：新增 `test_device_humanize`（分类器真跑+后端接线+前端接线约 40 条断言）进 phase12 门禁，防回归。

### 10.2 实施中超出原方案的再优化（边做边想）

1. **持久化位置从「角色档案」改为「后端全局偏好文件」**：设备是"这台机器插了什么"，不是角色属性——跟角色走会导致切角色时设备互跳；改成 `audio_prefs.json` 后，**启动器/手机端/服务端自愈等所有入口**共享同一偏好（不止浏览器），也躲开 localStorage 多浏览器分裂。
2. **粘性偏好语义**：设备缺席时只"临时顶替"，绝不改写偏好本身——这是"插回来自动恢复"能成立的根基（业界常见做法是缺席即覆盖，用户插回设备还得手动选一遍）。
3. **「跟随系统默认」也是一种明确选择**：偏好文件区分「没保存过」（键不存在）与「明确选了默认」（空串），后者不会被自动推荐顶掉。
4. **手选护窗**：手动改选后 8 秒内，任何"在途刷新带回的旧偏好"不得回滚新选择（防 devicechange 刷新与用户操作赛跑的竞态）。
5. **零轮询为主、轮询兜底**：拔插感知优先用浏览器 `devicechange` 事件（1 秒级响应、零开销），45 秒慢轮询只做保险，不给 RVC/枚举加压力。
6. **试音黄灯按「设备指纹」失效**：试音结论绑定"当时测的那只麦"，换麦/超 7 天自动降回"没试过"——绿灯永远可信。
7. **词典合并规则**：JSON 词条排在内置词条前（先匹配、可覆盖排序），同词去重；`_readme` 直接写在 JSON 里，运营不用翻代码文档。

### 12.2 实施中超出原方案的再优化（边做边想）

2. **热切顺序的防御性设计**：不是朴素的 stop→config→start，而是 **config 先行**——RVC 的 `/config` 自带设备/模型校验（400 即拒），把"整单不动"做成默认失败姿态：校验不过时正在跑的转换毫发无损，用户最多看到一条报错 toast，不会"变声没了"。
3. **stop→start 之间强制 0.8s**：RVC 拾音循环每 0.25s 才检查一次停止标志，立刻 start 会把标志翻回 True → 旧流永不退出 → 新旧两条流叠在同一设备出「机器人音」。0.8s 等待 + 演练断言「再 start 必 400」把双流防线钉进门禁。
4. **切回 CTA 不自动执行**：首选设备插回时只亮蓝条给按钮，不自动切——自动切意味着直播中一次未经同意的断声（约 1.5s）。断声的决定权必须在主播手里。
5. **热切不写偏好**：顶替是临时行为。写了偏好，"插回自动恢复"（P2-1 的核心语义）就被自己人破坏了。
6. **审计只记变更**：每次保存都记会让 10 条额度被"同值重存"刷穿，真正的变更反而被挤出去——记账的价值在信噪比。

### 15.2 实施中超出原方案的再优化（边做边想）

1. **自动切走服务端而非前端**：原本最省事的做法是前端 devLost 时自动点 CTA——但「无人值守」场景浏览器根本没开，前端自动切是伪无人值守。改为 hub 守护循环（10s tick），页面关了照样切；且开销自限：开关关/转换没跑时零枚举成本。
2. **决策/执行分离**：护栏逻辑收进纯函数 `_dev_autosw_decide`（与既有 `_sh_plan` 同一设计哲学），确认窗/冷却/上限/撤防 12 个场景在门禁里离线断言，不用真拔设备。
3. **确认窗 25s > 枚举缓存 20s**：开火前保证至少隔着一次「新鲜」的子进程重枚举，拔插风暴的一过性空窗（实测 1.5s 级）不会触发误切——把「不误切」做成时序上的必然而不是运气。
4. **转换旗标空窗防误清**：热切链自身 stop→start 有 1~2s 的旗标 False 空窗；「连续 2 个 tick 不在跑才算真收场」防止每次热切把每场限次弹药重置成无限。
5. **auto 与人工同一本账**：自动切不另起一套统计——expose/click/ok/fail 带 src=auto 进同一漏斗、同一账本、同一成绩单。ops 来源 chips 天然能对比「人点 vs 自动」的成功率，为「要不要默认开」提供数据。
6. **周报是补发式而非准点**：周一 9 点 hub 不在线是常态（主播下播关机）；每 30min 看表 + 当周去重标记，当天任意时刻起来都会补发，且跨重启不重发。

### 16.2 实弹演练揪出的真问题：RVC 进程猝死（比原计划重要）

P7 本身是「体验层」，但实弹演练把 P6 埋着的定时炸弹炸出来了——两轮演练 RVC 进程两次原地消失（连接被重置、端口拒连，靠 P4 音频自愈 108s 后拉回）：

1. **死法一（23:42）起停沿重初始化**：热切链 stop→start 之间 `flag_vc` 短暂为 False，20s 级健康轮询 `/inputDevices` 恰好打进这个窗口 → `sd._terminate()` 落在残流/新建流上 → PortAudio C 层直接带走进程。P6 时修的「转换中不重初始化」只看旗标，护不住起停沿。
2. **死法二（23:57）第二 PortAudio 实例**：只冻结 sounddevice 还不够——枚举链里 `pyaudiowpatch.PyAudio()` 是**第二个** PortAudio 实例，转换中创建/销毁它同样在活驱动上动土（DroidCam 虚拟声卡驱动尤其脆）→ /config 一到就 ConnectionReset。
3. **根治（三重防线，全部进门禁回归锁）**：①转换中 `get_devices` 一律返回**内存快照**（最近一次完整枚举），零原生调用——快照在每次空闲枚举时自然刷新，转换开始前必然新鲜；②起停沿 10s 宽限期内被动枚举不重初始化（`_vc_transition_ts`），`set_devices` 用 `force=True` 主动解冻（热切停转换后必须看到新插设备，调用方已保证旧流退净）；③枚举全程持 `_dev_enum_lock`（两条 FastAPI 线程并发 terminate/initialize 是 C 层未定义行为）。
4. **教益**：无人值守功能的可靠性必须用「无人值守方式」验证——门禁/E2E 全绿也测不出这种时序炸弹，实弹演练（真转换+真守护+真热切链）才是它的主场。这也正是把演练脚本留成可重复资产的价值。

### 16.3 实施中超出原方案的再优化（边做边想）

1. **顺风车而不是新增量端点**：原方案说「timeline 增量拉取」——但开播页已有 4s 轮询 `/realtime/status`，挂一个 10min 窗口的轻字段成本为零；新拉取通道反而多一条要维护的轮询线。
2. **基线防重放**：字段带 10min 窗口意味着刷新页面必然又收到同一事件——首轮只记基线不弹 toast，「新鲜度」用「本页打开期间新出现」定义而不是墙钟窗口。
3. **裁决在后端而非前端**：门槛逻辑（剔 auto、算率、样本量）放前端会散落在 JS 里没法离线断言；后端纯函数 + 门禁 6 场景真跑，前端只认 `suggest` 布尔。
4. **建议聚合改走 60 天分桶而非终身 total**：终身账把一年前旧环境的战绩算进「近期适合开」的依据里；推荐看的本来就是当下环境，超窗旧账不作数。
5. **趁热打铁复查**：建议条不只在进页时算——手动热切成功回调里延迟 0.8s 再查一次，第 3 次成功的瞬间提示就出现，比下次进页才看到的说服力强。

### 17.2 实施中超出原方案的再优化（边做边想）

1. **advice 独立键而非 src=advice**：原方案说「三事件进同一本账（src=advice）」——实施时发现照做会把建议条曝光算进主漏斗 expose 分母，直接污染「拔插恢复达成率」和 P7-2 裁决的人工战绩（自动切的开启依据被自己的推广数据灌水）。改为 advice_* 独立键：同一本账（文件/端点/周报），统计上完全隔离。
2. **自救卡退场按真相不按乐观假设**：「重启变声」点击后不立即清卡——前端 rvcStartConversion 拿不到真启动结果（历史接口语义），改用 `rvc_conv` 顺风车让卡在转换真恢复后 4s 内自动退场；误清比慢 4 秒更伤。
3. **刷新页面不豁免进行中的事故**：P7-1 的基线防重放会吞掉页面打开前的旧事件（防重复 toast），但失败是持续状态不是瞬时通知——基线轮特判「最近事件=失败且转换没在跑」→ 静默挂卡（不弹旧 toast）。toast 与卡的语义就此分开：toast=新事件通知，卡=未解除的危机。
4. **rescue 进主漏斗、advice 不进**：自救卡的再试是真实的恢复动作（expose/click/ok/fail, src=rescue，ops 来源 chips 直接可比自动/人工/自救的成功率）；建议条是元层面的推广，只记效果不进漏斗。两者边界即「是否真动了设备线路」。
5. **任何一次成功切换都解除自救卡**：不只 rescue 按钮——手动 CTA、切回首选、自动切成功（ok 事件）都清卡，危机解除的判定与解除路径解耦。

### 20.2 实施中超出原方案的再优化（边做边想）

1. **不只加计划任务，还补 /ops 卡**：原案「零新代码只加计划任务」——实施中发现 selfcheck 有 history.jsonl + /ops 卡 + 顶栏 warn，acceptance 也有 `/api/ops/acceptance`；日巡若只有 bat+任务，失败只能翻 `fe_patrol_last.log`，与 P12「账本从拉式变推式」哲学不一致。补 endpoint+card 成本 <0.5h，收益是「计划任务跑挂 → alerts 叫醒 + /ops 一眼看趋势」双轨。
2. **Hub 离线 SKIP 是刻意设计**：Hub 停机时日巡若 FAIL+告警，会与「Hub 挂了」重复报警且无法自证前端。SKIP 写入 history（`overall=SKIP`），/ops 显示灰色 chip，**不误报、不 silence 真实前端回归**（Hub 在线后的第一次 FAIL 仍会 fire）。
3. **smoke+interact 打包而非只跑 interact**：interact 已含 F/G 零 pageerror 断言，但 smoke 还覆盖 `/phone` `/ops` 两页——日巡多 14s 买的是「三页零 pageerror」完整覆盖，acceptance 快速集同结构。
4. **register 脚本 ASCII 化**：首版中文 Write-Host 含 `[OK]` 被 PowerShell 当类型字面量解析失败——改为与 selfcheck 注册脚本同风格的 ASCII 拼接，已在生产 register+status 验证。
