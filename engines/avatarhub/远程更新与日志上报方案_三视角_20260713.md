# 远程更新 + 使用日志/错误上报 · 三视角分析与优化方案（2026-07-13）

> 背景：140/198 两台标机装机 QA 已把「安装→组件下载→运行」链路磨顺（1.0.6 已官网发布）。
> 本方案回答两个问题：**① 程序如何做到"远程更新，只下必要的差量"；② 用户使用日志/错误如何
> 回传服务器，支撑 bug 处理与 AI 驱动的持续升级**。原则：最大化复用已有基建，不造第二套轮子。

---

## 〇、现状盘点（先搞清楚已经有什么，避免重复建设）

| 能力 | 现状 | 结论 |
|---|---|---|

| **程序本体更新** | **缺失**：`*.py`/`static/`/`tools/` 只随 46MB 安装包走；改一行代码就要用户重跑安装器（或我们手工 scp） | **本方案核心补齐项** |

| 崩溃/错误上报 | 缺失（客户机出错靠微信发日志截图） | **本方案核心补齐项** |

**深度思考后的关键判断**：用户要的"远程更新"其实缺的不是下载器（已经很强），而是三件事——
**(a) 把"程序本体"也变成一个可增量的组件**；**(b) 把"手动点更新"变成"后台静默、直播避让、
一键生效"的体验**；**(c) 给更新链路加上签名与回滚的安全兜底**。

## 一、三视角分析

### 1. 开发工程师视角

**痛点**
- 热修复无通道：昨夜 tools 缺失这类 bug，修好后只能重发整个安装包，存量用户永远停在坏版本。
- 客户机是黑盒：出错全靠用户口述+截图，无版本分布、无错误聚类，同一个 bug 反复处理 N 次。
- 发布无质量反馈：装机成功率/组件下载失败率没有数字，只能等用户骂。

**主张**
- **一套更新管道覆盖四类产物**：exe（极少变）/ 程序代码包（高频）/ conda 环境（低频）/ 模型（低频），全部走 manifest+sha+多源+回滚这一套，运维心智只有一份。
- 错误上报直接进现有告警/工单流（`alerts.py` 三出口已有），错误簇每日喂给内部 AI agent 做归因与修复草案——形成「用户报错 → AI 定位 → 增量热修上线」的 24-48h 闭环（这就是"让 AI 升级服务"的落地形态）。
- 红线：更新必须**原子可回滚**（应用到一半断电，下次启动仍能跑旧版）；manifest 必须**验签**（密钥基建已有：ed25519 已用于 provenance/license）。

### 2. 市场经理视角

**痛点**
- 功能迭代触达不了存量用户（新功能=新装机才有），迭代的营销价值被浪费。
- 版本碎片化 → 客服成本高、口碑事故无预警（"装完用不了"要发酵到群里才知道）。
- 定价/包装拍脑袋：不知道用户真实用哪些功能、什么显卡占比多少。

**主张**
- **更新弹窗=自有运营位**：每次增量更新附 changelog（"新增 XX 引擎/画质提升 2 倍"），是零成本的存量触达渠道；beta 通道运营成"尝鲜群"，天然社区。
- **遥测=决策依据**：DAU/留存、功能漏斗（换脸/克隆/直播各多少人真的在用→档位与定价校准）、GPU 档位分布（决定优化优先级）、每版本错误率（发布质量 KPI，红了自动停止放量）。
- **谨慎点（品牌红线）**：本产品卖点是"本地部署、数据不出机"。遥测策略必须克制且透明——崩溃报告默认开（可关）、使用统计首启明示询问、**内容数据（人脸/声音/视频）永不上传**并把这句话写进产品页；对 B 端客户，"可完全关闭/可指向自建服务器"反而是卖点。

### 3. 用户视角

**痛点**
- "更新=重下十几个 G"不可接受；很多用户网络差、磁盘紧。
- 直播/录制中被更新打断=生产事故，比不更新严重一万倍。
- 出 bug 求助流程长：截图、找日志、发群、等回复。
- 不清楚软件到底上传了我什么——换脸用户对隐私高度敏感。

**主张**
- 更新前明确告知体积（"本次更新 3.2MB，重启后生效"），后台下载不占前台；**开播/录制/同传会话中绝不重启**，顺延到空闲。
- 更新失败一键回滚/一键修复，永远不至于"更新把软件搞坏了"。
- 「一键反馈」：出问题点一下，日志（脱敏后、发送前可预览）自动带上，下个版本真的修了并在 changelog 点名——信任闭环。
- 隐私开关一目了然：设置页两行人话+两个开关，本地可查"我发送过什么"（`runtime/telemetry/` 已可审计）。

## 二、优化后的方案（分阶段落地）

### A. 远程增量更新

**P0（1-2 天）——程序本体组件化 + 静默更新体验**
1. `build_packs.py` 新增 **`app` 组件**：`*.py + static\ + tools\ + requirements\ + assets\`
   打成 `app-<ver>.tar.zst`（实测预计 3-6MB），入 manifest `components.app`，版本号=发布号。
   排除清单与 AvatarHub.iss 的 `[Files]` 对齐（单一真相：从 .iss 生成或共用一份排除表）。
2. `pack_installer.py` 新增 `app` 安装器：下载校验 → 解压到 `app_staged\<ver>\` →
   **应用时**：停子服务（复用 service_manager 停启）→ 现版目录换名留存（`app_prev`）→ staged 转正 → 重启服务。
   回滚=`--rollback` 翻回 `app_prev`。应用前检查 `/realtime/status`、interpreter `/health`：**直播/会话中→顺延**。
3. launcher 体验：启动即后台 `check_updates`（已有）→ **后台预下载**全部差量到 `_pack_cache` →
   托盘/控制台徽章"更新已就绪（3.2MB），重启生效"→ 用户确认或下次启动自动应用。
4. 发布侧：`publish_release.py` 把 app 包纳入上传清单；**修发布规约**：先传 `*.tmp` 再原子改名
   （今晨 140 装机撞上"发布中途重写包文件→客户端 sha 反复失败"的实锤坑）。

**P1（2-3 天）——安全与 exe**
5. **manifest ed25519 签名**：发布时用私钥签 manifest（字段 `sig`），客户端内置公钥验签后才应用
   （公钥随安装包分发；密钥管理复用 license 体系）。防 CDN/镜像被篡改的供应链攻击。
6. **exe 摆渡自更新**：exe 变更时下载 `AvatarHub.exe.new` → 退出时 spawn 一个 cmd 摆渡脚本
   （等待进程退出→替换→重启）。exe 极少变，此路径为兜底。
7. 组件粒度优化：swapcore 拆 `swapcore`(换脸核)+`swapcore_det`(buffalo 检测)两包，
   平均更新体积再降；大模型组按"变更频率"继续拆（已有 latentsync unet/rest 先例）。

**P2（按需）**
8. 文件级模型增量：把 shared `_store` 内容寻址机制推广到模型组（改一个 onnx 只下一个 onnx）。
9. 百分比灰度：manifest 变体 + `anon_id` 哈希取模放量；错误率超阈值自动停止放量（联动遥测）。
10. 企业离线更新包：差量组件打成单文件，U 盘导入（`pack_installer --import`）。

### B. 使用日志与错误上报（扩展 telemetry.py，不新造）

**P0（1-2 天）——崩溃/错误上报 MVP**
1. 事件模型扩展：`kind: install|update|rollback`（已有）+ **`crash | error | heartbeat | feedback`**。
2. 采集钩子（新增 ~150 行的 `telemetry_client.py`，各服务一行接入，风格同 `service_auth.secure()`）：
   - launcher：Qt `sys.excepthook`；各 python 服务：uvicorn 异常中间件 + 进程退出码非 0 时由 supervisor 补报。
   - 内容：**栈签名**（模块名+行号+异常类，剥离盘符/用户名）、日志尾 50 行（**脱敏管道**：剥用户名/IP/token/b64 长串）、环境指纹（版本/edition/GPU 型号+驱动/显存/OS build）、当时功能上下文（如"开播中/照片换脸中"）。
   - 防骚扰：栈签名去重 + 单签名 24h 限频 + 单日总量上限；离线排队本地，恢复网络补发（复用本地先落盘设计）。
3. 服务端 ingest（部署在下载站同 VPS）：单文件 FastAPI `/t/ingest`——token 校验、限流、单条 ≤256KB
   → 按天 jsonl + SQLite 聚簇表（签名→计数/版本分布/首末见时间）。
4. **首启同意页**：两行人话+两个开关（崩溃报告=默认勾、使用统计=默认不勾），设置页可随时改；
   企业授权档支持 `telemetry_url` 指向客户自建服务器或一键全关。

**P1（2-3 天）——使用统计 + 反馈 + 看板 + AI 闭环**
5. 使用统计：功能计数器（换脸次数/克隆次数/开播时长/引擎延迟分位——hub `metrics.py` 已在本地统计，只做日聚合上报）+ 每日心跳（版本/edition/GPU）→ DAU/留存/漏斗。
6. 「一键反馈」：控制台按钮 → 打包设备体检 JSON+近期日志（脱敏，**发送前给用户预览**）→ ingest 标记为 feedback，自带联系方式选填。
7. 运营看板 v1（静态页读 SQLite）：版本分布、错误 Top10 及趋势、装机成功率、DAU。
8. **AI 升级闭环**：每日定时任务拉"新增错误簇"→ 喂给内部 agent（带代码库检索）→ 产出归因报告+修复 PR 草案 → 人审合入 → 走 A 的增量通道热修发布。指标：错误簇从上报到热修 ≤48h。

**隐私红线（继承 telemetry.py 现有红线并加严，写进产品文档）**
- 永不采集：人脸/声音/视频等内容数据、角色资产、b64、完整路径、用户名、license key、IP 明文。
- 一切先落本地可审计（`runtime/telemetry/`），UI 提供"查看我发送了什么"。
- `AVATARHUB_TELEMETRY=0` 环境变量一票否决（已有）。

### C. 验收标准（做完必须逐条真机验，按 140/198 QA 流程）

1. 改一行 `avatar_hub.py` → 发布 → 客户机徽章提示"程序更新 3.x MB"→ 应用+重启 <60s，期间若开播则自动顺延。
2. 应用中途断电/杀进程 → 下次启动自动回滚旧版，功能完好。
3. 篡改镜像上的 manifest/包 → 客户端验签拒绝并告警，不应用。
4. 拔网线装机 → 崩溃回执本地排队，恢复网络后补发成功。
5. 人为制造引擎崩溃 → 服务器 24h 内出现聚簇记录（含版本分布），回执中检不出任何 PII。
6. `--rollback prev` 对 app 组件生效。

### D. 工作量与依赖

| 阶段 | 内容 | 估算 |
|---|---|---|
| P0 | app 组件 + 静默更新 + 崩溃上报 MVP + 同意页 + ingest 服务端 | 3-4 天（可并行两条线） |
| P1 | 验签 + exe 摆渡 + 使用统计/反馈/看板 + AI 闭环流水线 | 3-4 天 |
| P2 | 文件级增量 / 灰度放量 / 离线包 | 按需排期 |

依赖：VPS 现有（下载站同机加一个 ingest 进程 + SQLite）；签名密钥复用现有 ed25519 基建；
无新增第三方服务依赖（不引 Sentry 等重型件，保持"可自托管"卖点）。

## 三、P0 实施结果（2026-07-13 当日落地 + 140 真机验收）

**已交付并在 .140 真机跑通的能力：**

| 能力 | 落地 | 140 实测 |
|---|---|---|
| 程序本体组件化 | `build_packs.py` 新增 `app:core` 组件（`--build-app --app-version`，全量 build 也自动产出）：根 *.py/*.bat（同 .iss 排除表）+ static/tools/requirements/assets + 预热脸，打成 `app-<ver>.tar.zst` | 280 文件 → **1.84MB** 包（vs 47MB 整装包，**热修体积降 96%**） |
| 增量应用 | `pack_installer.py`：`app` 类走"下载校验→解暂存→**快照**→逐文件 `os.replace` 突发覆盖→清消失成员→记账" | check-updates 正确列出 `app:core`；应用后 `app_build.json` 版本翻新、覆盖 280 文件 |
| 直播避让 | `_live_busy()`：直播/同传/**中枢运行中**任一 → 转 `pending`，不当场覆盖运行中代码 | hub 运行时 `--update` → "已就绪待应用"；停栈后 `--apply-pending` 秒级应用；生产 STT 全程 200 |
| 一键回滚 | `--app-revert`：还原 `runtime\app_prev` 上一代快照 + 记账回退（回退后该组件重新可更新） | 1.0.8→回滚→1.0.7→前滚→1.0.8 三段闭环全过 |
| 启动自动应用 | launcher 启动早期（服务未起的安全窗口）调 `apply_pending_app()`；`notify_updates_async` 升级为**轻量更新静默预下载**（app + ≤64MB 小组件，总量≤128MB） | — |
| 崩溃/错误上报 | 新增 `telemetry_client.py`（栈签名去重、24h 限频、日 20 条闸、离线队列、脱敏管道）；hub/faceswap/launcher 各一行接入 | 触发含盘符/token/IP 的异常 → 服务端聚簇记录为 `boom at ~\secret=<redacted> ip x.x.x.x`（**PII 全脱敏**） |
| 服务端接收 | `tools/telemetry_ingest.py`（**纯标准库**，非 FastAPI）→ VPS systemd 服务 `avatarhub-ingest`（127.0.0.1:8787，MemoryMax 256M）+ nginx `/t/` 反代 + 令牌 + 限流 | `https://bd2026.cc/t/health` 200；错误簇入 SQLite；错误令牌→401 |
| 发布规约 | `_upload_packs_101.ps1` 改**先传 `.uploading` 临时名→服务端 `mv` 原子改名**（根治 140 装机撞上的"发布中途重写包→客户端 sha 反复失败"） | — |

**实施中相对原方案的 5 处再优化（深入思考后的改动）：**
1. **ingest 用纯标准库而非 FastAPI**：接收端跑在生产下载站上，引 venv+pip 是额外故障面与内存开销；客户端已做 24h 去重+日限频，负载极小，`http.server` 足够 → 生产站零新依赖、systemd 一个进程搞定。
2. **代码热覆盖从"仅直播避让"收紧到"中枢运行中即延后"**：自查发现运行中的 python 进程被覆盖 .py 后，其后续惰性 import 会撞"新旧混合代码"。代码变更本就需重启才生效，所以正确模型 = 服务全关时（启动器下次启动）再原子应用；只有 hub 未运行（全新装机/离线）才当场应用。避免了一类隐蔽的偶发崩溃。
3. **app_build.json 标记去掉 built_at、确定性打包**：否则同版本同代码每次打包 sha 都变，会误报"可更新"。改为只含 version+files+sort_keys → 同输入同 sha，与增量判定地基自洽。
4. **崩溃上报端点 `remember_endpoint` 落持久配置**：app 热更新不含 manifest，本地 manifest 停在安装期旧版会导致上报端点漂移；改为启动器每次检查更新时把生效端点落进 telemetry 配置，上报不依赖本地 manifest 新鲜度。
5. **更新检查判定 = 安装记录 ∪ 物理就位**（承接昨日 buffalo 修复）：让"新增成员文件"的新版组件在存量机上正确出现在更新清单，app:core 首次下发也据此把 Inno 铺盘的存量机纳入受管版本。

**未触碰红线的守护**：全部改动对老部署机"缺文件=完全无感"（try 导入）；遥测默认只开崩溃报告（栈签名级），使用统计仍 opt-in；`AVATARHUB_TELEMETRY=0` 一票否决；内容数据永不采集；本地 `runtime/telemetry/` 可审计。生产 .140 STT 全程未受影响。

## 五、P1 实施结果（2026-07-13 当日续接 + 140 真机验收）

按"安全 > 触达 > 洞察"排序全部落地，逐条 140 真机跑通。

| 能力 | 落地 | 140 实测 |
|---|---|---|
| **manifest ed25519 验签**（最高优先·堵供应链缺口） | 新增 `release_sign.py`：**公钥硬编码在代码里**（绝不从 manifest 读，否则连签带钥一起换就绕过）；`build_packs` 出清单即签、改清单必重签；`pack_installer.load_manifest` 读入即验签，不过则**抛错阻断**。三态策略：有签名必验、迁移期无签名放行、**一旦见过签名即拒无签名**（防"剥签名"降级攻击） | 正常签名→通过并放行；**篡改组件 sha→InvalidSignature 拒绝阻断**；**剥签名→防降级拒绝**。三态全绿 |

**P1 实施中的 4 处再优化（深入思考后偏离初版）**：
1. **验签公钥硬编码而非放 manifest**：这是签名方案成立的**前提**——若公钥随 manifest 下发，攻击者篡改组件后用自己的私钥重签、连公钥一起替换即可绕过。公钥必须钉在客户端代码里（随 app 组件/安装包分发、可轮换）。
2. **防降级设计（no-downgrade）**：只"有签名才验"会被"删掉 sig 字段"绕过。改为客户端一旦见过合法签名就落 `seen_signed` 标记，此后拒绝无签名 manifest——迁移期平滑（老机首次放行）、迁移后自动收紧。
3. **热修自愈做成"语法闸（应用前）+ probation（应用后）"双层**而非单一运行探针：语法错是坏热修最常见形态，应用前 `py_compile` 就能确定性拦下、连覆盖都不发生（比"覆盖了再回滚"更安全、不留半新代码）；运行期错误才交给 probation 兜底。隔离坏 sha 避免"下载→失败→再下载"的死循环。
4. **exe 摆渡用轮询父进程 PID 而非固定 sleep**：固定等待要么太短（exe 仍被占用替换失败）要么太长（用户干等）；轮询 `tasklist /FI "PID eq <pid>"` 直到父进程真的退出再替换，稳且快。

## 十三、P5 实施结果（2026-07-13 当日续接 + 140 真机验收）

**本轮核心是修一个我自己在 P3 埋下的安全缺陷**：P3 让 VPS 上的自动 halt「重签 manifest」——那需要**代码签名私钥（密钥 A）放到互联网下载站**，直接摧毁 P1 签名防篡改的根基（攻击者拿到 VPS 即可伪造代码更新）。P5 重构为**双密钥架构**并落地看板一键操作。

| 能力 | 落地 | 验收 |
|---|---|---|
| **双密钥架构（安全重构）** | 密钥 A 签 manifest=代码完整性，**私钥只在构建机**；新增密钥 B 签 `rollout_control.json`=运行时放量控制（halt/紧急百分比），**私钥可在 VPS**。客户端拉 manifest(A 验)+control(B 验)叠加。VPS 即使被攻陷，密钥 B 只能停更/改比例（fail-safe DoS），**无法伪造代码** | VPS 实测：`release_sign_ed25519_sk.pem`(A) **不在** VPS、`rollout_control_ed25519_sk.pem`(B) 在 |
| **控制通道 halt/resume/百分比覆盖** | ingest `_auto_halt` 改为写密钥 B 签名的控制文件（不再碰 manifest）；客户端 `_fetch_rollout_control` 多源拉取+验签+叠加到灰度准入 | 140：控制通道 halt 1.5.0→无可更新；resume→恢复可更新；伪造(改内容不重签)→验签拒绝忽略 |
| **看板一键放量操作** | `/t/control`（令牌鉴权，密钥 B 签）+ 看板"版本×错误率"行内挂 停放/恢复/设% 按钮（二次确认）；控制未启用则显示禁用态 | VPS：`/t/control` halt 写入→https 服务→密钥 B 验签通过、指纹 2472a94637020a5f |
| **自动刹车（安全版）** | 错误率超阈 → 写控制通道 halt（密钥 B），非重签 manifest；生产需配 `AH_INGEST_CONTROL` 启用 | VPS 已配 `AH_INGEST_CONTROL`，控制文件目录 ubuntu 可写 |

**P5 实施中的关键再思考（比原计划更重要的改动）**：
1. **发现并修复"私钥上服务器"的安全倒退**：原 P5 计划直接做"看板一键 halt 按钮"，但深想其实现路径时发现——要在 VPS 侧改放量就得在 VPS 签名，而当时的实现会把代码密钥 A 放上去。这等于给了"看板便利"却拆了"签名安全"的墙。**正确解是双密钥分权**：代码完整性（A）与放量控制（B）用不同密钥、不同存放位置、不同信任级别。这条比"加个按钮"重要得多，是本轮真正的价值。
2. **控制通道设计成"叠加覆盖层"而非"改 manifest"**：manifest 是发布时（A）定的不可变代码清单；控制通道是事后（B，可自动/一键）下发的运行时覆盖。两者职责、密钥、可变性彻底分离——manifest 永远只被构建机改，控制永远只影响"要不要/给多少比例装"，从不影响"装的是什么代码"。
3. **伪造控制 fail-safe**：客户端验不过控制签名就**忽略控制、按 manifest 正常放量**（而非拒绝更新）——因为控制通道是可用性层，伪造它最多让客户端"没收到 halt 指令"（回到正常放量），不会造成装坏码；而 manifest（代码层）验不过则硬拒。两层不同的失败姿态对应两层不同的风险。

**当前产物**：`release_sign.py`(密钥 B 控制通道签/验，指纹 2472a94637020a5f，私钥 B 已上 VPS、A 仍只在构建机)、`pack_installer.py`(多源控制通道拉取+叠加)、`tools/telemetry_ingest.py`(控制通道 auto-halt + `/t/control` + 看板按钮)、`tools/rollout_ctl.py`(manifest 初值不变)；app v1.5.0 签名+灰度入两份 manifest；VPS ingest 升级 + 密钥 B 就位 + `AH_INGEST_CONTROL` 启用。**live 站仍未推**（待拍板；上线清单同步补充"密钥 B 部署 + 控制文件路径"）。

**下一阶段（P6）**：①【需拍板】按《上线操作清单》真上线（含密钥 A 备份、密钥 B 已在 VPS）；②AI 闭环真机跑一轮（cursor-sdk+KEY）；③留存漏斗接付费转化；④ai_fix_agent cloud+auto_create_pr 夜间产修复 PR；⑤app 二拆/文件级模型增量/企业离线包/Mac。额外改进：控制通道加"过期时间"（halt 带 TTL 防永久遗留）、看板操作审计留痕（谁在何时 halt 了什么）、密钥轮换演练（A/B 都要能换）。

**P6 关键再思考**：

## 四、下一阶段（P1）实施顺序与改进点

按"安全 > 触达 > 洞察"排序，仍每步走 140/198 真机验收：

1. **【最高·安全】manifest 验签**：发布用 ed25519 私钥签 manifest（`sig` 字段），客户端内置公钥验签通过才应用 app/组件更新。当前只有 sha256（完整性）无真实性，是唯一的供应链缺口——**上线自动更新前必须补**（否则镜像/CDN 被黑可推恶意代码包）。密钥复用 provenance/license 的 ed25519 体系。
2. **【高】exe 摆渡自更新**：exe 变更时下载 `.new` → 退出时 spawn cmd 摆渡脚本（等进程退出→替换→重启）。exe 极少变，作兜底；配合 app 组件即可实现"从不需要用户手动重装"。
3. **【高】使用统计 + 每日心跳**：`telemetry_client` 加 `heartbeat/usage` 事件（复用 hub `metrics.py` 本地统计做日聚合）→ DAU/留存/功能漏斗/GPU 档位分布。用于定价校准与优化优先级（市场视角的核心诉求）。
4. **【中】一键反馈 + 运营看板**：控制台"反馈"按钮打包脱敏日志（发送前预览）→ ingest 标 feedback；`/t/stats` 已有聚簇 API，加个静态看板页读它（版本分布/错误 Top/装机成功率/DAU）。
5. **【中】AI 升级闭环**：每日定时拉"新增错误簇"→ 喂内部 agent（带代码检索）产出归因+修复草案 → 人审 → 走 app 增量通道热修。目标：上报→热修 ≤48h。
6. **【中】灰度放量**：manifest 变体 + `anon_id` 哈希取模按百分比放量；错误率超阈值自动停止放量（联动本阶段的错误聚簇）。
