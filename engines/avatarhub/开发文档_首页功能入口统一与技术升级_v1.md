## 0. 背景与目标

**现状一句话**：无界 BOUNDLESS · AvatarHub 是一套本地实时数字人平台（音色克隆 / 实时对话 / 直播换脸口型 /
实时同传 / AI 对话），由 FastAPI 中枢（`avatar_hub.py` :9000）编排 16+ 微服务、10 个 conda 环境、
241 条 HTTP/WS 路由、14 个前端页面。能力很强，但**缺少一个把全部能力聚合起来的"前门"**：
打开根路径 `/` 是 404，创作者用 `/ui`、终端用户用 `/phone`，两套各自为政，许多能力散落在 tab 深处或独立端口，
新用户/客户/运维都难以发现"这套系统到底能干什么、从哪进"。

**本次目标（用户原话拆解）**：
1. **首页功能入口**：把每个功能都做到首页有功能按钮、点击即进入该功能的入口。
2. **技术对标**：把现有技术栈与 2026 SOTA 对照，找出可升级项。
3. **三视角优化**：以开发工程师 / 市场经理 / 用户三个角度分析并优化。
4. 先深度思考、再给方案；先写全面开发文档、再按文档从 P0 开发。

**铁律（沿用项目既有工程文化）**：
- **零回归**：不破坏 `/ui`、`/phone` 等任何既有页面与路由；新增为主、改动最小面。
- **优雅降级**：依赖服务不在线时，入口卡片显示"离线/未部署"而非报错；新桥接上游不可达即关闭、让前端回退。
- **单一真相**：功能清单做成注册表（后端 `/api/features`），同时驱动首页、（后续）侧栏、命令面板、移动端。
- **复用设计系统**：首页一律用 `static/brand.css` 令牌与 `.bd-*` 组件，跨端白标随 `brand.js` 同步。
- **可移植**：不写死机器相关参数；CUDA 源等用环境变量可覆盖。

## 3. P0 详细设计

### P0-1 统一首页 + 功能注册表

**后端（`avatar_hub.py`，新增、零改既有）**
1. `FEATURE_REGISTRY`：Python 常量，功能条目列表（单一真相）。每条：
   ```
   {id, line, name, desc, icon, href, service?, edition, external?}
   ```
   - `line`：产品线分组（VoiceX/ChatX/LiveX/LingoX/运营/合规）。
   - `href`：入口 URL。站内深链用 `/ui#<tab>` 或漂亮路由（`/phone /converse /verify /delivery /ask /setup /dashboard /ops`）。
   - `service`：可选，关联 `SERVICES` key，用于首页就绪点（不在线→灰显"离线/未部署"，不报错）。
   - `edition`：`free`/`pro`，仅作展示标注（不做强拦截，避免误伤）。
2. `GET /api/features` → `{ok, features:[...], lines:[...]}`（免鉴权开放，供首页/后续侧栏/命令面板共用）。
3. `GET /` → 读取并返回 `static/home.html`（镜像 `/ui` 的 `HTMLResponse + no-store` 写法）。

**前端（`static/home.html`，新增）**
- `<head>` 复用与 `phone.html` 同款：`_clientlog.js`/`_authtoken.js`/`_errutil.js`/`brand.css`/`brand.js`。
- 结构：
  1. 顶部品牌区：渐变品牌名 + 可信徽章（本地运行 / AI 生成可验真）+ 版位徽章（取 `/api/license/status`）+ 右上"进入工作室 `/ui`"。
  2. 角色快速分流行（五大入口按钮）：我要创作→`/ui#voice`、我要直播→`/ui#stream`、我要对话→`/phone`、
     我要同传→`/ui#interp`、我要运维→`/ops`。
  3. 产品线分区（VoiceX/ChatX/LiveX/LingoX/运营/合规），每区一组 `.bd-card` 功能卡：图标+名称+一句卖点+
     就绪点（绿/灰）+ 版位标 + "进入"按钮（→`href`）。数据来源：`fetch('/api/features')` + `fetch('/health')`。
  4. 页脚：版本、关键链接、合规声明。
- 纯原生 JS（不引 Alpine，和 `phone/dashboard/ops` 一致）。就绪点：`/health` 的 `services[key]` 真值→绿，否则灰。
  无 `service` 字段的条目恒绿（站内页面，无外部依赖）。

**深链映射（首页按钮 → 既有入口，全部已验证可达）**

### P0-2 修复免提对话（`WS /api/converse/listen` 桥接）

**职责**：Hub 端把前端麦克风音频桥接到 Nemotron 流式 STT，命中真实语音即回 `{barge_in:true}`，
让"AI 说话时用户开口→立即打断"用语义判定（比纯能量更准），并在上游不可用时优雅降级回前端能量法。

**契约（前端既有，不改前端）**
- 入：`?session_id=&language=`；上行二进制 PCM16/16k/mono；上行文本 `{event:'reset'}`（清窗口）/`{event:'close'}`。
- 出：`{barge_in:true}`（每个 reset 窗口至多一次，避免抖动）。

**实现要点**
- 解析上游：`_svc_url("nemo_stt","http://127.0.0.1:7857")` → `ws://.../ws/transcribe?language=...`（不开 `auto_eou`，只取 partial 做打断信号）。
- 连接前先探 `/health`（短超时）；不可达 → 立刻 `ws.close()`，前端 `onclose` 置空 → 自动回退能量打断（零额外改动）。
- 双向泵：`client→nemo`（透传 PCM；`reset`→转发 `{event:'reset'}` 并清 barged 标记；`close`→收尾）与
  `nemo→client`（收到非空 `{partial}` 且本窗口未打断过 → 发 `{barge_in:true}` 并置 barged=True）并发运行。
- 出站鉴权：loopback 默认免令牌；若 `app_config.service_token()` 非空，则带 `X-AH-Svc` 头（`additional_headers`，回退 `extra_headers`）。
- 健壮性：任一侧断开/异常都安静收尾两侧；不抛错到前端。

**验收**
- 选 Nemotron 流式 STT + 免按键聆听：AI 说话时开口能被快速打断；环境噪声/咳嗽不误打断。
- Nemotron 未部署：聆听仍工作（回退能量打断），无报错。
- 普通（非流式）STT 模式行为不变。

### P0-3 工程可交付健康

1. **CUDA 源统一且可覆盖**：`install_facefusion.bat` / `install_rvc.bat` 的 torch 安装行——
   引入 `set "TORCH_CUDA_INDEX=https://download.pytorch.org/whl/cu128"`（默认对齐实际 cu128 部署），
   旧显卡可改 cu118；`pip install ... --index-url %TORCH_CUDA_INDEX%`。注释同步改为"CUDA 12.8（默认）"。
2. **faster-whisper（核实后修正）**：经查 `cosytts` env 刻意未含 `faster-whisper`/`ctranslate2`/`av`
   （CTranslate2 对 Blackwell/sm_120 滞后），`stt_server.py` 已优雅回退 `openai-whisper`(cu128 可用)，**非 bug**。
   处置：在 `requirements/cosytts.txt` 加**非安装性说明注释**（解释为何省略 + 受支持显卡如何 opt-in），
   **不盲钉版**（避免拉入不兼容 ctranslate2 破坏干净装机）。
3. **三方可复现说明**：在两脚本 `git clone` 处给出"如需完全可复现：`git checkout <commit>` 钉版"指引
   （不强钉具体 commit，避免选错破坏现网；落地由运维按当前可用 commit 决定）。

**验收**：干净机器按脚本安装得到 cu128 torch；STT 工作正常（faster 缺失即回退 openai-whisper，无报错）；脚本/freeze 注释自洽。

## 5. 技术对标矩阵（2026 SOTA，P2 择优灰度）

> 结论：现网已相当先进（Fish-Speech/Qwen3-TTS/VoxCPM、Nemotron 流式 ASR、MuseTalk/LatentSync、InsightFace）。
> 升级以"低风险高收益、可灰度并存"为准，不为新而新。

| 能力 | 现状 | 2026 可选升级 | 收益 | 优先级 |
|---|---|---|---|---|
| STT | ~~默认 Whisper~~ | ✅ **已默认灰度 Nemotron 3.5 流式**（健康门控，上轮完成） | 首音延迟↓、并发↑、增量字幕 | 高（已做） |
| TTS 首音 | 两段式首音**已默认开**(`CONV_TTFA_TWO_PHASE=1`，5s→~2s) | **单句内流式 TTS 已建好但默认关**(`CONV_TTS_STREAMING=0`，注释明确"开启前 GPU 机实测验收")→**灰度打开 + 内置 stream/whole 首音对照 + SLO 验收** | 对话/直播更跟手 | 高（已建·待 GPU 验收灰度） |
| 口型 | MuseTalk 默认 + LatentSync 可选 | 高清场景 LatentSync 设为可选默认档（同需 GPU 实时 RTF 余量验收） | 直播观感 | 中 |
| 换脸 | InsightFace inswapper 128 | hyperswap_256 高清副本（已留 `faceswap2` 路由位） | 近景清晰度 | 中 |
| 对话 LLM | Ollama Qwen2.5 | 跟随本地可商用新版权重灰度 | 质量/时延 | 低 |
| 可观测 | `/health`+monitor | ✅ **首页就绪度总览卡已上线**（复用 /health，本轮完成）；后续指标埋点 telemetry | 运维效率 | 低（卡片已做） |

## 6. 三视角分析与优化

- **开发工程师**：痛点=入口分散、`avatar_hub.py` 单文件 14k 行、三方未钉版、freeze 有缺口。
  优化=注册表单一真相（P0-1）、freeze/CUDA 修复（P0-3）、后续按域拆路由（P3）。
- **市场经理**：痛点=打开 `/` 是 404、能力强但"看不见"、新客户难感知价值。
  优化=首页产品橱窗 + 五大角色入口 + 可信徽章/版位，开箱即"一屏看懂能干什么"（P0-1/P1）。
- **用户**：痛点=不知从哪开始、免提对话打断断裂、依赖未起时报错吓人。
  优化=首页角色分流 + 首启向导串联（P1）、修复语义打断（P0-2）、统一优雅降级（贯穿）。

## 8. 决策与更优方案记录

- **首页形态**：选"独立 `home.html` 前门"而非 `/`→`/ui` 重定向——满足"所有能力一屏入口"，且不与 `/ui`(创作台)/`/phone`(终端) 抢定位。
- **注册表落点**：选后端 `/api/features` 单一真相（而非把清单写死在 home.html）——为 P1 反哺侧栏/命令面板/移动端留接口。
- **免提修复**：选"Hub 桥接 Nemotron"而非"在 Hub 内跑 STT 模型"——模型在 `nemoasr` 独立 env/进程，Hub 只做代理，符合现有架构；上游不可达即关闭让前端回退，零体验倒退。
- **CUDA**：选"env 可覆盖、默认 cu128"而非硬切——对齐实际部署（freeze 实为 `+cu128`）同时保旧显卡可回 cu118，符合项目可移植原则。
- **faster-whisper（核实后修正初判）**：原列为"freeze 缺口/bug"，核实发现 `cosytts` env 刻意未含
  `faster-whisper`/`ctranslate2`/`av`（CTranslate2 对 Blackwell/sm_120 滞后），且代码已优雅回退 `openai-whisper`(cu128 可用)，
  **非 bug**。故不盲钉版（防破坏干净装机），仅加非安装性说明注释——体现"先排查、零回归"。
- **STT 灰度切 Nemotron（健康门控 + 客户端自适应默认，而非硬切后端默认）**：后端 `defaults.stt` 保持 `whisper_stt` 不动，
  Nemotron 仅作「可选引擎 + 前端健康默认 + 请求级回退」。三重安全网（前端只在 nemo 在线时默认它 / `_conv_prepare` 掉线回退 / listen 桥接能量兜底）
  保证 Nemotron 任意时刻掉线都零体验倒退。选此法而非"改后端默认引擎"——后者一旦 nemo 没起，所有对话 STT 直接失败，违背"优雅降级"铁律。
- **首启引导落点（首页卡片 ↔ 既有 `/api/setup/status`，复用不造轮子）**：直接消费既有就绪检测端点（服务/角色/音色/黄金包四项），
  首页只做"未就绪才置顶引导"的轻 UI，不新增后端逻辑；与首页其它卡（注册表里的 `/setup` 卡）并存——引导卡负责"首次/未就绪时的强提示"，常态由 `/setup` 卡承载入口。
- **就绪度总览卡（复用同一份 /health，而非新增 /api 或重复请求）**：状态条与首页主渲染共用 Promise.all 里的 `/health`，
  零额外请求/零新增后端；只读 + 直达 `/ops`（不在公开前门放控制动作，避免鉴权/误触）；隐藏标签停轮询、就地刷新不重渲染——把"运维一屏掌握"做成低成本常驻而非新页面。
- **TTS 首音"灰度而非重写"（核验后改变打法）**：原计划列"TTS 首音延迟优化(开发)"，核验发现流式首音**早已建好、只是默认关且要求 GPU 验收**。
  故改为最小动作（翻 env/灰度）+ 用**内置 stream/whole 对照指标 + SLO** 验收——避免重复造轮子，把风险收敛到"验收"而非"开发"。
