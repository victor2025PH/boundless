# 授权在线刷新 `/api/refresh` 对接说明（客户端已就绪，待后台补对端）

> 目的：把「后台续费/升档 → 客户端自动生效」的**正向拉取**闭环补齐。客户端半场（`license.refresh_online`
> / Hub 周期刷新 / `/api/license/refresh` 手动端点 / 授权卡「🔄 刷新授权」按钮 / 失效即时 JIT 刷新 / 可观测
> `summary().refresh`）已实现并单测通过（`_license_test.py` 124/124）。**只差后台补一个 `/api/refresh` 端点。**
> 本文是给官网后台 `C:\web117`（或任何自建授权码后台）落地那一端的单一真相。

---

## 0. 一句话契约

客户端按**机器指纹**（无需重输码）POST 到 `{激活地址}/api/refresh`，后台查该指纹**当前应得授权**、
重签一份绑定该指纹的授权回传；客户端内置公钥复验签名+指纹后，**只升不降**地落盘。

---

## 1. 请求 / 响应

**`POST /api/refresh`**  ·  `Content-Type: application/json`

请求体：
```json
{ "fingerprint": "XXXX-XXXX-XXXX-XXXX", "lic_id": "可选：客户端当前 license 的序列号（定位提示）" }
```

响应：

| 场景 | HTTP | body |
|---|---|---|
| 查到当前应得授权（已签好） | 200 | `{"ok": true, "license": {"payload": {...}, "sig": "<hex>", "alg": "Ed25519"}, "code": "AVH-..."}` |
| 该指纹无任何有效授权记录 | 404 | `{"ok": false, "error": "该机器无有效授权记录（未激活或已被移除）。"}` |
| 异步签发中（见 §4 网站形态，私钥不在服务器） | 200 | `{"ok": false, "error": "正在签发，请稍后重试"}` |
| 缺参数 | 400 | `{"ok": false, "error": "缺少 fingerprint。"}` |

客户端对 `ok=false` 一律「保持现状、下次再拉」，不报错不清空——所以「签发中/查无记录/更旧」都安全。

---

## 2. `license` 文档格式（签名两端必须逐字节一致，否则客户端验签必失败）

```
doc = { "payload": <payload>, "sig": <hex(Ed25519 sign)>, "alg": "Ed25519" }
```
`payload` 字段：
```json
{ "v": 1, "machine": "<指纹或 *>", "edition": "trial|standard|pro",
  "licensee": "被授权方", "issued": <int 秒>, "expires": <int 秒，0=永久>,
  "lic_id": "<序列号，用于精确吊销>", "features": { "可选：覆盖能力键" } }
```
签名前的**规范化序列化**（与 `license.canonical_payload` 完全一致，务必照抄）：
```
canonical = JSON.stringify(payload, 键升序 sort_keys, 分隔符 (",", ":"), 不转义非 ASCII ensure_ascii=false)
sig = hex( Ed25519_sign(vendor_sk, utf8(canonical)) )
```
公钥已内置在客户端 `license.py._VENDOR_PUBKEY_PEM`（也是 `license_pubkey.pem`）——**后台用同一把厂商私钥签**。

---

## 3. 后台查「当前应得授权」的语义（务必与客户端单调闸门配套）

按 `fingerprint` 在授权/兑换码台账里选该机**最优**授权：
1. **跳过**已停用的码（`disabled`）与已吊销的 `lic_id`（在 CRL 名单里的）；
2. 同机多次激活（=续费/加购）**取到期最晚**（`expires=0` 永久视为最大）；到期相同再**取档位最高**（`pro>standard>trial`）；
3. 用该授权的 `edition/issued/expires/features/licensee/lic_id` **原样重签**回传——**只读，不新占座、不改台账**。

> 参考实现（已单测）：`license_server.py` 的 `find_entitlement_for_fp()` + `refresh()`；私有部署可直接用它当后台。

**客户端单调闸门**（`license._doc_is_upgrade`）：仅当「升档」或「同档到期更晚(>1h)」才应用；
更短/降档一律忽略（背景刷新绝不把好授权改差）。因此：

| 管理动作 | 后台怎么做 | 客户端结果 |
|---|---|---|
| 续费 | 把该激活 `expires` 改晚（或加一条新码激活） | 刷新即延后 |
| 升档 | 把该码 `edition` 调高 | 刷新即升档 |
| **降级 / 退款 / 换机** | ①该码置 `disabled` 停止再发 ②对旧 `lic_id`/`machine` **签 CRL 吊销**（`/api/revocations`） | 旧 key 被吊销即失效→回落试用，再刷新拉到新的低档（若有） |

即：**`refresh` 只负责「变好」，降级走「码停用 + CRL 吊销」两条负向通道**——职责清晰、互不耦合。

## 4. 两种部署形态

**A. 私有部署 / 本仓参考后台（有私钥，内联签）**：直接用 `license_server.py serve`，`/api/refresh` 已内置、
即时重签返回。管理动作用 `renew` / `addcode` / `whois` CLI（见 §6）。

**B. 官网 `web117`（私钥不上服务器，异步签发桥）**：`/api/refresh` **不能内联签**（服务器无私钥）。模式与
现有 `/api/activate` 路线 B 一致：
1. 查该指纹当前应得授权（同 §3）；按 `(fingerprint, edition, expires)` 做缓存键；
2. 若已有该「当前应得」对应的已签 `doc` 缓存 → 直接返回 200 + license；
3. 否则把一条重签请求写入签发队列 `console-signq`（`signPayload` = §2 的 payload），返回 `{"ok":false,"error":"正在签发，请稍后重试"}`；
4. 本机 `sign_worker.py` 轮询队列 → 本地私钥签 → 回填缓存；客户端下次刷新（JIT ~60s / 周期 ≤1h）即取到。

> 关键：缓存键必须含 `expires`/`edition`——续费改了到期就自然 miss、触发重签，不会一直返回旧 doc。

## 5. Next.js 参考实现（`web117` · `app/api/refresh/route.ts`，按你们 `lib/console-*` 适配）

```ts
// app/api/refresh/route.ts  —— 授权在线刷新（私钥不上服务器：命中缓存直接给，否则入签发队列）
import { NextRequest, NextResponse } from "next/server";
// 你们现有的数据层（官网后台_CONSOLE_OPS.md 记载）：
import { findEntitlementForFingerprint, getSignedDoc, enqueueSign } from "@/lib/console-issue";
import { isRevoked } from "@/lib/console-data";

export async function POST(req: NextRequest) {
  let body: any = {};
  try { body = await req.json(); } catch {}
  const fp = String(body?.fingerprint || "").trim();
  if (!fp) return NextResponse.json({ ok: false, error: "缺少 fingerprint。" }, { status: 400 });

  // §3：该指纹当前应得授权（跳过 disabled 码与已吊销 lic_id；取最晚到期/最高档）
  const ent = await findEntitlementForFingerprint(fp);              // → {edition,expires,issued,features,licensee,licId,code} | null
  if (!ent || isRevoked({ lic_id: ent.licId, machine: fp })) {
    return NextResponse.json({ ok: false, error: "该机器无有效授权记录（未激活或已被移除）。" }, { status: 404 });
  }

  // §2 的 payload（键会在签名端按 canonical 规范化；此处顺序无所谓）
  const payload = {
    v: 1, machine: fp, edition: ent.edition, licensee: ent.licensee || "",
    issued: ent.issued, expires: ent.expires, lic_id: ent.licId,
    ...(ent.features ? { features: ent.features } : {}),
  };

  // §4-B：缓存键含 expires+edition —— 续费/升档自然 miss 触发重签
  const cached = await getSignedDoc({ fp, edition: ent.edition, expires: ent.expires });
  if (cached) return NextResponse.json({ ok: true, license: cached, code: ent.code });

  await enqueueSign({ kind: "refresh", licId: ent.licId, signPayload: payload });  // sign_worker 会签并回填
  return NextResponse.json({ ok: false, error: "正在签发，请稍后重试" });            // 客户端保持现状，下次刷新即取到
}
```
> 私有部署（有私钥内联签）版：把「命中缓存/入队」两步换成一次本地 Ed25519 签名即可——逻辑同 `license_server.refresh()`。
> `sign_worker.py` 已支持从 `sign/pull` 取 `signPayload` 就地签发回填，无需改动（新增的 `kind:"refresh"` 与续费/发码同管道）。

## 6. 自查

**客户端 → 后台**（任意机器）：
```bash
curl -X POST {激活地址}/api/refresh -H "Content-Type: application/json" \
  -d "{\"fingerprint\":\"<本机指纹>\"}"
# 期望：200 {"ok":true,"license":{...}} 或 404/{"ok":false,...}
```
**后台/运维核对某机应得**（私钥机）：
```bash
python license_server.py whois --fingerprint <指纹>     # 当前应得档位/到期 + 全部激活/试用记录
python license_server.py renew --fingerprint <指纹> --days 365 --edition pro   # 续费+升档（改台账）
```
**看某机是否已拾取到改动**：客户端 `GET /api/license/status` → `state.refresh`（`ts/applied/edition/age_s`）
或 `state.refresh_configured`；`age_s` 过大＝该机在线刷新没在跑（查激活地址可达性 / license.key 是否存在）。

## 7. 安全红线（不变）

- **厂商私钥永不上互联网服务器**：`web117` 只做控制面 + 入队；签发在本机 `sign_worker.py`。服务器被攻破也伪造不了授权。
- 客户端拿到任何 doc 都用**内置公钥复验签名 + 校验机器指纹**，且**单调闸门**保证背景刷新绝不降能力——
  即便服务器被控，也只能对「请求的那台机」签、且不能把它改差。
- `/api/refresh` 对外无鉴权（威胁面同 `/api/activate`）：**务必限速**；按指纹反查「谁持有什么」的 `whois`
  只在私钥机做 CLI，**不开放为未鉴权 HTTP 端点**。
  - 参考后台 `license_server.py` 已内置固定窗口限速（`activate/refresh/trial/quickissue` 共用，默认
    每 IP 30 次/60s，`serve --rate-max-ip 0` 可在回环/内网关闭）；`web117` 侧请用你们现成的
    登录限流同款中间件套住 `/api/refresh`（对接说明照此打样）。
