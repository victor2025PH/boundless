# AvatarHub 打包与分发方案 v1

> 目标：让**非技术用户**下载、安装、在**本地电脑用自己的算力**运行 AvatarHub。
> 本文是工程实施基线，配套脚本：`build_packs.py` / `pack_installer.py` / `launcher_qt.py`（首启向导）/ `sign_artifacts.bat`。

---

## 1. 背景与定位

原 `installer\AvatarHub.iss` 打出的安装包只含「控制台 exe + 编排脚本 + 前端 + 依赖基线 + 文档」，**不含 conda 环境与模型**；目标机还要自己装 Miniconda、装 Ollama、`provision.bat --create`（pip 现装 5–6 个环境）、下载几十 GB 模型、手动匹配 CUDA。这是**集成商部署**流程，普通用户无法完成。

本方案把「用户机现装」改成「**CI/本机预构建 + 用户机只下载解压校验**」：环境用 `conda-pack` 冻结成可重定位归档，用户**无需 conda**、无需 pip、可离线安装。

### 可行性（已实测验证）
- `cosytts`（5.28GB，含 torch cu128）→ `conda-pack` 41.9s → 3GB；解压到全新路径 + `conda-unpack` 后，`torch 2.11.0+cu128`、`cuda_available=True` 正常。
- 关键原因：`app_config.py` 用**直接 python.exe 绝对路径**探测解释器（非 `conda activate`），解压出的环境登记到 `config.json["conda_python"]` 即可用。

---

## 2. 三视角结论摘要

- **开发工程师**：废弃用户机 pip 现装（脆弱、依赖网络/版本）；改为 CI 预构建可重定位环境包 + sha256 校验 + 按显卡架构出 `cu128`/`cu121` 变体 + 组件版本化增量更新。
- **市场经理**：一个安装包双击即用；装前验机分档管理期望；试用→付费转化（输码激活、pro 去水印）；EV 代码签名消除 SmartScreen 告警；低带宽市场出 U 盘离线版。
- **用户**：双击安装→自动认显卡→按档下载（进度/续传/校验）→桌面图标→点开即用；无需碰 conda/Ollama/命令行；出问题有「组件/体检」一键修复。

---

## 3. 总体架构

```
[构建端] build_packs.py
   └─ conda-pack 冻结各环境 → dist/packs/<env>-<archtag>.tar.gz
      （--include-models 时另打 model-<group>.tar.gz）
   └─ 生成 dist/manifest.json（版本 / 分档映射 / 各组件 file+sha256+size）
   └─ 上传 packs/* + manifest.json 到自有下载站 / Git（base_url）

[用户端] launcher（首启向导，pack_installer 内核）
   验机(nvidia-smi) → 选档(默认推荐) → 16线程断点续传下载 + sha256 校验
   → 解压到 runtime\envs\<env> → conda-unpack → 写 config.json[conda_python]
   → 进控制台（无需 conda）
   增量更新：比对 manifest sha256，只重下变化的组件
   授权：输码 / 导入 license.key 激活，pro 去水印
```

## 5. 分档策略

| 档位 | 目标硬件(显存) | 含环境 | 卖点 |
|---|---|---|---|
| **Lite·幻颜/幻声** | ≥6–8GB | facefusion, rvc | 换脸 + 克隆音，入门、包小 |
| **标准·幻影256** | ≥16GB(4090) | + fishspeech, cosytts, musethepeak | 实时数字人标清 |
| **旗舰·幻影HD** | ≥24GB(5090) | + latentsync | 高清 / 活体 25fps |

档位↔组件映射定义在 `build_packs.py` 的 `EDITIONS`；验机最低显存阈值在 `pack_installer.py` 的 `EDITION_MIN_VRAM_GB`。向导对不满足显存的档位自动置灰，并推荐最高可跑档。

### 模型组（`MODEL_GROUPS`，`--include-models` 时打包，已实测）
每组可配 `paths`（打包目录/文件）与 `exclude`（剔除子路径，避免重复/分档精确）。各组源体积：
facefusion 2.89GB · gfpgan 0.32GB · rvc_weights 3.47GB · cosyvoice 5.23GB ·
musetalk(基座，排除 V15) 源 4.4GB→打包 1.1GB · musetalk_hd(=musetalkV15) 3.17GB ·
fishspeech 1.37GB · latentsync 6.4GB · liveportrait 0.61GB。
> 关键：`MuseTalk/models` 已含 `musetalkV15`(HD)。基座组排除它、HD 单列，使**标准档不下载 HD 权重**、旗舰档才加；同时 `rvc_weights` 只取 `RVC/assets/weights` 一份，避免与同内容的长名目录重复打包。模型解压回项目根（保留相对路径），安装后 `is_installed` 按成员路径核验。

## 6. 操作手册

### 6.1 构建（发布前，本机或 CI，用 base 解释器）
```bat
python build_packs.py                 :: 体检：列出将打包的环境与体积
python build_packs.py --build --version 1.0.0 --base-url https://你的下载站/avatarhub/1.0.0
python build_packs.py --build --include-models ...   :: 连模型一起打（大）
python build_packs.py --build --arch-tag cu121 ...    :: 老显卡变体（另一套环境）
```
产物：`dist\packs\*.tar.gz` + `dist\manifest.json` → 上传到下载站 / Git。

### 6.2 制作安装包（薄核心，签名）
```bat
build_launcher.bat                    :: PyInstaller → dist\AvatarHub.exe（含 PySide6 + cryptography）
installer\build_installer.bat         :: Inno Setup → dist\AvatarHub-Setup-1.0.0.exe
set AVATARHUB_SIGN_SHA1=<证书指纹> & sign_artifacts.bat   :: Authenticode 签名 exe + 安装包
```
> 把 `manifest.json` 一并放进安装包目录（程序根），或给用户机设 `AVATARHUB_MANIFEST_URL` 指向下载站；首启向导据此工作。

### 6.3 用户端（命令行等价物，向导即此流程的 GUI）
```bat
python pack_installer.py --manifest <path|url> --gpu              :: 验机 + 推荐档位
python pack_installer.py --manifest <path|url> --edition standard --install
python pack_installer.py --manifest <path|url> --check-updates    :: 看可更新组件
python pack_installer.py --manifest <path|url> --update           :: 增量更新
```

### 6.4 授权（厂商 / 客户）
```bat
:: 客户取指纹（或在 launcher「授权」面板复制）
python license_admin.py fingerprint
:: 厂商签发（私钥在 secrets\，不入库）
python license_admin.py issue --machine <指纹> --edition pro --days 365 --licensee "客户名"
:: 客户激活：launcher「授权」面板粘贴授权码 / 导入 license.key（pro 自动去水印）
:: 强制按授权限制能力（默认只评估不拦截）：
set AVATARHUB_LICENSE_ENFORCE=1
```

## 7. 增量更新

- `manifest.json` 每个组件带 `sha256`；用户机 `runtime\installed.json` 记录已装组件的 sha/版本。
- `pack_installer.check_updates()`：已安装但 sha 与最新 manifest 不一致 → 可更新。
- 代码改动通常只动核心安装包（几百 MB）；环境/模型包未变则不重下。`launcher` 启动后台静默检查，有更新在日志区提示，点「组件」即增量更新。

---

## 8. 信任与安全

- **代码签名**：`sign_artifacts.bat` 支持 PFX 文件 / 证书库指纹 / EV 令牌主体三种来源 + RFC3161 时间戳。**EV 证书**给 SmartScreen 即时信誉，强烈建议对外发布前签名。
- **授权防伪**：Ed25519 离线验签，厂商持私钥，产品内置公钥；`license.key` 篡改 / 换公钥 / 绑错机器均被拒。
- **完整性**：所有组件下载后 sha256 校验，损坏即报错重试。
- **不入库**（`.gitignore` 已含）：`dist/`、`build/`、`runtime/`、`_pack_cache/`、`secrets/`、模型与大二进制、`license.key`、私钥。

---

## 9. CDN / 分发渠道

- **首选阿里云 OSS**（与现有 pip 镜像统一、国内回源快、分片续传 SDK 成熟，配合 16 线程下载器）；腾讯云 COS 亦可。
- 大头成本是**外网下行流量**：资产桶开 CDN + 缓存，避免每次回源计费。
- **U 盘 / 离线介质版**：同一批 packs + manifest 放本地目录，`manifest.base_url` 留空时 `pack_installer` 以 manifest 所在目录为源，离线安装。低带宽 / 企业客户分流。

8. **环境包去重**（本轮新增，数据先行 + PoC 实测；这是首装/占盘的最大杠杆）：
   - ✅ **`build_packs.py --survey-envs` 量化工具**：按文件尺寸预筛、只对同尺寸候选做 sha256，快而准。**实测 6 环境源体积 49.39GB，全局内容去重后唯一仅 22.66GB，可去重 26.73GB（54.1%），且其中 26.72GB 全来自跨环境组**（即冗余几乎全是“同文件多环境各存一份”）。Top 全是 torch/CUDA 大二进制：`cublasLt64_12.dll`(643M×6)、`cusparse64_12.dll`(362M×6)、`torch_cuda.dll`(774M×3)、`cufft64_11.dll`(263M×6)、`cudnn_*`/`cusolver*` 等，均在 `Lib/site-packages/torch/lib/`、全在 C:\ 同卷。

   - ✅ **PoC 实测全过**：合成三环境同名大文件→硬链共享同一 inode、内容字节不变、差异文件不被链、小文件忽略、省盘统计正确(links=2/4MB)、二次运行幂等(0)。
   - ✅ **B-2 共享基座包（省下载，非仅省盘；本轮完成，真实+合成双验证）**：
     - **真实 make-or-break 验证（cosytts）**：`conda_pack` 带 `filters=[("exclude","Lib/site-packages/torch/lib/*.dll")]` 打包 → 环境包从 ~3GB **降到 451MB**（排除 37 个 DLL/4035MB）→ 解压 → 把 DLL 硬链回去 → `conda-unpack rc=0` → `import torch 2.11.0+cu128`、`cuda.is_available()=True`、识别 RTX 5090。**证明排除+硬链+unpack 后 torch/CUDA 正常加载**。

     - **合成端到端验证全过**：`--shared` 构建→环境包确不含 dll、落位/基座/档位展开正确→装基座→解压环境→硬链落位→**两环境树字节一致还原**（含 dll）→ `cublas.dll` 在 A/B/基座**三方共享同一 inode**（省盘）→ is_installed 全绿。
     - **效果**：环境部分由「6×重叠」变「唯一 blob 只下一次 + 各环境瘦身包」，**首装/下载量预计砍到一半量级**（唯一 ~12GB vs 原 ~28GB 压缩）。`make_release.py` 已同步：`_iter_components` 纳入 shared、新增 `--shared` 透传，基座包随 publish 树发布。

       - 注：该 manifest 为「仅环境+基座」构建；正式发布加 `--include-models`（模型组另计，见 B-1/A 节）。
   - ✅ **B-4 基座可复现分片（增量更新）+ 发布烟测门禁（本轮）**：
     - **① 基座按 sha 分片**：单一 5.93GB 基座包改为多个独立分片组件（id 形如 `torch-cuda.a`，各带自己的 `blobs/sha256`）。环境 `needs_shared` 据其各 blob 落入的分片自动取并集，`pack_installer`/`make_release` **零改动**（早已支持多个 shared 组件）。
     - **分桶规则的纠错（诚实记录）**：先用「sha 前 1 位 → 16 片」，实测发现**按 blob 数均衡 ≠ 按体积均衡**——多个大 CUDA 库恰好哈希到同一前缀，产出一个 **4.17GB(压缩 1.61GB) 巨片**，更新命中即重下 ~1.6GB，granularity 收益被吞。改为**混合稳定分桶**：大 blob（≥256MB）各自独占一片、小 blob 仍按 sha 前 1 位分桶。两者均「仅取决于 blob 自身 sha+size」→ **稳定**（未变 blob 永不迁移分片，增量成立），又**无巨片**。重建后 **30 片**，单片压缩上限从 1.61GB 降到 **~636MB**（其中一个 2.17GB 单库压到 401MB——单物理文件即增量下限，无法再切）。
     - **诚实的代价**：分片基座压缩总量 **6.37GB**，比单体 5.93GB **多 ~440MB（+7.4%）**——因 gzip 无法跨独立分片复用压缩字典。即「首装多付 7.4%、换取小升级重下量从 100% 降到 ~10% 量级」，对频繁迭代的本项目明显划算（后续可用 zstd 长程匹配把这 7.4% 找回）。
     - **可复现打包是增量的前提**：普通 `tar+gzip` 会嵌入 mtime → 同内容每次 sha 都变 → 增量失效。故新增 `_reproducible_tar_gz`：固定 `mtime=0/uid=gid=0/mode=644` + 按 arcname 排序 + `gzip mtime=0` → **同内容字节级一致、sha 稳定**。
