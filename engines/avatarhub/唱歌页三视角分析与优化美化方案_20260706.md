## 〇、一句话结论

**这个页面卖的是「GPT-SoVITS v4 唱歌、AI 生成旋律」，实际交付的是「CosyVoice3 轻柔朗读歌词」，而且这个差距在本机是永久性的：**

1. **GPT-SoVITS 运行时已被物理清空。** `GPT-SoVITS/GPT_SoVITS/` 下只剩 `AR/` 和顶层 py 文件——`TTS_infer_pack/`（推理源码）、`configs/tts_infer.yaml`（配置）、`pretrained_models/`（权重）、`BigVGAN/` 全部不存在（2026-06-01 路线图 Phase 9 已记录"神经 TTS 被清理"）。`singing_server.py` 启动即在 `CFG_PATH` 检查处失败，**7853 端口在本机永远起不来**。横幅里"将降级"三个字，真相是"永远降级"。
2. **就算服务活着，它也不会唱歌。** GPT-SoVITS 是 TTS（朗读），不是 SVS（歌声合成）——没有旋律、没有音高曲线。所谓"唱歌模式"= `speed=0.85` 的慢速朗读。等待文案"AI 逐句生成旋律与咬字"是虚假承诺。
3. **降级产物以绿色✅呈现，且比正常语音「低配」**：`/avatar/sing` 与 `/avatar/speak` 相比，缺水印凭证（合规不一致）、缺历史入库（刷新即丢）、缺并发信号量、缺结构化引擎标识。
4. **真正能兑现「唱歌」二字的资产其实已经在手里**：RVC 变声(6242 在线) + 口型/数字人管线。「AI 翻唱」（歌曲人声分离 → RVC 换成你的音色 → 混回伴奏）才是这个产品能交付的真唱歌，缺的只有人声分离一个依赖（demucs，~80MB）。

| 统计（页面 11 个可见元素 + 2 条隐性链路） | 数量 |
|---|---|
| 真实生效 | 5 项（歌词框/示例/语速/参考上传/就绪体检） |
| 名不符实 / 话术越权 | 3 项（标题引擎名、"生成旋律"文案、绿色✅展示降级结果） |
| 永久空转 | 1 项（SingingTTS 服务承诺） |
| 关键信息缺席 | 2 项（音色来源不可见、引擎标识非结构化） |
| 防线缺失 | 2 项（歌词长度、取消/超时） |

## 二、「唱歌」的三个能力等级（定位问题先于优化问题）

| 等级 | 技术本质 | 本仓资产现状 | 结论 |
|---|---|---|---|
| L1 歌词念白 | TTS 慢速+情感朗读（现状真实能力） | CosyVoice3 在线可用，Fish/Qwen3 备选 | 已有，**该诚实包装而非冒充唱歌** |
| L2 AI 翻唱 | 现成歌曲 → 人声分离 → RVC 换音色 → 混回伴奏 | RVC 6242 在线、角色激活时已推 RVC 配置；**缺 demucs/UVR 分离权重（~80MB）与离线批量变声入口** | **推荐主攻**：真唱歌、真传播力，增量最小 |
| L3 作曲级 SVS | 从歌词+乐谱生成歌声（DiffSinger/ACE-Step 等） | 无谱面输入、无模型、需新 GPU 环境 | **判不做**（理由留档：输入门槛高、受众窄、维护重） |

## 三、三视角分析

### 3.1 开发工程师视角

**痛点：**
- 死服务当活服务维护：健康面板每轮对 7853 白探 2s；UI 为一个永远不会出现的模式写降级文案；`start_all_services.bat` 还在为它保留启动项和专属 conda 环境（musethepeak）；历史上它还引发过 STT 端口冲突（7853/7854）。
- 同类端点不同权：sing 与 speak 一个有水印/历史/并发闸、一个全裸——合规审计和存储语义都不一致，克隆音色唱歌恰恰是最易被滥用的产物。
- 降级不可观测：引擎信息藏在 `warning` 字符串里靠前端拼接展示，没有结构化 `engine_used` 字段，无法埋点、无法做门禁断言。
- 暗雷两处：GPT-SoVITS 路径缺 `prompt_text` 注入（若复活必系统性失败）；`singing_server` 推理无锁（sync 端点 + 线程池并发，管线非线程安全）。

**主张：** 拍板退役 7853（或明确复活成本后择期）；sing 端点补齐水印/历史/信号量三件套 + `engine_used` 结构化；歌词长度服务端 400 防线；参考音频走 STT(7854) 自动转写补 `reference_text`（为任何未来克隆路径铺路）。

### 3.2 市场经理视角

**痛点：**
- 名不符实是差评之源：门户把「AI 唱歌·用克隆音色演唱歌曲」列为 **pro 版卖点**（`avatar_hub.py:19318`），实际交付慢速朗读——付费卖点空心化比没有卖点更伤。
- 强传播产物没有传播闭环：唱歌是全产品最可能被用户主动分享的产物（代码注释自己都写了"唱歌是强传播产物"），现状产物无水印、无角色署名文件名、无分享卡、不入历史，刷新即蒸发。
- 零数据：页面没有任何使用埋点，无法回答"有多少人用过、听完率、下载率"。

**主张：**
- **把「AI 翻唱」立为招牌**：「让你的数字分身翻唱任何歌」+ 接现有口型管线出 15 秒竖屏 MV——这是能自传播的钩子，也是 pro 定价的真支撑。
- 念白模式**改名再卖**：「深情念白 / 为你读诗」自身就是可营销场景（睡前、表白、情话），不冒充唱歌反而立得住。
- 版权口径先行：翻唱功能上线即带"个人娱乐用途"声明 + 强制水印；示例继续只用公有领域曲目。
- 补齐埋点：演唱次数/引擎分布/下载率进现有 metrics 面板。

### 3.3 用户视角

**痛点：**
- "说好的唱歌，出来是慢速念课文"——期待落差是这个页面最大的伤害，比任何 UI 瑕疵都重。
- "它用谁的声音唱？"——页面全程不显示当前角色；上传参考后也不能试听确认。
- 黑箱等待：点了按钮只有一行跳动文字，不知道进度、不能取消、长歌词等更久也没人提前说。
- 橙色横幅一堆代号+端口号，像报错不像产品；出错时直接看到 `singing 服务不可达: ClientConnectorError...`。
- 结果易丢：辛苦等来的音频，切页刷新就没了，历史页里也找不到。

**主张：** 一眼看懂"**谁在唱、用什么方式演绎、大概等多久**"；等待有进度有取消；产物自动进历史、文件名带角色和日期；所有提示讲人话。

## 五、页面改版设计（美化方案）

> 原则：全部复用现有 brand.css v2 体系（`bd-card` / `bd-btn bd-btn-primary` / `bd-badge` / `bd-progress indeterminate` / chips rounded-full），不引入新视觉语言；1440x900 单屏放下，820 宽单列不横滚。

### 5.1 布局：从单列表单 → 双列创作台

- **lg 及以上**：12 栅格。左 7 列 = 歌词卡（输入+计数+示例+格式提示）；右 5 列 = 音色来源卡 + 演绎参数 + 就绪状态 + 主按钮；**作品卡全宽置底**（生成后滚入视野）。当前页面右侧/下方大面积空白（见截图）即被利用。
- **sm**：自然堆叠为 音色 → 歌词 → 参数 → 按钮 → 作品。

### 5.2 五个区块

| 区块 | 内容 | 关键点 |
|---|---|---|
| ① 音色来源卡 | 当前角色头像+名字+「音色就绪 ✓」；segmented 切换 `用当前角色 / 换个声音`；换声音时展开拖拽上传（保留现有拖拽交互）+ 上传后内嵌迷你播放器试听 + 时长徽章 | 回答"谁在唱"；无激活角色时此卡变引导位「去角色库选一个声音」 |

| ③ 演绎参数行 | 演绎风格 chips（深情/欢快/忧伤/平静）+ 语言下拉（样式化）+ 节奏滑杆（改名，含刻度与「回默认」小按钮） | "语速"→"节奏"；滑杆气泡显示 0.85x·舒缓 |
