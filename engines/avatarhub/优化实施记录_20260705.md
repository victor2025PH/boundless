## 2. 做了什么（按执行顺序）

### P0-A 翻译 LLM 右移（A/B 数据定案，推翻原方案两处）
- 用与 `live_interpreter._translate_llm` 完全相同的请求形状（system prompt / think:false / temp 0.2 / num_ctx）对候选模型做六句双语向实测（含品牌占位 Z1Q/Z2Q 压力句与长句）：

| 模型 | GPU 驻留 | 每句中位 | 解码速度 | 结论 |
|---|---|---|---|---|
| qwen2.5:32b（原） | 67.7% | 3632ms | 6.8 tok/s | 溢出即慢，弃 |
| qwen3:32b（原计划升级目标） | 68.3% | 3695ms | 7.2 tok/s | **同体积同样溢出，升级无意义（证伪）** |
| **qwen2.5:14b（采用）** | **100%** | **236ms** | **140 tok/s** | 译质抽验同级、占位全保持 |
| qwen3:30b-a3b（MoE） | 93.4% | 7591ms | 69 tok/s | 拉到的是 thinking 版，think:false/无效 → 思维链污染译文，弃；已改拉 instruct-2507 版待测 |

- 落地：`deploy.env.bat` 设 `INTERP_LLM_MODEL=qwen2.5:14b` + `INTERP_MT_TAG=llm:qwen2.5:14b`（基线分档，回退=删两行）；重启 interp；`ollama stop` 卸载 32b。
- 质量护栏：术语存活探针两轮 healthy；`acceptance.py --only glosssurv` PASS；词表锁定/翻译缓存机制不变。

### P0-B 根因级新发现：ollama num_ctx 错配（本轮最值钱的一课）
- 现象：切 14b 后 `ollama ps` 一度显示 **17G / ctx 32768**——hub 的预热与保活调用 `/api/generate` 不带 options，ollama 0.24 按**模型声明的全量上下文**加载（KV +7.7G）；interp 按 2048 请求 → 两路交替时 runner **反复整模重载**（几秒卡顿）。32B 时代 20G→31G 的膨胀同源，这是当初 CPU 溢出的深层根因之一。
- 修复（双侧对齐）：机器级 `setx OLLAMA_CONTEXT_LENGTH 4096`（重启 ollama 生效）+ `INTERP_LLM_NUM_CTX=4096`。复验：探针与无参调用交替后 `ollama ps` 恒 **9.7G / ctx4096 / 100% GPU**，零重载。
- **运维铁律：同卡多调用方的 num_ctx 必须一致，机器级 OLLAMA_CONTEXT_LENGTH 兜底。**

### P1-A .198 冷备点亮 = 双卡口型池
- WMI `Win32_Process.Create` 远程拉起 `D:\faceX\start_lipsync_standby.bat`（脱 SSH 存活），并注册开机计划任务 `LipSync_Standby`（SYSTEM/ONSTART）→ 断电重启自动回池。
- `deploy.env.bat`：`SVC_LIPSYNC=http://127.0.0.1:8090,http://192.168.0.198:8090` → hub 双副本池（healthy=2，10s 探活，摘除/回池自动）；`CONV_MAX_CONCURRENT=auto` 的准入 K 自动 1→2。
- **异构池调度补丁**（`avatar_hub.py` `_SvcPool.pick`）：新增 `SVC_POOL_TIEBREAK=first`——在途数平局时按 SVC_* 列表序优先取 5090，不再轮转打散；**单路会话永远落快卡，4070 只接真并发溢出**。默认 rr 行为不变（同构多卡零回归）。
- 并发实测（3s 音频，`tools/dual_lipsync_bench.py`）：5090 稳态 1.6~1.8s（RTF≈0.6）；.198 4070 稳态 4.2~4.6s（**RTF≈1.5，超实时线**）。定位：.198 = 第 2 路溢出/排队/离线出片承接（排队 38s 不如慢速出帧）；真·双路实时高清需第 2 张 5090 级卡。

### P0-C 显存预算入拓扑单一源（防复发门禁）
- `cluster_map.json` 新增 `vram_budget`：五机逐卡登记 GPU/物理显存/各服务实测预算，阈值 0.92。
- `tools/topology_lint.py` 新增 `check_vram_budget`：每机 Σ预算 ≤ 物理×阈值，超编 → `topology_drift` 告警。正样本 8 项全绿；负样本（注入 30G）实测退出码 1、文案正确。看门狗 30min 巡检自动带上——**下次谁想把第 4 个服务塞进 5090，lint 直接红灯。**

### P1-B .140 清僵尸 + Nemotron 迁移预检（下阶段执行）
- .140 残留 FaceSwap/EmotionTTS 计划任务与进程（服务早已迁 .104/.117，任务未删，白吃 3.9G）→ 禁用任务 + 杀进程，11.6G→7.7G，whisper+NLLB 健康不受影响。
- 迁移材料已核实：本机 nemoasr env 5.5G + HF 缓存 2.2G；.140 D 盘余 157G；bf16(4.3G)+whisper(4.4G) 同卡 Σ8.7G/12G=73% 可行（服务端 bf16 开关+噪声自检回退已有）。注意：.140 无现成 nemoasr 环境（需 conda-pack 打包传输）；该机还挂着 WLAN 旧网段 192.168.1.43。

## 4. 「实施中再优化」清单（相对原方案的改进）

1. **原方案 P0 被 A/B 数据推翻**：原计划"NEMO bf16 + 同传时段停驻 Ditto + 切 qwen3:32b"。实测证明 qwen3:32b 同样溢出（3695ms），而 14b 让"腾挪显存保 32B"整个思路失效——右移模型后全服务常驻还余 8.8G，nemo 保住 fp32 精度、ditto 保住常驻 512 高清。**方案从"裁服务保大模型"进化为"右移模型保全家"。**
2. **挖出更深一层病灶**：num_ctx 错配导致的 KV 膨胀与隐形重载——不修它，任何模型换上来都会被无参调用悄悄撑回全量 ctx（32768）。
3. **异构池 tiebreak**：发现 .198（RTF1.5）与 5090（RTF0.6）轮转会把单路会话随机打到慢卡 → 把"负载均衡"改成"快卡优先+溢出下沉"。
4. **.140 僵尸清理**：预检时顺藤摸瓜发现 3.9G 显存被退役服务白吃两天，顺手收回。
