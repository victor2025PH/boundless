## 〇、一句话结论

**开播页的设置不是无用的：12 项里 7 项真实生效。但你现在正踩着两个坑，它们恰好让你产生"设置无用"的感觉：**

1. **你选中的「增强 = CodeFormer」在生产环境是空转的。** 生产换脸机(.104)没有部署 CodeFormer（上次部署时就因它缺依赖导致服务起不来，已做优雅降级）。引擎收到 codeformer 请求后发现模型不存在，**静默跳过增强、也不回退到 GFPGAN**——等于精修完全没开。这是画面糊、且"调了没变化"的直接原因。
2. **分辨率调 1080p 反而让脸更糊。** 这个设置只放大画布，脸的实际处理宽度仍固定在 384px（这个决定性旋钮在开播页根本没暴露，只在 /ops 运维页有）。384px 拉伸到 1920 是 5 倍放大，比 720p 时的 3.3 倍更糊，还多花带宽。

| 统计 | 数量 |
|---|---|
| 真实生效（其中 3 项"下次开播生效"） | 7 项 |
| 被自适应变档覆盖后不还原 | 2 项 |
| 生产环境空转（CodeFormer） | 1 项 |
| 决定性旋钮缺席（引擎处理宽/精修档） | 1 个 |

## 一、开播页 12 项设置的真相表（代码审计结论）

| 设置项 | 实际接线（代码事实） | 对画质/清晰度的作用 | 判定 |
|---|---|---|---|
| 分辨率 / 高度（1080p） | 传给采集与输出画布（`--width/--height`） | 只放大画布。脸的处理宽仍是 384px：1080p 把同样的信息拉得更大，脸部反而更糊 | ⚠ 生效 · 但对脸部清晰度是负收益 |
| 摄像头 FPS | `--fps` 采集帧率 | 流畅度相关，与清晰度无关 | ✓ 生效 |
| 换脸 FPS | `--swap-fps` 送检帧率上限 | 启动时生效；自适应一变档就被预设值覆盖，且不会还原 | ⚠ 会被自适应覆盖 |
| MJPEG FPS | `--mjpeg-fps` 输出流帧率 | 流畅度相关，与清晰度无关 | ✓ 生效 |
| 画质（JPEG）低/中/高 | `--jpeg-quality`。**同一个变量同时管「送检压缩」和「观众端输出压缩」** | 启动生效；变档后被预设覆盖。档位「高」也只有 75，对观众端仍偏低 | ⚠ 会被覆盖 · 上限偏低 |
| 换脸平滑(s)（crossfade） | `--crossfade` 切角色渐变时长 | 与清晰度无关 | ✓ 生效 |
| 融合强度（blend） | 每帧请求参数（下次开播生效） | 调低保留原脸轮廓，1=完全换脸 | ✓ 生效 |
| 检测阈值 | 每帧请求参数（下次开播生效） | 影响漏检率，与清晰度弱相关 | ✓ 生效 |
| 换脸平滑（时序） | `smooth_alpha` 请求参数（下次开播生效） | **调大 = 时域混帧 = 更糊。这是一个"反清晰度"旋钮** | ⚠ 生效 · 慎调大 |
| 增强（当前选中 CodeFormer） | 每帧请求 `enhance`；固定值会压过自适应的按档增强策略 | **生产换脸机(.104)未部署 CodeFormer，引擎静默跳过增强。这正是"调了没变化"的直接原因** | ✗ 生产环境空转 |
| 画质预设按钮（低延迟/均衡/高画质/风格） | 只写入上面 6 个画布层参数 | **完全不触碰引擎档位：点「高画质」不会让脸更清晰，只会放大画布** | ⚠ 名不符实 |
| 引擎画质档（省电/自然/美颜/高清） | 处理宽 288~512px + 按档精修开关，只在 /ops 运维页可切 | **这才是真正决定脸部清晰度的旋钮，但开播页上根本没有** | ✗ 开播页缺席 |

## 二、画面模糊的四个来源

1. **整帧降采样**：送检前整帧压到 384px 宽，换完再拉回 720p/1080p，线性信息丢约 70%（1080p 下约 80%）。
2. **默认档不开精修**：默认「自然」档增强 = 关，而 GFPGAN 以 512×512 重建人脸，是"脸变清楚"的最大单一来源。
3. **模型上限**：inswapper 换脸核心输出 128×128，脸部细节天花板低。
4. **双重 JPEG 压缩**：送检一道（55~75）、输出一道（55~75），直播平台还会再压一道。

> **关键前提已就绪，方案不是画饼**：
> GFPGAN 精修已提速 65%（单帧 286 → 212ms）；自适应阈值已完成无人值守标定（降档 650ms / 升档 475ms）；.104 高清档并发均值约 429ms，画质全开常驻已在延迟预算内；整帧 PSNR 44dB 验证精修优化视觉无损。

## 三、三视角分析

### 3.1 开发工程师视角

**痛点：**
- 两套预设系统平行存在（画布层 vs 引擎层）且名字撞车：「高画质」不等于「高清」，没有单一真相；
- UI 值与自适应互相踩踏：变档覆盖用户的 JPEG/FPS；UI 的增强固定值压过按档策略，降档降不掉增强成本；
- 前端敢下发生产机不存在的 CodeFormer，缺能力探测；关键旋钮（处理宽）根本没暴露。

**主张：** 合并为一个画质档位同时驱动两层；送检/输出 JPEG 解耦；用 `/health` 能力驱动 UI 选项；补参数热更接口。

### 3.2 市场经理视角

**痛点：**
- 差异化能力被埋没：智能自适应画质 + 精修提速 65% 都藏在运维页，用户全程无感；
- GFPGAN/CodeFormer/blend 是工程黑话；"调了没用"是应用商店差评高频词，选项空转是口碑硬伤；
- 90% 用户从不动设置，默认档就是产品力本身。

**主张：** 档位按结果命名（流畅优先/自然/高清脸/影棚级）；前后对比滑杆可截图分享；状态条讲人话；把已有标定工具包装成「开播前一键画质标定」的仪式感。

### 3.3 用户视角

**痛点：**
- "分辨率调到 1080p、画质调到高，脸还是糊"——操作没有正反馈，自然得出"设置无用"的结论；
- "下次开播生效" = 改了看不到变化，以为功能坏了；画质突然变差也没人告诉我为什么；
- 不敢乱动：不知道哪个旋钮会导致卡顿，试错成本是一场直播事故。

**主张：** 一个旋钮 + 即时预览 + 变化有提示；体检给分数和建议；直播中不可改的项置灰并说明原因。

## 十、P2 实施结果（2026-07-06 完成）：脸区原生分辨率通道上线

**这是本方案链条的主菜**：不再把整帧缩到 512 送检，而是按上一帧回传的脸框**裁原图送检**——脸按原生像素进引擎，背景原生直出完全不过引擎。滞回稳框 + 周期全帧探测(3s,发现新脸) + 失脸自动回退全帧 + 多脸自动回退；羽化回贴无缝；运行时可热切(`/realtime/swap/crop?on=0|1`，/ops 有开关芯片)。

**同源 A/B 实测（合成运动人像 720p，脸约 150×200px，hd 档 + GFPGAN）：**

| 指标 | 裁剪通道 开 | 裁剪通道 关(旧路径) | 提升 |
|---|---|---|---|
| 脸区清晰度(Laplacian) | **167.9** | 37.6 | **≈ 4.5×** |
| 脸区清晰度保留率(vs 原始) | **124%**(超过原摄像头) | 40% | +84pp |
| 嘴部保留率 | **113%** | 35% | +78pp |
| 单帧引擎时延 | **243ms** | 304ms | **快 61ms** |

画质翻倍的同时更快（裁剪块 304×352 < 全帧 512×288 像素数，且背景不再经引擎 JPEG 往返）。验收目标「脸部清晰度 +60%」实际达成 +350%。证据图：`logs/ab_crop_on_20260706.png` / `ab_crop_off_20260706.png`。

**六、验收指标 达成情况**：脸部清晰度 +60% → **实测 4.5×，远超**；并发均值延迟 <650ms → 243ms；高清档出帧 >7fps → 达标(swap_fps=8 满速)；首屏旋钮 1 个 → P0 已达。

## 十一、P3 实施结果（2026-07-06）：1080p 放开 + 增强器/256 模型循证结论

**1. 超清 1080P 档放开（已上线）**：P2 脸区原生通道解决了「1080p 稀释脸部」的老问题（脸按原生像素送检、背景原生直出），1080p 才第一次有意义。实测（synth_cam 1080p·裁剪通道·hd+GFPGAN）并发均值 **323ms < 650ms 预算**，裁剪窗 448×544、269 命中 0 回退、出帧 29.9fps。新增「超清1080P」opt-in 预设（专家模式可见，高清脸仍为安全默认）。

**2. 增强器循证结论（A/B 实测，非拍脑袋）**：

**3. ## 十二、P4 实施结果（2026-07-06）：增强并发池上线——GFPGAN 吞吐 +65%、CodeFormer +52%

**根因**：瓶颈不在神经网权重（eval+no_grad 下同一 module 并发前向安全），而在 **FaceRestoreHelper 的逐调用可变状态**（cropped_faces/affine 等存实例属性）——多 worker 只能排队等 `_enhance_lock`。

**解法**：`FACESWAP_ENH_CONCURRENCY=3`（与 `SWAP_WORKERS=3` 对齐）→ 每槽独占 helper、共享 GFPGAN/CodeFormer 权重；Queue 兼作并发上限与分配器。`concurrency=1` 时完全走旧锁路径（零回归）。

**3 路并发 A/B（原生脸裁剪·.104 CUDA·经 hub）**：

| 增强器 | P3 锁串行 | P4 池×3 | 提升 |
|---|---|---|---|
| GFPGAN 聚合吞吐 | 5.3 fps | **8.8 fps** | **+65%** |
| GFPGAN 中位 RTT | 445ms | **324ms** | **-27%** |
| CodeFormer 聚合吞吐 | 3.65 fps | **5.55 fps** | **+52%** |
| CodeFormer 中位 RTT | 786ms | **510ms** | **-35%** |

CodeFormer 并发 RTT 已压到 510ms，仍不适合做 hd 实时默认（单路预算 ~8fps 仍偏紧），但「极致精修 opt-in」体验明显改善。GFPGAN 作为 hd 默认的实时余量显著增加。

**已上线**：`.104` health `enh_concurrency=3 gf_pool=true cf_pool=true`；`env_config.bat` + `.104` 启动脚本已写入；`/ops` 引擎芯片显示「增强×3」。

## 十五、P7 实施结果（2026-07-06）：观察自动化——停播自动出「场次换脸质量报告」

**动机**：P3-P6 反复出现的下一步都是「真人直播观察，记录 hits/miss、降档、时延、增强观感」——而这些量化项本就是引擎计数器，只是活在 `realtime_stream` 进程里、停播即随进程销毁（`terminate()` 无优雅退出，进程自己写不了终帧报告）。人肉记数既不可靠也没必要。

**方案**：hub 健康守护（已有 4s tick）在播时顺带采样本机 `/swap/status`（1.5s 超时、独立 try，采不到只记缺样、绝不影响自愈主流程）；停播（健康态回 idle）自动聚合落 `logs/swap_sessions.jsonl`：

- 时延 med/p95/max、fps med/min（逐 tick 序列聚合，EWMA 之外抓到尾部）
- 裁剪 hits/miss/命中率、换脸成败/失败率——**carry 进位**吸收「自愈重启视频管线→计数器归零」（`cur<last` 视作重启，last 进位后重新累计）
- 生效档分布、降档 tick 占比 + 降档原因计数、增强引擎分布（gfpgan/codeformer/无）、虚拟背景使用、健康状态分布

**出口**：`/realtime/swap/sessions?limit=N`（最新在前，在播附 current 半场预览）+ `/api/swap_sess/selftest`（合成快照含一次计数器重置，8 断言走真聚合函数，部署后已验 ok=true）。前端停播成绩单自动补第二行四格：**裁剪命中 / 换脸时延 / 自动降档 / 精修引擎**（悬停降档格可见原因），「复制成绩单」一并带量化行——实测场次的记录产物直接可交。

**对真人观察的意义**：观察清单里可量化的（hits/miss、降档、时延、增强路径）全部自动化；人工只剩三件主观项——贴缝观感、CodeFormer 满意度、体检卡点误拦。开播前无需任何操作，播完看成绩单/复制即交。

**P7 二期（2026-07-06l，观察闭环收口）**：①往期报告并入 /ops「场次复盘（甘特）」——健康与画质按开播时刻配对（≤60s），一场一行、下钻见完整画质报告、CSV 追加 8 画质列（弃「另立列表卡」原案：同批场次两处列表=重复+对不上号）；②停播成绩单新增「**对比上场**」趋势行（裁剪/时延/降档 上场→本场，绿好橙差）——连续实测 1–2 场时直接看变化方向，复制成绩单一并带趋势。

## 二十、P12 实施结果（2026-07-06）：稳定性哨兵 + 探测子进程善终——隔离体系收官

**动机（P11 账本自己暴露的两个尾巴）**：①账本首跑抓到 18:39:14 `libportaudio64bit.dll 0xc0000005`——dev_probe 子进程**把结果交回之后**、解释器收尾触发 PortAudio 原生 teardown 时崩。不伤 Hub（P10 设计目标达成），但每次崩溃都拉起 WerFault(1-2s CPU)、在事件日志攒 python.exe 崩溃记录、给账本「已隔离崩溃」计数添永久性噪声——**已隔离的崩溃也该在源头消灭**。②账本是拉式看板：没人开 /ops 就没人知道回归；P10 若真出回归，用户只感到偶发 20s 卡顿，很可能耸耸肩就过去了——验收指标需要**推式哨兵**。

**1. dev_probe 交卷即硬退出**：`main()` 打印 JSON（flush 落定）后 `os._exit()` 直接终结进程——跳过解释器 finalizer/atexit/原生库 teardown 全链。一次性探测进程无任何需要善终的状态（设备句柄由内核随进程回收），父进程只解析 stdout 不看返回码，零风险。实弹验证：真录音 2s / 输出试听 / 枚举 三路跑完 rc=0，事件日志**零新增崩溃**（修复前同样操作在 18:39 留下过实锤）。

**2. 稳定性哨兵（账本从看板升级为哨兵）**：hub lifespan 挂 `_bg_stability_sentinel` 后台环——启动 3min 后首查、之后每 30min（`HUB_STAB_SENTINEL_SEC` 可调）子进程跑账本：`restarts_after_p10>0` → `alerts.raise_alert("hub_stability_regression")`（钉钉/企微 webhook + 本机 toast，去抖由 alerts 层管，detail 带逐条归因模块）；归零 → 自动 clear「已恢复」。结果回填 /ops 缓存（开页秒出，省一次现算）。端点 `?refresh=1` 走同一裁决路径，E2E 可即时验证。

**验证**：门禁新增 `test_p12_sentinel_probe_exit`（dev_probe 真跑退出码必须是干净 0/1 而非 0xC0000005=3221225477 + 哨兵四件套接线 + lifespan 挂载）全绿；重启 Hub 后 `/api/ops/stability?refresh=1` 实测 `sentinel=clear`（P10 后 0 拉活 vs 前 11 次，alerts fire/clear 链路真实走通）。

**至此隔离体系闭环**：不进 Hub（P10 子进程隔离）→ 崩了有账（P11 三源账本）→ 回归有人叫（P12 哨兵）→ 连子进程自己的善终崩溃也消灭在源头（P12 硬退出）。

**P12 补强（2026-07-06 晚，生态测绘暴露的两处修正）**：部署哨兵时对本机做了一次进程生态测绘，发现 Hub 的「拉活主体」远不止守护——`start_avatar_hub.bat` 接管窗口（`:loop` 每 10s 轮询 9000，端口一空立刻抢跑）、今天残留的 10 个 `cmd /K _launch_hub_detached.bat` 僵尸窗口（与各次 Hub 死亡时刻 ±1s 吻合的第二快速反应主体）、boot_stack/看门狗，实测杀掉 Hub 后 6-10 秒内必被某个主体抢拉。由此修正两处：①**Boot 溯源落专用账页** `logs/hub_boots.jsonl`——经 `start_avatar_hub.bat` 直启的 Hub stdout 进它自己的控制台窗口，`hub_console.log` 里 [Boot] 完全隐身（19:37 实证），账本以专用账页为主源、控制台解析降级为历史兜底（顺手修掉 str/Path 拼接静默失败）；②**哨兵口径收窄为「崩溃实锤击中 Hub」才告警**——部署竞态里 taskkill 秒被抢拉会记成「无实锤拉活」，旧口径(任何拉活>0 就 fire)让哨兵在 1 天统计窗内误报一整天；新口径无实锤拉活进账本供人判读（/ops 拉活 chip 琥珀色区分）但不触发告警。竞态还意外完成哨兵全生命周期实弹：旧口径 19:51 真 fire（webhook/toast 链路走通）→ 新口径部署后自动 resolved——fire/clear 两条路都在生产验证过。

## 二十一、P13 实施结果（2026-07-06 晚）：激活幂等防抖——守护「疑似泄漏」软警的真凶拔除

**取证（两条日志一交叉，真凶不是泄漏是风暴）**：①内存曲线形态：0.2G→1.0G(启动预热)→2.8G 后**平台稳定**——是高水位棘轮（Python 堆不还 OS），不是无界泄漏；②访问日志抓到 **/profiles/\*/activate 连发风暴**：19:25 一分钟 10 次、19:35 一分钟 12 次、20:01 更是 **12 秒内 9 次**（手机页 `selectProfile` 每次进页/切卡/URL 预选都无条件 POST activate，多客户端叠加）；③而 `activate_profile` 每发都全量重跑昂贵预热链：换脸底座切脸 + 口型人脸预计算 + **跨机** spk 注册(.117) + **跨机** fish 参考预热(90s 超时) + 114 条开场白音频缓存重载 + RVC 配置推送。风暴期间这些并发翻倍 → RAM 冲顶后堆驻留=守护看到的「泄漏」；预热请求挤占远端 TTS，也解释了当日体检 qwen3 克隆 15.8s 红旗的一部分。

**修法（Hub 侧幂等防抖，30 行）**：同角色 + 配置版本未变 + 30s 窗口内的重复激活 → 短路返回 `{ok,active,face_ready,face_eta_s,deduped:true}`（响应形状与全量路径一致，客户端无感）。三个关键设计：①**key 含 `_profiles_version`**——改垫话/情绪参考/角色内容会 bump 版本号，天然穿透防抖，「改完重激活即生效」语义零破坏；②**固定窗口不滑动**——风暴持续也只压 30s，窗口过后重新全量，保留「再点一次=手动重试预热」的恢复语义（预热曾因远端服务未就绪而失败时用户能自救）；③短路路径**不重发 profile_activated 广播**——状态没变不惊动其它客户端，也杜绝事件反馈环。窗口 `AVATARHUB_ACTIVATE_DEBOUNCE_S` 可调。

**实测（复刻风暴 + 前后对照）**：①复刻 12s 内 9 连发：deduped 序列 `[False, True×8]`——预热链整场只跑一次；②**内存对照最有说服力**：旧 Hub 在 20:01 风暴后冲到 2.8G 高水位；新 Hub 部署后经历同样的启动自动激活 + 复刻风暴，RSS 稳在 **0.2G**，守护泄漏软警零再现；③门禁新增 `test_p13_activate_debounce`（防抖状态/env 窗口/短路标记/版本穿透/「短路→锁→落 key」结构顺序/单调钟六路断言），全套 379 OK / 0 NG。

**边界（诚实口径)**：2.8G 里有一部分是开场白缓存+口型预计算的合法工作集，防抖消灭的是「同一份工作集被连发重复支付」的部分；切换**不同**角色仍然每次全量（应该的）。前端 selectProfile 的无条件 activate 属另一层冗余，服务端已兜底，前端防抖降为低优。

**顺手的生态卫生**：清掉今天生态测绘发现的 10 个 `cmd /K _launch_hub_detached` 僵尸窗口（严格过滤：命令行匹配 + 子进程仅剩 conhost 才杀），用户桌面不再有一排「假 Hub 窗口」可认错；`hub_boots.jsonl` 专用账页继续工作，20:28 部署拉活完整入账。

**下一阶段建议**：① 自然使用观察——守护「疑似泄漏」软警与哨兵告警双静默即本轮修复终验；② 若 RSS 高水位仍缓慢爬升（排除激活因素后），下一个勘察对象是会话/历史类长活缓存（按本轮「曲线形态+访问日志交叉」方法论复查）；③ 官方级 256/512 换脸核跟踪（P3 加载器就绪，即插即用）→ **✅ P14 已把「跟踪」做成发布哨兵，机器日查**。
