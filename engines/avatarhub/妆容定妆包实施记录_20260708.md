## 二、深入思考产生的 3 次方案改进（比原方案更优）

### 改进 1：放弃 FLUX-Makeup 重型扩散路线 → MediaPipe 478 点 + LAB 区域颜色迁移
原方案拟用 FLUX-Makeup（FLUX.1-Kontext 底座）。实施前推演发现：
- 权重 ~24GB、需新 conda 环境，直播生产机上下载/占显存都不现实；
- 更本质：**inswapper 换脸走 ArcFace 身份向量，而 ArcFace 对妆容鲁棒**——
  妆烘进源脸后在直播换脸输出里基本不存留，重型生成投入产出严重不成比。

改用 MediaPipe FaceMesh(478 点，模型仅 3.7MB) 构建唇/眼影带/腮红/磨皮区掩码，
LAB 空间只迁移 ab 色度保 L 纹理。实测 Inside 真脸 **~200ms/张、纯 CPU 零显存**，
预设+参考图两种模式都出图自然（见 logs/look_pack_impl_20260708/）。
FLUX-Makeup 保留为后端插槽（MAKEUP_BACKEND），权重就位可无缝换。

### 改进 2：妆容双路架构——离线定妆管数字人，直播层管换脸
由改进 1 的洞察推导出：妆容必须分两条链路才都可见：
- **离线定妆**（8004）→ face_styled_b64：数字人照片链（Ditto/LivePortrait/lipsync
  直接用照片像素）妆容 100% 可见；
- **直播妆容层**（faceswap_api `_apply_live_makeup`）：换脸输出帧的目标脸区上妆，
  这是换脸直播里妆容唯一真正可见的位置。复用已检测的 tgt_face landmark（零二次检测），
  唇色按 LAB a 通道红度加权——张嘴露齿牙齿不染色。

### 改进 3：直播层性能迭代 61ms→10.4ms（实测驱动）
首版全分辨率 ROI 大核高斯模糊，720x1280 实测 61ms/帧——不可上直播链。
改「降采样工作区(≤288px)算色移 delta → 双线性上采样叠回」：妆容是低频色偏，
delta 上采样视觉无损、全分辨率纹理原封不动。同图实测 **10.4ms/帧**（含预设三件套）。
另发现两个环境坑并防御：mediapipe 中文路径打不开模型（改 model_asset_buffer 内存加载）、
cv2.imwrite 中文路径静默失败（改 imencode+write_bytes）。

### 3-B Ditto 静默驱动实测 + 两处真相修正

- **纯零静音是错的**：A/B（tools/_ditto_idle_ab*.py）发现全零 WAV 让 HuBERT 特征
  落在分布外，Ditto 反而幻觉出"说话嘴型"（嘴部动作分 2.05，肉眼可见虚假张嘴）。
  掺 **-80dB 高斯底噪**后嘴部虚假动作减半(0.99)、唇形闭合，全局呼吸/眨眼微动不受影响。
  `_silence_wav` 已改为固定种子底噪版（字节级可复现）。
- **时装全身照小脸检测失败**：VITON 类照片人脸仅 ~100px，Ditto 人脸配准返回 None。
  `_tryon_animate_ditto` 升级为**二段策略**：①原图直跑；②失败裁上半身(45%×70%)×2 放大
  重跑，动画帧 ffmpeg overlay 贴回原位（身体不动、头部微动=待机观感）；仍失败回落静帧。
- 真机全链冒烟（tools/_lookpack_live_smoke.py，直播中跑）：临时角色 创建→妆容定妆→
  一键定妆包(复古红唇 173ms)→**试衣定妆(FitDiT 13.4s + Ditto 动化 animated=true)**→

### 4-C 生产事故复盘 → 显存准入闸（本阶段最重要的一课）

现象：直播中 GPU 占用爬到 28.5G（空闲 ~4G）时跑 768 档试衣，**600 秒无结果、
GPU 98% 假忙碌**（14:33 同样请求仅 9.3s）。根因：Windows WDDM 驱动在显存不足时
不报 OOM，而是**静默回落到共享系统内存**——PCIe 每权重页往返，慢 60 倍以上还拖累直播。
处置与加固：
1. 杀掉磨盘进程（显存立即回落 22.3G）；
2. `tryon_api` 新增 **显存准入闸 `_vram_gate`**：按档位阶梯（768→7G / 1152→9G /
   1536→11G，`TRYON_MIN_FREE_GB` 可覆盖）检查空闲显存，不足**立即 503** 带人话提示
   （"直播占用中，请稍后再试"）——宁可快失败，不做假忙碌；
3. 加 **推理串行锁**：FastAPI 同步端点默认 40 线程并发，两单同时进来=双倍显存峰值，
   现在同一时刻只跑一单其余排队；
4. 闸后复测：768 档 15s、1152 档 19.8s 双双通过，直播换脸链无感知。

|---|---|

### 18.2 Windows 落地三件套（零编译、零环境污染）

外界普遍认为 CatV2TON 依赖 detectron2/densepose「Windows 没法跑」——实测是
**误判**：vendored detectron2 推理路径全程纯 Python（ROIAlign 走 torchvision），
只缺 `fvcore av` 等纯 Python 包。真正的坑是三个**版本漂移**，全部用运行时垫片
（`tools/_catv2ton_shim.py`，import 即注入，不改仓库一行）解决：

1. **pkg_resources 没了**（setuptools≥81 移除）→ `packaging.version` 造等价
   `parse_version` 注入 `sys.modules`。
2. **torchvision 0.25+ 移除 io.read_video/write_video** → cv2 等价实现补挂。
3. **RoPE 位置编码漂移（本次最深的坑，值得记）**：diffusers 0.38 的
   `get_2d_rotary_pos_embed` 把网格从 numpy `linspace(endpoint=False)` 换成
   torch 含端点版本，cos/sin 相对训练期（0.29）最大漂移 0.16 →
   **位置编码整体错位，衣区直接出纯噪声**。垫片按 0.29 原版逐行复刻（Δ=0 位级
   一致）。教训：**跨大版本跑扩散模型，位置编码必须逐位核对，不能只看能跑**。

另有 modules 包 `__init__` 顶层就 import AutoMasker（拖 detectron2 全家）→
垫片注册空壳包骨架跳过 `__init__`，子模块照常按路径加载。

### 18.3 根因链：从「纯噪声」到「能穿衣」的排查记录

现象：修完 RoPE 仍纯噪声。分层定位（工具全留在 tools/）：

- `_catv2ton_vae_test.py`：VAE 往返 L1=2.1 正常；DiT 前向无 NaN、对 t 有响应。
- `_catv2ton_x0_test.py`：一步 x0 重建 err≪base（t=500: 0.20 vs 1.02）——
  前向/权重/RoPE 全对，问题在**条件缺失**。
- gs=1.0 出「模糊鬼影人」→ 判定：**DensePose 条件是画质命门**。
  CatV2TON 的 posenet（289MB）是训练必备组件，`load_pose=False` 推理属于
  分布外输入，CFG(2.5) 会把这种分布外垃圾放大成纯噪声。
- 补上逐帧 DensePose（viridis 上色，与训练分布一致）→ 出图正常、时序稳定。

### 18.4 质量观察与遗留

- ✅ 衣服花纹跨帧一致、无闪烁（AdaCN 生效）；花纹保真度高。
- ⚠️ FitDiT 的 Upper-body 遮罩对该样本偏大（盖到裙子）→ 模型把整身都
  织成了衣服布料（连体裤效果）。视频链要用更紧的分部位遮罩。
- ⚠️ 手部姿态与源视频有偏差（叉腰 vs 交叠）——遮罩盖住手臂后 DensePose
  引导权重有限。遮罩收紧后应缓解。
- 产物：`logs/catv2ton_poc/`（对比帧、6 帧时序条、结果视频）。

### 19.2 冻机事故与「显存工程五件套」（本阶段最重要的沉淀）

首次冒烟把整机拖死重启（显示驱动饿死，非蓝屏）。根因链：
**WDDM 给每进程虚拟化显存预算，`torch.cuda.mem_get_info()` 报的是本进程预算
视图而非物理空闲——物理只剩 6.3G 时它还报 30.2G**。闸门读假数放行 → VAE 解码
整段 13 潜帧激活 ~14.5G 爆穿物理 → WDDM 不报 OOM 而是溢出到共享内存 →
显示驱动被饿死 → 整机冻结。此后八轮冒烟逐项加固：

1. **物理显存闸**（vram_gate 重构）：NVML 优先（µs 级）→ nvidia-smi（~150ms）
   → torch 视图仅兜底。1s 缓存。
2. **动态进程配额**：每单开跑前 `set_per_process_memory_fraction(驻留+物理空闲
   -预留3G)`——越界抛干净的 CUDA OOM（作业转 error 可恢复），绝不与桌面抢物理页。
   八测发现配额是「提交时快照」会勒死自己（腾挪后物理反而更宽），解码开跑前
   按最新空闲重定一次。
3. **顺序 CFG**：原 step() 把 uncond/cond 拼 batch2 一次过 DiT，激活×2；
   改顺序两次前向（数值恒等），去噪 21s→42s 可忽略，激活减半。
4. **VAE 编/解码时间分块**：上游 `_slice_vae` 名为 slice 实为整段编码
   （FIXME 是上游自己留的）；解码同病。编码 16 像素帧/窗+8 重叠丢弃；
   解码 chunk=2+ov=2（≤4 潜帧/窗）+ 2 潜帧重叠线性融合。
   **六测教训：重叠宽度是画质命门**——ov=1 时序 L1 0.58→2.66 肉眼可见接缝，
   显存要从别处省，不能砍融合宽度。
5. **解码期 Transformer 下卡**：去噪结束后 DiT 就是死重（bf16 ~3G），
   `.to("cpu")` 换出 VAE 窗口余量，下一单去噪前搬回（PCIe 秒级）。

另修一处**契约陷阱**（四测衣区纯白爆掉的根因）：官方 `smooth_video_mask`
进出都是 **0-255 字节张量**，直接当 0-1 喂 VAE = 遮罩放大 255 倍。除 255 归一。

### 19.3 自愈系统与长作业的对抗（七测 30s 解码拖成 340s 的根因）

现象：解码期 GPU 利用率 2-19%、物理空闲 8.5G→2.1G 直坠。
根因：冒烟脚本裸 `engine/stop` 停掉 lipsync 腾显存，**hub self-heal 20s 一拍
把它拉回来**（+5G），物理页被挤穿 → WDDM 把本进程显存换页到共享内存 →
解码在换页抖动里爬行。对策双管齐下：
- `engine/stop?suspend=1`：停+登记 `_PARK_SUSPEND` 挂起自愈；`engine/start`
  对称解除。腾挪场景（冒烟/运维）必带。
- Hub 提交链 `_videotryon_vram_rescue()`：卸闲置模型（发型/高清口型/听写）+
  free_unused 泊本模式闲引擎 + **被泊者挂起自愈**；作业终态/15min 兜底解泊。
- **提交即腾挪，不等 503**（Hub 侧 E2E 揪出的设计课）：只在被拒后才腾挪的话，
  「闸门擦边放行」的单子会带着重引擎同卡开跑——去噪勉强撑过，解码期动态配额
  被当时的物理空闲卡死在 12G 直接 OOM。作业分钟级、被泊引擎完事自动回场，
  主动腾挪代价为零；提交失败路径逐一 `_PARK_SUSPEND.discard` 兜底解泊。
