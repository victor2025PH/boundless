## 3. S1 · 空 `auth_token` 时 API 全放行

### 3.1 现状证据
`token = web_cfg.get("auth_token", "")`（`admin.py` L245，默认空串）。`_api_auth`：

```711:734:D:\workspace\boundless\engines\chengjie\src\web\admin.py
    def _api_auth(request: Request):
        def _agent_guard():
            if request.session.get("role", "") == ROLE_AGENT:
                if not _agent_api_allowed(request.url.path):
                    raise HTTPException(status_code=403, detail="坐席角色仅限工作台相关接口")
        if request.session.get("user_id"):
            if not _check_session_valid(request):
                request.session.clear()
                raise HTTPException(status_code=401, detail="Session 已失效，请重新登录")
            _agent_guard()
            return
        if not token:
            return
        auth_header = request.headers.get("Authorization", "")
        if auth_header == f"Bearer {token}":
            return
```

### 3.2 根因
`if not token: return` 是**失败即放行（fail-open）**。默认 token 为空时，任何未登录请求打到该分支直接通过。页面路由由 `_require_auth` 走登录保护，于是形成「HTML 要登录、JSON API 不设防」的 split-brain，运维会误判为安全。

### 3.3 补丁位置
`src/web/admin.py`，`_api_auth` 内 `if not token: return`（约 L722-723）；同时在文件顶部确认 `import hmac`（用于 S6 恒定时间比较），并新增本机判定 helper。

### 3.4 补丁草案（完整见第 7.1 节）
- 将 `if not token: return` 改为：**仅当 `user_store.user_count()==0` 且请求来自本机**时放行（供全新安装跑 `/setup`）；否则 `401`。
- Bearer 与 legacy session token 比较改 `hmac.compare_digest`（顺带完成 S6-a）。
- 新增 `_is_local_request(request)`：`request.client.host in {"127.0.0.1","::1"}`。

### 3.5 配置变更
无。行为对「已配 `auth_token`」或「已用登录系统」的现网**零影响**（那两条路径本就先命中）。仅堵住「空 token + 未登录」的裸奔。

### 3.6 兼容性 / 回滚
- 若某些内部脚本此前依赖「空 token 直连 API」，需改为带 `Authorization: Bearer <token>` 或走 localhost。落地前 grep 内部调用方。
- 回滚：单 commit revert 即恢复旧行为。

### 3.7 验证步骤
```bash
# 复现（打补丁前应 200，补丁后应 401）——在设置了用户但 auth_token 为空的实例上：
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:18799/api/audit          # 本机：期望仍可（若 user_count>0 则 401，需带登录态）

curl -s -o /dev/null -w "%{http_code}\n" -X POST http://<lan-ip>:18799/api/data-purge  # 期望 401
```
新增单测 `tests/security/test_api_auth_empty_token.py`（见第 8 节）。

### 3.8 DoD
- 空 token + 有用户 + 无会话 → 所有 `/api/*` 返回 401（除本机 setup 白名单）。
- 全量回归绿；新增用例绿。

## 5. S3 · CSRF 非 JSON 写请求旁路

### 5.1 现状证据
```298:301:D:\workspace\boundless\engines\chengjie\src\web\admin.py
        ct = request.headers.get("content-type", "")
        if "application/json" in ct:
            return JSONResponse(status_code=403, content={"detail": "CSRF token missing or invalid"})
        return await call_next(request)
```

### 5.2 根因
仅当 `Content-Type: application/json` 时才 403；其余（`application/x-www-form-urlencoded`、`multipart/form-data`）在 token 与 Origin/Referer 校验**双双失败**后仍 `call_next` 放行。基于 session 的表单写端点（如 `/users/create`、`/templates/update`、`POST /import`）可被 CSRF。

### 5.3 补丁位置 & 草案
`admin.py` L298-301：非安全方法在所有校验失败后**一律拒绝**：
```diff
-        ct = request.headers.get("content-type", "")
-        if "application/json" in ct:
-            return JSONResponse(status_code=403, content={"detail": "CSRF token missing or invalid"})
-        return await call_next(request)
+        # 非安全方法，且 CSRF token / 同源 Origin/Referer 均未通过 → 一律拒绝（含表单编码）。
+        return JSONResponse(status_code=403, content={"detail": "CSRF token missing or invalid"})
```
> 前置豁免（Bearer、`/chat/*`、LINE webhook、同源 Origin/Referer、双提交 cookie）保持不变，正常浏览器操作（同源带 Referer）不受影响。

### 5.4 兼容 / 回滚
- 若有**跨源表单**自动化提交历史行为，会被拦。落地前确认无跨源表单端点依赖；如有，改为带 `X-CSRF-Token` 或 Bearer。
- 回滚：revert 单 commit。

### 5.5 验证
```bash
# 伪造跨源表单 POST（无 CSRF token、Origin 为外站）→ 期望 403
curl -s -o /dev/null -w "%{http_code}\n" -X POST \
  -H "Origin: http://evil.example" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data "username=x&password=y&role=master" \
  http://127.0.0.1:18799/users/create
# 同源正常操作（带正确 Referer + csrf token）→ 期望非 403
```
单测 `tests/security/test_csrf_form_reject.py`。

### 5.6 DoD
表单/multipart 写请求在无 CSRF token 且非同源时返回 403；同源正常操作不受影响。

```python
            if request.session.get("role", "") == ROLE_AGENT:
                if not _agent_api_allowed(request.url.path):
                    raise HTTPException(status_code=403, detail="坐席角色仅限工作台相关接口")
        if request.session.get("user_id"):
            if not _check_session_valid(request):
                request.session.clear()
                raise HTTPException(status_code=401, detail="Session 已失效，请重新登录")
            _agent_guard()
            return
        # 修复：空 token 不再 fail-open。仅「全新安装（无任何用户）+ 本机」放行，供 /setup 创建首个 master。
        if not token:
            if user_store.user_count() == 0 and _is_local_request(request):
                return
            raise HTTPException(status_code=401, detail="Unauthorized")
        auth_header = request.headers.get("Authorization", "")
        if hmac.compare_digest(auth_header, f"Bearer {token}"):
            return
        sess = request.session.get("auth")
        if sess and hmac.compare_digest(str(sess), str(token)):
            if not _check_session_valid(request):
                request.session.clear()
                raise HTTPException(status_code=401, detail="Session 已失效，请重新登录")
            _agent_guard()
            return
        raise HTTPException(status_code=401, detail="Unauthorized")
```
> 顶部确保 `import hmac`。逻辑差异仅在「空 token 分支」与「比较改恒定时间」，其余保持。

## 12. 风险与开放问题（需评审拍板）

1. **S1 本机白名单**：若生产经反向代理，`request.client.host` 可能恒为代理 IP（127.0.0.1）——需确认是否读 `X-Forwarded-For` 且**仅信任可信代理**，否则本机白名单可能被伪造。建议：生产必配 `auth_token` 或强制登录，使白名单分支永不触发。
2. **S2 `must_change` 改密引导**：是否本 Sprint 就改 `WebUserStore._ensure_master` 签名？若求稳，先落「随机口令 + 落盘提示」最简版，改密引导后置。
3. **S3 跨源表单**：确认现网无跨源表单自动化（如外部页面 POST 到本后台）。有则需迁移到带 CSRF/Bearer。
4. **S5 CORS 默认收敛**：确认没有独立前端域名依赖跨域带凭据访问本引擎；有则配精确白名单。
5. **S6-d 登录爆破**：per-username 锁定是否纳入本 Sprint（+0.5d）还是 Sprint 1。
6. **CAS 多进程**：当前单进程 `threading.Lock` 足够；若未来上多 Uvicorn worker/多连接，需将租约抢占下沉为 SQLite 语句级条件写（本补丁已是条件写，天然兼容）。
